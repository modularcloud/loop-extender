# ADR-0004: Run-Scoped Temporary Directory, Programmatic Env, and Project-Root CWD

**Status:** Proposed

---

## Context

Three pain points have emerged as workflows have grown.

1. **Intermediate data between scripts in a run.** Workflows like `review-adr`, `apply-adr`, and `spec-test-adr` stash hand-off files (`.feedback.tmp`, `.claude-output.tmp`, `.answer.tmp`, `.prompt.tmp`) in their own workflow directory or in `.loopx/shared/`. Cleanup is `rm -f` on the happy path; leaked files accumulate on error/signal/unexpected termination and are indistinguishable from legitimate content. `ralph`'s hand-maintained `.iteration.tmp` counter has the same issue — intra-run state leaking into the workflow directory because there is no per-run scratch space.

2. **Per-run parameterization from programmatic callers.** `run()` / `runPromise()` callers have no way to inject per-invocation env vars without mutating `process.env` globally, which is racy under concurrent calls. The CLI already handles this natively via shell env-var prefix (`adr=0003 loopx run review-adr`); the programmatic API has no equivalent.

3. **`$LOOPX_PROJECT_ROOT` prefix boilerplate in every script.** SPEC §6.1 currently makes each script's cwd its workflow directory, so `.loopx/` scripts open with `ROOT="$LOOPX_PROJECT_ROOT"` and prefix every reference to reach project files. The §6.1 justification ("relative imports and `node_modules/` resolve naturally") is inaccurate — Node ESM, tsx, and Bun all resolve bare specifiers and relative imports from the importing file's directory, not cwd.

An earlier draft proposed a `loopx run … -- name=value` CLI surface. It is not adopted in v1: shell env-var prefixing is already sufficient for passing parameters, and the parser surface would add substantial complexity (two-phase parsing, name grammar, duplicate detection, `LOOPX_*` reservation, precedence tier reordering) without matching value. Scripts validate inputs themselves (`: "${adr:?need adr}"` in Bash; explicit checks in TS).

A generic session-state mechanism (first-class iteration counter, key-value store) was deferred. Scripts that need such state maintain it in `$LOOPX_TMPDIR`.

## Decision

### 1. `LOOPX_TMPDIR` — run-scoped temporary directory

For each `loopx run` (CLI) or `run()` / `runPromise()` (programmatic) invocation that reaches execution, loopx creates a unique temporary directory before the first child process spawns and injects its absolute path into every script's environment as `LOOPX_TMPDIR`.

**Location, naming, and mode.** The directory is created under Node's `os.tmpdir()` via `mkdtemp` with a `loopx-` prefix, mode `0700`. The exact name format beyond the prefix is implementation-defined. The tmpdir parent (the `TMPDIR` / `TEMP` / `TMP` values that `os.tmpdir()` reads) is captured on the same schedule as the inherited `process.env` snapshot: synchronously at the `runPromise()` call site for `runPromise()`, and at first `next()` alongside the rest of the pre-iteration sequence for `run()` and the CLI. `TMPDIR` / `TEMP` / `TMP` entries in env files or `RunOptions.env` reach spawned scripts but do not mutate loopx's own `process.env` and therefore do not redirect the tmpdir parent. Because the tmpdir parent is captured on the same schedule as inherited env, a synchronous `process.env.TMPDIR = …` mutation after `runPromise()` returns does not affect the tmpdir parent; callers who need a specific parent for `runPromise()` set `TMPDIR` in loopx's inherited environment before the call. For `run()`, mutations to `process.env.TMPDIR` after `run()` returns but before the first `next()` **are** observed, for the same reason that inherited-env mutations in that window are observed.

**Scope and lifecycle.**

- **Created:** once per run, after the pre-iteration sequence (discovery, env-file loading, target resolution, version check) and immediately before the first child spawns. Pre-spawn failures, `-n 0` / `maxIterations: 0` early exits, and aborts observed before tmpdir creation do not create a tmpdir.
- **Shared across iterations.** All scripts — starting target, intra- and cross-workflow `goto` destinations, loop resets — observe the same `LOOPX_TMPDIR` value.
- **Persisted within the run.** Files written by one script remain visible to later scripts.
- **Concurrent runs are isolated.** Each run receives its own distinct directory.

**Cleanup.** loopx runs cleanup on every terminal outcome of a run that reached tmpdir creation:

- Normal completion (`stop: true` from a script, or `-n` / `maxIterations` reached).
- Non-zero script exit; invalid `goto` target; missing workflow or script during `goto` resolution.
- Child launch / spawn failure after tmpdir creation (discovered script removed or renamed per §5.1; runtime rejection of a child env entry from any env tier, most reliably an entry whose name or value contains an embedded NUL byte; `exec` failure; any other pre-first-line-of-user-code spawn-path error).
- SIGINT / SIGTERM to loopx — after any active child process group exits (per §7.3, including SIGKILL escalation), before loopx exits with the signal's code. If no child is active, cleanup runs immediately.
- Programmatic `AbortSignal` abort — after any active child process group exits per §9.1, before the generator throws or the promise rejects.
- Consumer-driven cancellation under `run()` (`break` from `for await`, `.return()` / `.throw()` after first `next()`) — after terminating any active child per §9.1, before the generator settles.

For `run()`, cleanup on normal completion is guaranteed only once the generator is driven to settlement (`{ done: true }`, `.return()`, `.throw()`, or a `for await` loop that runs to completion). The final yielded `Output` is not itself settlement; a caller that consumes the final output and abandons the generator may leak the tmpdir.

Cleanup does **not** run when loopx itself is killed via SIGKILL or the host crashes; leaked tmpdirs are expected to be reaped by OS temp-cleaning policy. loopx does not reap stale tmpdirs at startup.

**Cleanup idempotence.** loopx performs at most one cleanup attempt per created `LOOPX_TMPDIR` over the lifetime of the run. Racing terminal triggers (SIGTERM during cleanup of a prior SIGINT; abort concurrent with consumer `.return()` / `.throw()`; non-zero exit concurrent with abort; abort after final yield concurrent with a consumer `.throw()`; child spawn failure concurrent with abort; abort concurrent with tmpdir creation failure; etc.) do not start a second cleanup attempt.

**Terminal-outcome precedence.** The selection of which terminal outcome is surfaced is governed by explicit precedence rules elsewhere in this ADR where they apply — the pre-first-`next()` consumer-cancellation carve-out, the abort-wins-over-pre-iteration-failures rule, the CLI pre-iteration signal-wins rule, and the abort-after-final-yield rule. Except where an explicit precedence rule applies, the first terminal trigger observed by loopx determines the surfaced outcome among genuinely racing triggers. "Observed by loopx" refers to the point at which the trigger enters loopx's own tracking (signal handler, abort listener, child-exit callback, consumer interaction) — not OS-level delivery order. Idempotence and warning cardinality are independent of which outcome surfaces.

**Cleanup safety.** Cleanup is path-based and best-effort — not a sandbox against actively racing same-user processes. loopx captures an identity fingerprint (e.g., device/inode pair) of the created directory at creation time, before any child is exposed to the directory. At cleanup time, loopx `lstat`s the `LOOPX_TMPDIR` path and dispatches on the outcome:

1. **Path no longer exists (ENOENT):** no-op.
2. **Path is a symlink:** unlink the symlink entry; do not follow the target.
3. **Path is a regular file, FIFO, socket, or other non-directory non-symlink:** leave in place with a stderr warning. Unlinking would risk mutating unrelated data (hard-link `nlink` decrement, or data renamed into the path with `nlink == 1`).
4. **Path is a directory whose identity matches the recorded identity:** recursively remove. Symlink entries encountered during the walk are unlinked but not traversed, so symlinks pointing outside the tmpdir do not collateral-delete their targets.
5. **Path is a directory whose identity does not match:** leave in place with a stderr warning. loopx does not recursively remove a directory it did not create.

If the top-level `lstat` fails for any reason other than ENOENT, the `unlink` in case 2 fails, or the recursive removal in case 4 fails, loopx emits a single stderr warning and makes no further changes. Per-cleanup-attempt warning cardinality is at most one; combined with cleanup idempotence this bounds every run to at most one cleanup warning. The warning text format — including whether to identify a category, how to render the affected path, and how to describe the underlying errno — is implementation-defined. Warnings do not affect the CLI exit code, the generator outcome, or the promise rejection reason.

A script that removes or renames its tmpdir during the run defeats automatic cleanup of the moved directory; loopx does not chase renamed tmpdirs. loopx does not detect or avoid mount points inside `$LOOPX_TMPDIR`; scripts that mount into `$LOOPX_TMPDIR` take responsibility for unmounting before cleanup. A race-resistant guarantee against a same-user process actively mutating the path during cleanup (fd-relative `openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`) is out of scope for v1.

**Creation order.** loopx creates the directory with `mkdtemp`, captures its identity fingerprint, then verifies or secures mode `0700` — in that order. The path is not exposed to any child process before all three steps complete successfully. This order is normative because it determines what cleanup-safety guarantee applies if a creation step fails.

**Creation failure.** If any step fails, loopx does not spawn any child; the CLI exits `1`, `run()` throws on first iteration, `runPromise()` rejects. The cleanup performed before surfacing the creation error depends on which step failed:

- **`mkdtemp` itself fails:** no path exists, so no cleanup is needed.
- **Identity capture fails (after `mkdtemp` succeeded):** loopx attempts a single non-recursive `rmdir` on the path. This succeeds on an empty directory (the expected state immediately after `mkdtemp`) and fails without side effects on a symlink, non-directory, or non-empty directory. Without a recorded identity fingerprint, this remains outside the race-resistant guarantee: a same-user process that swapped the original `mkdtemp` directory for a different empty directory before the `rmdir` would have that replacement removed. loopx accepts this narrow trade-off as the cost of cleaning up the common identity-capture-failure case. If `rmdir` fails, loopx emits a single stderr warning and leaves the path in place.
- **Mode-securing fails (after `mkdtemp` and identity capture succeeded):** loopx runs the full cleanup-safety routine above, with the recorded identity fingerprint in hand.

Cleanup failure during creation-failure handling emits at most one warning but does not mask the original creation error. If a SIGINT/SIGTERM or `AbortSignal` abort arrives concurrently with a creation failure, the signal/abort wins and any partial directory is cleaned up under the same rules.

**Programmatic settlement and pre-iteration precedence.** The rules below refine SPEC §9.1's existing "abort always produces an error, even if it occurs before the first `next()` call" through the full pre-iteration sequence.

- **Pre-first-`next()` consumer cancellation on `run()` wins over all pre-iteration state.** A `.return(value)` or `.throw(err)` call as the consumer's *first* interaction with a generator returned by `run()` settles the generator per standard JS async-generator semantics; the loop body is never entered. No pre-iteration step runs and no pre-iteration error is surfaced: captured call-time option-snapshot errors (invalid `options` values, invalid `options.signal`, non-string `options.cwd` / `options.envFile`, invalid `options.maxIterations`, invalid `options.env` shape, throwing option-field getters, throwing `options.env` enumerable getters or proxy traps), invalid `target` values and target-syntax validation, `.loopx/` discovery failures, env-file loading failures, target resolution failures, and tmpdir creation failures are all suppressed. A `signal` captured at call time is not consulted; even an already-aborted signal does not surface the abort error through this path. This applies only to `run()`; `runPromise()` has no equivalent carve-out. Once the consumer's first interaction is `next()`, the body is entered and captured state is observed.
- **Once a usable `AbortSignal` has been captured, abort wins over all other pre-iteration failures on the same call.** A signal that was already aborted at call time, or that aborts during pre-iteration before the first child spawn, displaces all other pre-iteration failure modes (captured option-snapshot errors, target argument / target syntax validation, `.loopx/` discovery, env-file loading, target resolution, tmpdir creation). An invalid `options` value or non-`AbortSignal`-compatible `options.signal` captures no signal and does not enter this pathway — they are ordinary option-snapshot errors. Version checking is not a failure mode per SPEC §3.2, so it does not compete.
- **`options.signal` is read before other `RunOptions` fields.** This guarantees an already-aborted signal is captured before any other option-field read can produce a snapshot exception. The order of the remaining option fields (`options.env`, `options.cwd`, `options.envFile`, `options.maxIterations`) is implementation-defined. Each option field is read at most once per call, and a throwing getter or proxy trap is not re-invoked to retry.
- **Project-root-blocking and envFile-path-blocking option errors surface before the four project-root-dependent failures.** A non-string `options.cwd`, throwing `options.cwd` getter, non-string `options.envFile`, or throwing `options.envFile` getter must surface before `.loopx/` discovery errors, env-file loading errors, target resolution errors, or tmpdir-creation errors — otherwise an unrelated non-options error detected under a garbage project root would obscure the actual cause. The pinning is scoped to that set only; relative priority between these pinned option errors and target argument / target syntax validation is implementation-defined. Relative priority among the remaining non-abort pre-iteration errors (invalid `options`, invalid `options.signal`, invalid `options.maxIterations`, invalid `options.env` shape and snapshot-time throws, target argument / target syntax validation, and the four project-root-dependent failures) is implementation-defined.
- **CLI signal precedence during pre-iteration.** SIGINT / SIGTERM *observed by loopx's installed pre-iteration signal handler* wins over non-signal pre-iteration failures not yet surfaced at that point: loopx cleans up any partial tmpdir and exits with `128 + signal number` regardless of which non-signal pre-iteration step (target syntax validation, `.loopx/` discovery, env-file loading, target resolution, tmpdir creation) would otherwise have failed. The contract is phrased around handler observation, not OS-level delivery — implementations are not required to make pre-iteration async or poll for pending signals between steps. Signal-wins precedence begins only after the `run -h` / `--help` short-circuit and usage-level argument parsing have completed *and* loopx has installed its pre-iteration signal handlers. Signals delivered in the pre-handler-installation window (process startup, argument parsing, help short-circuit) are outside this precedence: either the runtime applies POSIX default signal disposition and terminates the process, or parsing completes and the parser-level usage error surfaces. Both outcomes are conforming.
- **Abort after the final yielded `Output` still wins until generator settlement.** An `AbortSignal` that aborts after the final `Output` has been yielded (via `stop: true` or `maxIterations` reached) but before the generator settles via `{ done: true }` produces the abort error on the next generator interaction — `g.next()`, `.return()`, or `.throw()`. Normal completion is fixed only at settlement, not at the final yield. Cleanup runs before the abort error is surfaced.
- **The CLI retains SPEC §7.1's concrete pre-iteration ordering.** The programmatic-API precedence rules above are scoped to `run()` / `runPromise()`. CLI tests may assert §7.1's documented order.

### 2. `RunOptions.env`

`RunOptions` gains an optional `env` field:

```typescript
interface RunOptions {
  maxIterations?: number;
  envFile?: string;
  signal?: AbortSignal;
  cwd?: string;
  env?: Record<string, string>;
}
```

- **Shape.** `env` must be omitted, `undefined`, or a non-null, non-array, non-function object whose own enumerable string-keyed entries all have string values. Symbol-keyed, non-enumerable, and inherited properties are ignored. Invalid shapes (null, array, non-object, function, or an entry with a non-string value) are captured at call time and surfaced via the standard pre-iteration error path.
- **Lifetime.** Entries are captured synchronously at call time as a shallow copy — loopx reads the supplied object's own enumerable string-keyed properties once. The capture runs at the `run()` / `runPromise()` call site but never throws at the call site; any exception raised during the snapshot (a `Proxy` `ownKeys` trap that throws, a throwing enumerable getter, a throwing `options.env` getter on the options object) is captured and surfaced through the standard pre-iteration error path. Mutating the original object after `run()` / `runPromise()` returns has no effect on the running loop.
- **Merge position.** `env` entries merge into the child environment *after* global and local env-file loading and *before* loopx-injected protocol variables (`LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, `LOOPX_WORKFLOW_DIR`, `LOOPX_TMPDIR`). A `RunOptions.env` entry overrides same-named values from `-e`, the global loopx env file, and inherited `process.env`, and is overridden by protocol variables.
- **Applies to every script in the run.** Starting target, intra- and cross-workflow `goto` destinations, and loop resets all receive the same `env` additions.
- **No CLI surface.** Programmatic-only. CLI callers pass per-run values via the shell env prefix (`key=value loopx run …`), which flows through inherited `process.env`, or via `-e` / `loopx env set` for file-based config.
- **No name validation beyond string-to-string.** loopx does not enforce the POSIX `[A-Za-z_][A-Za-z0-9_]*` name pattern, does not reject `LOOPX_*` keys (they are silently overridden by protocol injection), does not reject `=` or empty-string names, and does not reject NUL bytes. Runtime-level rejections — most reliably a name or value containing an embedded NUL byte — surface as child launch / spawn failures at spawn time (any env tier may be the source; the runtime does not distinguish tiers). `RunOptions.env` shape validation (invalid object types, non-string values, snapshot-time throws) is independent of spawning and surfaces on the programmatic API under `maxIterations: 0` just as under any other iteration count. The CLI has no `RunOptions.env` to validate, so this shape-validation path does not apply to `-n 0`; `-n 0` does still validate env files.
- **`LOOPX_DELEGATED` is not a script-protocol-protected variable.** SPEC §3.2 sets `LOOPX_DELEGATED` only on the inner loopx process during project-root delegation, not on every script spawn. This ADR does not add `LOOPX_DELEGATED` to the per-spawn protected tier; env files, inherited `process.env`, and `RunOptions.env` may supply it to spawned scripts. The SPEC §3.2 startup behavior is unchanged.

**Invalid option values.** `options` must be omitted, `undefined`, or a non-null, non-array, non-function object. `options.signal`, when present and not `undefined`, must be an `AbortSignal`-compatible object exposing a readable `aborted` and an `addEventListener('abort', listener)` method that is callable and returns without throwing when loopx invokes it to register its abort listener. `options.cwd` and `options.envFile`, when present and not `undefined`, must be strings. `options.maxIterations` retains its existing SPEC §9.5 validation. Non-conforming values — including throwing getters on the options object — are captured at call time and surfaced via the standard pre-iteration error path rather than escaping at the call site. `removeEventListener` is not part of the `AbortSignal`-compatibility contract: loopx does not require its presence, and whether loopx invokes it on settlement is implementation-defined.

**Duck-typed signal reentrancy.** For non-`AbortSignal` duck-typed signals: if `addEventListener` synchronously invokes the registered listener during registration, or if `aborted` is observed as `true` at any point during call-time capture, loopx treats the signal as aborted. Real `AbortSignal` instances passed in an already-aborted state must be observed as aborted regardless of implementation latitude. Other reentrant behavior — `aborted` transitioning between reads within the same call, ordering between reading `aborted` and registering the listener for duck-typed signals — is implementation-defined.

**Inherited `process.env` snapshot.** loopx snapshots the caller's `process.env` once per run; mutations between spawns do not propagate. For `run()`, the snapshot is lazy — captured on the first `next()` call alongside the rest of the pre-iteration sequence, so mutations between `run()` returning and first `next()` **are** observed. For `runPromise()`, the snapshot is captured synchronously at the `runPromise()` call site, before `runPromise()` returns; synchronous mutations after return are not observed. Callers who need a deterministic per-run value under either API should use `RunOptions.env`, which sits above inherited env in §8.3 and is captured at call time.

Global env file path resolution (`$XDG_CONFIG_HOME/loopx/env`, with the documented `HOME`-based fallback) reads `XDG_CONFIG_HOME` / `HOME` from the inherited environment on the same schedule as the inherited-env snapshot (lazy for `run()`, at call time for `runPromise()`). `RunOptions.env` does not affect global-env-file path resolution because `env` merges after env-file loading.

### 3. Project-root cwd

Every child process loopx spawns for script execution uses `LOOPX_PROJECT_ROOT` as its working directory. This replaces SPEC §6.1's current rule that scripts run with the workflow directory as cwd.

- **CLI.** The CLI's invocation cwd is the project root (SPEC §3.2). The spawned script cwd becomes the same path — unchanged from the CLI user's terminal perspective but different from the child script's perspective.
- **Programmatic.** `RunOptions.cwd` (or `process.cwd()` at call time when omitted) specifies both the project root and the script execution cwd. A relative `RunOptions.cwd` is resolved via `path.resolve(process.cwd(), options.cwd)` once at call time; an absolute value is used unchanged. No `realpath` / canonicalization is applied. SPEC §9.5's current "cwd does not control script execution cwd" disclaimer is dropped.
- **Applies to every script in the run.** Starting target, intra- and cross-workflow `goto` destinations, and loop resets all receive the same cwd.
- **Scripts remain free to change directory.** `cd` / `process.chdir()` changes are scoped to that child process; later spawns always start at project-root cwd.
- **Script module resolution is unaffected by cwd.** Node ESM, tsx, and Bun resolve bare specifiers and relative imports from the importing module's file path, not cwd. SPEC §3.3's current claim that closer-`node_modules` precedence is "a natural consequence of running scripts with the workflow directory as cwd" is incorrect and retired by this ADR. The custom Node/tsx resolve hook (SPEC §3.3) must preserve standard precedence: it must first attempt standard file-relative resolution (via `defaultResolve` delegation or an equivalent file-path walk-up) and fall back to the CLI package exports only when no `node_modules/loopx` is found by standard resolution. Note that Node's default main-module handling applies `realpath` to the main module's file path; when the absolute discovery-time entry path traverses any symlink (workflow directory entry, entry script file, or intermediate ancestor), the resolver walks up from the canonical target rather than the symlink-preserving spelling, and workflow-local `node_modules/loopx` on the logical path is not guaranteed to win even with a correctly delegating hook.
- **No canonicalization.** loopx does not apply `realpath` beyond what the runtime already provides. For the CLI, `LOOPX_PROJECT_ROOT` is exactly the string returned by loopx's own `process.cwd()` at invocation; loopx does not consult `$PWD`. On POSIX systems where `getcwd(3)` canonicalizes, a CLI invocation from a symlinked directory yields a canonical `LOOPX_PROJECT_ROOT`. For the programmatic API, `LOOPX_PROJECT_ROOT` is `RunOptions.cwd` (with `path.resolve` applied only if relative) or `process.cwd()` at call time. `path.resolve` is a pure string operation; symlinked components supplied in `RunOptions.cwd` are preserved verbatim. Callers who need a canonical realpath compute one themselves.
- **Directory identity vs. string spelling.** loopx controls the child's *effective working directory* by spawning with `LOOPX_PROJECT_ROOT` as cwd (same device/inode as `LOOPX_PROJECT_ROOT`) but does not control the *string spelling* that runtime cwd APIs report. Under a symlinked project root, Node/Bun's `process.cwd()` typically canonicalizes via `getcwd(3)` while `LOOPX_PROJECT_ROOT` preserves the symlinked spelling; both name the same directory. Bash `$PWD` and bare `pwd` are shell-maintained and inherit loopx's own `PWD`, so byte-for-byte equality between Bash `$PWD` / bare `pwd` and `LOOPX_PROJECT_ROOT` is not a loopx guarantee in any regime. Scripts that need the loopx-chosen spelling read `LOOPX_PROJECT_ROOT`; scripts that need the kernel cwd spelling use `process.cwd()` (JS/TS) or `pwd -P` / `/bin/pwd` (Bash); scripts that need filesystem identity use `stat(".")`.
- **`PWD` is not a protocol variable.** loopx neither sets nor unsets `PWD` in the child environment; whatever value loopx's own process holds is inherited unchanged. A caller who wants a specific `PWD` in the child passes it via `RunOptions.env` or inherited env, subject to Bash's standard POSIX inheritance behavior. `PWD` is outside the §8.3 protocol-variable tier and outside the §13 reserved list.
- **Breaking change vs. current SPEC §6.1.** Scripts using `./foo` or `readFileSync("./foo")` to reach workflow-local files break. Migration: `"$LOOPX_WORKFLOW_DIR/foo"` (Bash) or `resolve(process.env.LOOPX_WORKFLOW_DIR, "foo")` (JS/TS). Existing workflow scripts in this repo must be audited and migrated as part of implementing this ADR; notably `.loopx/shared/dispatch.sh` uses cwd-relative references (`./send-telegram.sh`, `./send-codex.sh`, `./node_modules/.bin/tsx ./send-api.ts`, `./node_modules/.bin/tsx ./send-batch.ts`) that require migration, and the `.loopx/shared/` helpers' use of workflow-local `node_modules/` via `./node_modules/.bin/tsx` needs to be replaced with a `$LOOPX_WORKFLOW_DIR`-rooted path.

### 4. `LOOPX_WORKFLOW_DIR` injection

loopx injects `LOOPX_WORKFLOW_DIR` into every script's environment, set to the absolute path of the workflow directory containing the currently-spawned script.

- **Value.** The directory portion of the cached absolute discovery-time script path for the currently-spawned script — the same path by which loopx spawns the script (see "Scripts are invoked by absolute discovery-time path" below). In the v1 layout this normally corresponds to `"$LOOPX_PROJECT_ROOT/.loopx/$LOOPX_WORKFLOW"`, but the injected form is derived from the cached discovery path rather than recomposed from those components, so it remains consistent with the spawn path under trailing-slash variations, symlinked `.loopx/<workflow>` entries, and any lexical normalization the implementation applies during discovery. The injected form is authoritative; scripts should prefer it over re-deriving the path.
- **Per-spawn refresh — cross-workflow `goto` semantics.** `LOOPX_WORKFLOW_DIR` tracks the currently-spawned script's containing workflow, refreshed alongside `LOOPX_WORKFLOW` across starting-target spawn, intra-workflow `goto`, cross-workflow `goto`, and loop reset. A cross-workflow `goto` A → B sets `LOOPX_WORKFLOW_DIR` to B's directory before the B-side spawn. Deeper chains (A → B → C) each observe their own workflow's directory.
- **Stable within a single script execution.** A top-level workflow script and a sibling helper it sources or imports both read the same `LOOPX_WORKFLOW_DIR` — the env var is injected once per child spawn and inherited by the whole process.
- **Scripts are invoked by absolute discovery-time path.** loopx spawns each script using the absolute path by which it was discovered under `.loopx/` (e.g., `bash /project/.loopx/review-adr/index.sh`). For Bash, `$0` is an absolute path; because Bash does not canonicalize `$0`, `$(dirname "$0")` equals `LOOPX_WORKFLOW_DIR` byte-for-byte regardless of whether symlinks appear anywhere in the entry path (workflow directory entry, entry script file, or intermediate component). This Bash equality is normative. For JS/TS, `dirname(fileURLToPath(import.meta.url))` equals `LOOPX_WORKFLOW_DIR` only when the absolute entry path is symlink-free in every component; Node applies `realpath` to the main module's file path by default, and Bun's behavior is not specified to match. `LOOPX_WORKFLOW_DIR` is authoritative for JS/TS code that needs the discovery-time workflow path.
- **Symlink behavior.** `LOOPX_WORKFLOW_DIR` is derived from the project root plus the `.loopx/<workflow>` entry name as discovered, not via `realpath`. A symlinked `.loopx/<workflow>` directory entry, a symlinked `.loopx` directory, a symlinked entry script file, or a symlinked project-root ancestor (via `RunOptions.cwd`) all preserve their discovered spelling in `LOOPX_WORKFLOW_DIR`.
- **Protocol-variable tier.** `LOOPX_WORKFLOW_DIR` joins `LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, and `LOOPX_TMPDIR` in the top precedence tier of SPEC §8.3. User-supplied values in any lower tier are silently overridden.
- **Not a cross-workflow rendezvous.** Scripts handing data across a cross-workflow `goto` use `$LOOPX_TMPDIR` (preferred) or a shared fixed location like `$LOOPX_PROJECT_ROOT/.loopx/shared/…`.

### 5. CLI grammar — v1 scope

Outside the existing `run -h` / `--help` short-circuit (SPEC §4.2), `loopx run` does not accept a named-argument, key-equals-value, or per-run-parameter CLI syntax. A `name=value` token in the positional tail is not interpreted as a named argument: a second positional after the target is rejected as "more than one positional" (SPEC §4.1), and a sole positional `name=value` is parsed as a target string that fails target validation because `=` is not in the `[a-zA-Z0-9_][a-zA-Z0-9_-]*` pattern.

v1 also does not accept `--` as an end-of-options marker in `run`'s argument list (outside `-h` / `--help`): wherever it appears, it is rejected as an unrecognized flag / usage error under the existing §4.1 / §4.2 rejection rules. This follows from the v1 decision not to adopt a named-argument tail — the rejection is parser simplification, not a permanent product stance. A later ADR that adopts a first-class CLI named-argument surface may revisit both the tail and `--` support.

The `run -h` / `--help` short-circuit is preserved exactly as in SPEC §4.2: when `-h` / `--help` appears anywhere in the `run` argument list, loopx shows run help, exits `0`, and ignores all other run-level arguments unconditionally (including `--` in any position, extra positionals, duplicate or invalid `-n` / `-e`, unrecognized flags, and any `name=value` tokens). The sole CLI surface for per-run values is the shell env prefix (`key=value loopx run <target>`), which flows through inherited `process.env` at the §8.3 inherited-env tier. Callers who want file-backed precedence above inherited env use `-e <path>` or `loopx env set`.

## Consequences

- **Workflow directories stop accumulating temp files.** Existing workflows migrate `.feedback.tmp` / `.claude-output.tmp` / `.answer.tmp` / `.prompt.tmp` / `ralph`'s `.iteration.tmp` into `$LOOPX_TMPDIR/…` and drop manual `rm -f` branches.
- **Cleanup is automatic on failure.** Loops that error out, are signaled, or are aborted no longer leak scratch files. Callers who relied on lingering `.tmp` files for post-mortem inspection must write those outside `$LOOPX_TMPDIR` (e.g., under `$LOOPX_PROJECT_ROOT`).
- **Cross-run state is the caller's responsibility.** `LOOPX_TMPDIR` never persists across runs. Workflows needing cross-run state use project-root-relative storage. `ralph`'s `.iteration.tmp` is intra-run state and migrates to `$LOOPX_TMPDIR/iteration`.
- **Concurrent runs are safe.** Parallel `loopx run` invocations do not clobber each other's scratch files.
- **Scripts drop `$LOOPX_PROJECT_ROOT`-prefix boilerplate.** `ROOT="$LOOPX_PROJECT_ROOT"; cat "$ROOT/SPEC.md"` becomes `cat SPEC.md`. `$LOOPX_PROJECT_ROOT` remains injected for scripts that `cd` elsewhere or hand paths to tools that don't inherit cwd.
- **Tools invoked from scripts get project-root context by default.** `git`, `claude`, `codex`, and similar tools operate against the whole repo without scripts having to `cd` first. Scripts that want workflow-scoped tool behavior `cd "$LOOPX_WORKFLOW_DIR"` explicitly.
- **`RunOptions.cwd` semantics simplify.** Previously `cwd` set only the project root; now it sets both the project root and the script execution cwd.
- **Workflow-local cwd-relative references break.** Scripts using `./foo` now resolve against the project root. Migration: `"$LOOPX_WORKFLOW_DIR/foo"` (Bash) or `resolve(process.env.LOOPX_WORKFLOW_DIR, "foo")` (JS/TS). In this repo, `.loopx/shared/dispatch.sh` uses cwd-relative references and must be migrated as part of implementing this ADR.
- **Workflow-local assets reachable in one token.** `cat "$LOOPX_WORKFLOW_DIR/prompt.md"` replaces `$(dirname "$0")` / `import.meta.url` tricks.
- **Programmatic callers get per-run env without mutating `process.env`.** `RunOptions.env` is a clean shallow copy scoped to the call.
- **CLI parameterization uses existing shell idioms.** `adr=0003 loopx run review-adr` continues to work at the inherited-env tier — overridden by both env files and `RunOptions.env`. loopx does not introduce a dedicated CLI override tier above `-e`.
- **`RunOptions.env` fills a different gap than `-e`.** It lets programmatic callers pass per-run values without mutating `process.env`, without writing an on-disk file, and layered dynamically above `RunOptions.envFile`.
- **Migration is manual but straightforward.** Existing workflows (1) replace in-workflow `.tmp` paths with `$LOOPX_TMPDIR/…` and drop `rm -f` cleanup, (2) take per-run parameters via shell env prefix, (3) optionally drop the `ROOT=…` header and `$ROOT/` prefixes. No automated migration tooling is provided.

## Rejected / Deferred Alternatives

- **First-class CLI named-argument syntax (e.g., `loopx run <target> -- name=value`).** Deferred. Shell env-var prefixing serves the v1 use cases; a dedicated parser would add substantial complexity without matching value.
- **POSIX `--` end-of-options marker on `run` outside help.** Not supported in v1 as a consequence of the "no named-argument tail" decision. A later ADR may revisit alongside a named-argument surface.
- **Named-argument schema.** Deferred. Scripts validate their own inputs.
- **First-class iteration counter.** Deferred. Scripts maintain one in `$LOOPX_TMPDIR/iteration`.
- **Stronger race-resistant tmpdir cleanup (`openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`).** Deferred. v1 uses path-based best-effort cleanup with identity matching.
- **Treating `PWD` as a protocol variable.** Rejected. Scripts that need a reliable cwd already have `process.cwd()` and `pwd -P`.
- **`LOOPX_PROJECT_ROOT` canonicalization / `realpath`.** Rejected. Callers compute a realpath themselves if needed.

- **§7.3 — Signal Handling.** On SIGINT / SIGTERM, `LOOPX_TMPDIR` cleanup runs after any active child process group has exited (per the existing grace period, including SIGKILL escalation) and before loopx exits with the signal's code. When no child is active, cleanup runs immediately. SIGINT / SIGTERM observed by loopx's installed pre-iteration signal handler wins over non-signal pre-iteration failures not yet surfaced at that point; loopx cleans up any partial tmpdir and exits with `128 + signal number`. Signals delivered before handler installation (process startup, argument parsing, help short-circuit) are outside this precedence — POSIX default disposition and parser-level usage-error exit `1` are both conforming outcomes.
- **§7.4 — Run-scoped temporary directory lifecycle (new subsection).** Introduce a new §7.4 describing `LOOPX_TMPDIR` lifecycle end-to-end, covering:
  - **Parent selection.** Tmpdir parent is `os.tmpdir()` evaluated in loopx's own process environment on the same schedule as the inherited `process.env` snapshot (synchronous at the `runPromise()` call site; at first `next()` for `run()`; pre-iteration for the CLI). `TMPDIR` / `TEMP` / `TMP` entries in env files or `RunOptions.env` reach spawned scripts but do not redirect the tmpdir parent.
  - **Location, naming, and mode.** Created under the selected parent via `mkdtemp` with a `loopx-` prefix, mode `0700`. Exact name format beyond the prefix is implementation-defined.
  - **Creation order.** `mkdtemp` → identity fingerprint capture → mode verification / securing, in that order, before any child is exposed to the path. Per-step failure handling determines cleanup:
    - `mkdtemp` failure: no path exists; no cleanup.
    - Identity capture failure: single non-recursive `rmdir`, outside the race-resistant guarantee (see below).
    - Mode-securing failure: full identity-fingerprint cleanup-safety routine.
  - **Identity-fingerprint cleanup safety.** Cleanup `lstat`s the path and dispatches: ENOENT → no-op; symlink → unlink entry, do not follow target; non-directory non-symlink → leave in place with a warning; directory with matching identity → recursively remove, unlinking (not traversing) symlink entries encountered during the walk; directory with mismatched identity → leave in place with a warning. Top-level `lstat` failures (other than ENOENT), case-2 `unlink` failures, and case-4 recursive-removal failures emit a single stderr warning and halt further changes.
  - **Renamed-away and mount-point behavior.** Scripts that remove or rename the tmpdir during the run defeat automatic cleanup of the moved directory; loopx does not chase renamed tmpdirs. loopx does not detect or avoid mount points inside `$LOOPX_TMPDIR`; scripts that mount into the tmpdir are responsible for unmounting before cleanup.
  - **Race-resistance scope.** Path-based best-effort cleanup with identity matching is guaranteed; a race-resistant guarantee against a same-user process actively mutating the path during cleanup (fd-relative `openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`) is out of scope for v1.
  - **No stale-tmpdir reaping at startup.** Leaked tmpdirs from prior SIGKILL / host-crash terminations are expected to be reaped by OS temp-cleaning policy.
  - **Settlement-based cleanup on `run()` normal completion.** Cleanup is guaranteed only once the generator is driven to settlement; a caller that consumes the final yielded `Output` and abandons the generator may leak the tmpdir.

## Test Recommendations

Non-exhaustive — these highlight a few edge cases easy to overlook. A full test plan belongs in TEST-SPEC.md after this ADR is accepted.

- **Tmpdir cleanup under racing terminal triggers.** Verify at most one cleanup attempt — and therefore at most one cleanup warning — for a run when multiple terminal triggers arrive (e.g., SIGTERM followed by SIGINT during cleanup; abort concurrent with consumer `.return()` / `.throw()`; non-zero exit concurrent with abort; abort observed after final yield followed by `.throw()`).
- **Cleanup safety under path replacement.** Symlink replacement unlinked without following the target; non-directory non-symlink replacement left in place; mismatched-identity directory left in place; renamed-away tmpdir not chased; hard-link `nlink` of a shared inode unchanged after cleanup.
- **Pre-first-`next()` consumer cancellation carve-out.** A `.return()` / `.throw()` as the consumer's first interaction on a `run()` generator suppresses every pre-iteration error (captured option-snapshot, invalid / target-syntax `target`, missing `.loopx/` / discovery, env-file loading, target resolution, tmpdir creation) and any already-aborted signal. The tight scope matters: the same errors surface on first `next()`, and a `.return()` / `.throw()` after first `next()` follows normal consumer-driven-cancellation rules.
- **Abort wins over pre-iteration failures when a usable signal is captured.** An already-aborted `AbortSignal` passed alongside an invalid target, missing `.loopx/`, invalid `env` shape, or a throwing option-field getter read *after* `options.signal` surfaces the abort error on first `next()`. A non-compatible `options.signal` captures no signal and surfaces as an ordinary option-snapshot error, not as an abort.
- **Abort after final yield.** A consumer that manually iterates to the final yielded `Output` and then aborts before calling `next()` again observes the abort error on the subsequent `next()` — not `{ done: true }`. Cleanup runs before the abort error is surfaced.
- **Inherited-env snapshot timing split.** For `run()`, a `process.env` mutation between `run()` returning and the first `next()` **is** observed by every child; later mutations between iterations are not. For `runPromise()`, the snapshot is taken before return, so a synchronous `process.env` mutation after `runPromise()` returns is not observed. `RunOptions.env` is captured at call time under both APIs.
- **Tmpdir-parent snapshot timing.** The tmpdir parent is captured on the same schedule as the inherited-env snapshot: a synchronous `process.env.TMPDIR = …` mutation after `runPromise()` returns does not affect the tmpdir parent for that run, while a mutation between `run()` returning and first `next()` does. `TMPDIR` / `TEMP` / `TMP` entries in env files or `RunOptions.env` do not redirect the tmpdir parent under either API.
- **`RunOptions.env` precedence and shape validation.** Entries override same-named values from `-e`, the global env file, and inherited `process.env`, and are overridden by protocol variables. Invalid shapes (`env: null`, `env: []`, non-string values, function values) reject under `maxIterations: 0` without a spawn; the CLI has no `RunOptions.env` surface, so `-n 0` does not exercise this shape-validation path. Runtime-level rejections (NUL-byte entries) surface only at spawn time, so they require `maxIterations >= 1` programmatically or `-n >= 1` on the CLI.
- **CLI grammar.** `loopx run -- <target>`, `loopx run -n 1 -- <target>`, `loopx run <target> -- name=value`, and `loopx run <target> name=value` each exit `1` as usage errors. `loopx run name=value` from a directory with `.loopx/` fails target validation; from a directory without `.loopx/` surfaces the discovery error first. `name=value loopx run <target>` succeeds (the shell absorbs the prefix). The `run -h` / `--help` short-circuit ignores `--`, extra positionals, and `name=value` tokens in any position.
- **Project-root cwd and workflow-dir injection.** The same effective working directory (device/inode) is observed across the starting target, `goto` destinations, and loop reset; project-root-relative references (`cat SPEC.md`) resolve without `$LOOPX_PROJECT_ROOT`. `LOOPX_WORKFLOW_DIR` refreshes per-spawn across cross-workflow `goto` A → B (the B-side script sees B's directory). Under a canonical, symlink-free entry path, Bash `$(dirname "$0")` equals `LOOPX_WORKFLOW_DIR` byte-for-byte; under symlinks in any component of the absolute entry path, the Bash equality still holds (Bash does not canonicalize `$0`) but JS/TS `dirname(fileURLToPath(import.meta.url))` equality is not required.
