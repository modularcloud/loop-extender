# ADR-0004: Run-Scoped Temporary Directory, Programmatic Env, and Project-Root CWD

**Status:** Proposed

---

## Scope of this ADR

- **Observable edge-case behavior is part of the SPEC contract.** This ADR intentionally specifies observable edge-case behavior — including the cleanup-warning category tokens and cardinality rule (§1 "Warning cardinality"), the option-snapshot one-pass-enumeration / single-read rule (§1 "Options snapshot ordering"), pre-first-`next()` async-generator cancellation precedence (§1 Programmatic API), abort-after-final-yield semantics (§1 Programmatic API), the Bash / JS-TS path-spelling guarantees for `$0` / `$(dirname "$0")` / `LOOPX_WORKFLOW_DIR` (§3, §4), and the closed set of pre-iteration failure modes. These are part of the public SPEC contract once this ADR is in "Spec Updated" status, and tests may assert on the token set, the cardinality rule, the one-read-per-field rule, the pre-first-`next()` carve-out, and the Bash `$(dirname "$0")` equality — subject to the "implementation-defined" caveats this ADR calls out explicitly (for example, the relative priority among non-abort programmatic pre-iteration failures, JS/TS `import.meta.url` path-spelling under symlinked entry paths, and `runPromise()` inherited-`process.env` snapshot timing relative to return). **Scope of the "implementation-defined by default" carve-out.** For behavior newly introduced or explicitly changed by this ADR, details not fixed here are implementation-defined. Existing SPEC guarantees outside the change surface of this ADR remain in force unless this ADR explicitly supersedes them; this ADR does not de-specify current SPEC behavior by omission. Future ADRs may extend the contract but must not retroactively change the tokens, cardinality rule, or one-read-per-field rule without explicit revision.
- **Intentional breaking change vs. current SPEC §6.1.** Scripts that rely on the workflow directory being their cwd (e.g., `./helper.sh`, `readFileSync("./config.json")`) break under this ADR. Migration is to use `$LOOPX_WORKFLOW_DIR` (Bash) or `process.env.LOOPX_WORKFLOW_DIR` (JS/TS) for workflow-local files, and project-root-relative paths (`SPEC.md`, `adr/…`) for project files. See §3 "Project-root cwd" for the full treatment and the Consequences section for the migration shape.

## Decision Summary

- **`LOOPX_TMPDIR`.** Each `loopx run` gets a private `os.tmpdir()`-based directory (mode `0700`), injected into every script's env. Shared across iterations, cleaned up on terminal outcomes (normal completion, error exit, signal, abort, consumer-driven generator cancellation). For the programmatic `run()` generator specifically, cleanup on normal completion is guaranteed only once the async generator is driven to settlement (`{ done: true }`, `.return()`, `.throw()`, or a `for await` loop that runs to completion); the final yielded `Output` is not itself settlement, so a caller that consumes the final output and abandons the generator without driving it further may leak the tmpdir. See §1 "Programmatic API" for the full treatment.
- **`RunOptions.env`.** A new optional `Record<string, string>` field on `RunOptions`, shallow-merged into the child env between env-file loading and loopx protocol variable injection.
- **Project-root cwd.** Every script runs with `LOOPX_PROJECT_ROOT` as its child-process working directory — not the workflow directory. Scripts that need the workflow directory read `$LOOPX_WORKFLOW_DIR` (see next bullet) and remain free to `cd` / `process.chdir()` mid-run. The CLI's project root is the invocation cwd (SPEC §3.2, unchanged); the programmatic API's project root is `RunOptions.cwd` or `process.cwd()` at call time (SPEC §9.5, unchanged). Supersedes SPEC §6.1's current "workflow directory as cwd" rule.
- **`LOOPX_WORKFLOW_DIR`.** A new injected protocol variable holding the absolute path of the currently-spawned script's workflow directory (e.g., `/project/.loopx/review-adr`). Refreshes per-spawn like `LOOPX_WORKFLOW`, so cross-workflow `goto` destinations, intra-workflow `goto` destinations, and loop resets each observe their own containing workflow's directory. Gives scripts a one-token reference to workflow-local assets (`cat "$LOOPX_WORKFLOW_DIR/prompt.md"`) without `$(dirname "$0")` tricks or `dirname(fileURLToPath(import.meta.url))` in JS/TS.
- **No CLI named-argument syntax.** Outside the existing `run -h` / `--help` short-circuit (SPEC §4.2), CLI callers who want to pass a per-run value use the standard shell env-var prefix (`key=value loopx run …`), which already flows through loopx's env inheritance. This reuses the existing inherited-environment tier — the lowest user-controlled tier in SPEC §8.3 — and does **not** introduce a new CLI override-precedence level above `-e` or the global env file. `-e` and `loopx env set` remain available for file-based config. A `name=value` token in the positional / argument tail is **never** interpreted as a named argument. Additionally, `loopx run` does **not** accept a `--` end-of-options marker in any position outside the help short-circuit: `--` is simply an unrecognized flag / usage error wherever it appears in `run`'s argument list when `-h` / `--help` is not present. The rejected forms therefore include: `loopx run <target> -- name=value` (rejected as unrecognized `--` / no supported `--` tail), `loopx run -- <target>` (rejected as unrecognized `--` — `--` is not supported as an end-of-options marker before the target either), `loopx run -n 1 -- <target>` (same — `--` between a flag and the target is rejected), `loopx run <target> name=value` (rejected as more than one positional per SPEC §4.1), and `loopx run name=value` (a `name=value` as the *sole* positional is parsed as a target string and fails target validation — the `=` character is not part of the `[a-zA-Z0-9_][a-zA-Z0-9_-]*` name pattern in SPEC §4.1 "Target validation"). The only CLI surface that treats `key=value` as a per-run value is the shell env prefix (`key=value loopx run <target>`), where the shell absorbs the prefix before loopx sees its arguments. The `run -h` / `--help` short-circuit in SPEC §4.2 is preserved exactly as written: when `-h` / `--help` is present anywhere in the `run` argument list, loopx shows run help, exits 0, and ignores all other run-level arguments unconditionally — including extra positionals, duplicate `-n` / `-e`, invalid `-n` / `-e` values, unrecognized flags (including `--` in any position), and any `name=value`-style tokens that would otherwise be rejected under this ADR's "no named-argument tail" or "no `--`" rules, or under SPEC §4.1 target validation.
- **Cleanup safety.** Path-based best-effort. loopx never follows symlinks during cleanup. If the tmpdir path was replaced after creation, loopx only recursively removes it when the path still names the originally created directory (identity-matched against a fingerprint captured at creation time). A symlink replacement is unlinked without following its target. Non-symlink file replacements (regular file, FIFO, socket, etc.) and mismatched-identity directory replacements are left in place with a warning — *except in the identity-not-captured creation-failure carve-out*: when the partial-directory cleanup path runs because identity capture itself failed (a `stat`/`lstat` on the freshly created `mkdtemp` directory raised), loopx attempts a single non-recursive `rmdir` on the path, which will succeed and remove whatever empty directory is at the path even if a same-user race has swapped the original `mkdtemp` directory for a different empty directory in the meantime. See §1 "Creation failure" for the full treatment. Not a sandbox against actively racing same-user processes.

A later ADR may revisit a first-class CLI named-argument surface, a named-argument schema, or stronger race-resistant cleanup (e.g., `openat` / `unlinkat`) if evidence of need accumulates.

## Context

Three pain points have emerged as workflows have grown.

1. **Passing intermediate data between scripts during a loop run.** Workflows like `review-adr`, `apply-adr`, and `spec-test-adr` stash hand-off files (`.feedback.tmp`, `.claude-output.tmp`, `.answer.tmp`, `.prompt.tmp`) either inside their own workflow directory or in a sibling `.loopx/shared/` directory — the latter for files that must survive a cross-workflow `goto`, since `$LOOPX_WORKFLOW` updates on the other side of the goto and workflow-local references like `$ROOT/.loopx/$LOOPX_WORKFLOW/…` therefore point at a different directory after the hop. Both patterns rely on `rm -f` on the happy path with no cleanup on error, signal, or unexpected termination; leaked files accumulate and are indistinguishable from legitimate workflow content. The `ralph` workflow has the same issue with its hand-maintained `.loopx/.iteration.tmp` counter — intra-run state that has been leaking into the `.loopx/` directory because there is no per-run scratch space to hold it.

2. **Per-run parameterization from programmatic callers.** Callers of `run()` / `runPromise()` currently have no way to inject per-invocation env vars without mutating `process.env` globally, which is racy under concurrent calls. The CLI already handles the parameterization case natively — `adr=0003 loopx run review-adr` works today because loopx inherits the caller's `process.env` into every spawned script — but the programmatic API has no equivalent.

3. **`$LOOPX_PROJECT_ROOT`-prefix boilerplate in every script.** SPEC §6.1 currently sets each script's cwd to its workflow directory (e.g., `.loopx/review-adr/`), so relative paths in scripts resolve against workflow-local files — almost never what a script wants. Nearly every script in `.loopx/` opens with `ROOT="$LOOPX_PROJECT_ROOT"` and prefixes every reference — `$ROOT/SPEC.md`, `$ROOT/adr/…`, `$ROOT/.loopx/shared/…` — to reach project files. Tools invoked from scripts (`git`, `claude`, `codex`) also want project-root cwd to operate on the whole repo rather than a workflow subdir. Scripts that forget to prefix silently read or write against the workflow directory, producing confusing failures. The §6.1 justification — "ensures relative imports and `node_modules/` resolve naturally" — is inaccurate: Node's ESM resolver, tsx, and Bun all resolve bare specifiers and relative imports from the importing file's directory, not cwd, so changing cwd does not affect module resolution.

An earlier draft of this ADR proposed a `loopx run … -- name=value` CLI surface with its own precedence tier and dedicated parser. It is intentionally not adopted: for the identified use case (passing a parameter like an ADR number into a script), shell env-var prefixing on the CLI is already sufficient, and the parser surface would add substantial implementation complexity (two-phase parsing, name grammar, duplicate detection, `LOOPX_*` reservation, precedence tier reordering) without matching value for v1. Scripts that need to validate a required value do so themselves (`: "${adr:?need adr}"` in Bash; `if (!process.env.adr) throw …` in TS).

A generic session-state mechanism (first-class iteration counter, key-value store) was considered but deferred. Scripts that need such state maintain it inside `$LOOPX_TMPDIR` — including the iteration counter, which is a trivial read/increment/write against a file there.

## Decision

### 1. `LOOPX_TMPDIR` — run-scoped temporary directory

For each `loopx run` (CLI) or `run()` / `runPromise()` (programmatic) invocation that reaches execution, loopx creates a unique temporary directory before the first child process spawns and injects its absolute path into every script's environment as `LOOPX_TMPDIR`.

#### Location, naming, and mode

The directory is created under Node's `os.tmpdir()` via `mkdtemp` with a `loopx-` prefix. Mode is `0700` (owner read/write/execute only). The exact name format beyond the prefix is implementation-defined and must not be relied on by scripts.

#### Scope and lifecycle

- **Created:** once per run, after the lazy pre-iteration sequence (`.loopx/` discovery, env-file loading, target resolution, version check) — which also surfaces any errors captured during the call-time option snapshot (invalid `RunOptions.env` shapes, snapshot-time exceptions from `Proxy` `ownKeys` traps or throwing enumerable getters on exotic `env` values, throwing `options.*` getters on the options object itself; see §2 "RunOptions.env" for the call-time capture model that produces these deferred errors) — and immediately before the first child spawns. Call-time option snapshot work runs synchronously at `run()` / `runPromise()` call time but never throws at the call site; any deferred errors it captures are surfaced through this same pre-iteration path. Pre-spawn failures — including surfaced call-time option snapshot errors — `-n 0` / `maxIterations: 0` early exits, and aborts observed before tmpdir creation do not create a tmpdir.
- **Shared across iterations.** All scripts in the run — the starting target, scripts reached via intra-workflow `goto`, scripts reached via cross-workflow `goto`, and re-executions of the starting target on loop reset — observe the same `LOOPX_TMPDIR` value.
- **Persisted within the run.** The directory is not cleared between iterations. Files written by one script remain visible to later scripts in the same run.
- **Concurrent runs are isolated.** Each `loopx run` invocation receives its own distinct directory. Parallel runs of the same workflow do not share temporary state.
- **Not created under `-n 0` / `maxIterations: 0`.** No child process is spawned, so no tmpdir is created and `LOOPX_TMPDIR` is not injected into any environment.

#### Cleanup

loopx runs cleanup of the tmpdir on every terminal outcome of a run that reached tmpdir creation:

- **Normal completion:** `stop: true` from a script, or `-n` / `maxIterations` reached.
- **Error exit:** non-zero script exit, invalid `goto` target, missing workflow or missing script during a `goto` resolution.
- **Child launch / spawn failure after tmpdir creation:** the discovered script has been removed or renamed between discovery and spawn (per SPEC §5.1), the runtime rejects a child environment entry at spawn (most reliably, an entry whose name or value contains an embedded NUL byte — from any env tier, i.e., inherited `process.env`, global or local env file, or `RunOptions.env`; `RunOptions.env` is the most likely practical source because env-file parsers typically reject such values earlier, but the runtime rejection itself does not distinguish tiers; see §2 for the full runtime-rejection model), `exec` itself fails, or any other pre-first-line-of-user-code spawn-path error. Cleanup runs before loopx exits with code `1`, the generator throws, or the promise rejects. This applies equally to the starting target and to scripts reached via `goto`.
- **SIGINT / SIGTERM to loopx:** if a child process group is active, cleanup runs after the process group exits (per SPEC §7.3, including `SIGKILL` escalation to the process group if required) and before loopx exits with the signal's exit code. If no child is active when the signal arrives (including the window after tmpdir creation but before the first child spawns, and the window between one child exiting and the next spawning), cleanup runs immediately.
- **Programmatic `AbortSignal` abort:** if a child process group is active, loopx first terminates it per SPEC §9.1; cleanup runs after the process group exits, before the generator throws or the promise rejects. If the abort fires while no child is active, cleanup runs eagerly from loopx's signal listener and the next generator interaction (or the outstanding `runPromise()` promise) settles with the abort error as soon as cleanup completes.
- **Consumer-driven cancellation under `run()`** (`break` from `for await`, explicit `generator.return()`, explicit `generator.throw(err)` after the first `next()`): loopx terminates any active child per SPEC §9.1, runs cleanup, then settles the generator (`{ done: true }` for `break` / `.return()`; throws `err` for `.throw(err)`).

Cleanup does **not** run when loopx itself is killed via SIGKILL or the host crashes; leaked tmpdirs are expected to be reaped by OS temp-cleaning policy (`systemd-tmpfiles`, tmpfs reboot). loopx does not attempt to reap stale tmpdirs at startup.

**Warning cardinality.** Cleanup warnings are categorized by the cleanup *routine* that produces them, not by whether cleanup arose from a terminal outcome or from creation failure. loopx uses two cleanup routines, and only one runs per cleanup attempt:

- The **full cleanup-safety routine** (below under "Cleanup safety") runs whenever device/inode identity was captured before cleanup begins — both on terminal-outcome cleanup of a successfully created tmpdir *and* on creation-failure cleanup when identity capture completed before the subsequent failure (see "Creation failure" below).
- The **non-recursive partial-directory cleanup path** runs only on creation-failure cleanup when identity capture itself failed. It attempts a single `rmdir` on the path and cannot recursively remove, unlink, or `lstat`-dispatch.

Cleanup emits stderr warnings from a closed set of three categories:

1. **Leave-in-place refusal** — emitted by the full cleanup-safety routine (cleanup safety cases 3 and 5 below) when the top-level `LOOPX_TMPDIR` path is a non-directory non-symlink replacement (case 3) or a mismatched-identity directory replacement (case 5). Fires at most once per cleanup because the routine dispatches on the single top-level `lstat` outcome; the recursive walk under case 4 unlinks encountered symlinks without warning and has no nested leave-in-place logic, so nested entries cannot produce additional category-1 warnings.
2. **Top-level cleanup failure** — emitted by the full cleanup-safety routine on any top-level dispatch failure, covering three mutually exclusive sub-cases: (a) the initial `lstat(LOOPX_TMPDIR)` fails for a reason other than ENOENT (ENOENT is cleanup-safety case 1 below, a no-op — not a warning), (b) the `unlink` of a top-level symlink replacement (case 2) fails, or (c) recursive removal of the identity-matched tmpdir directory (case 4) fails. Per-entry errors encountered during the recursive walk under sub-case (c) are subsumed by this single warning; loopx does not emit one warning per failed entry. Fires at most once per cleanup because the three sub-cases dispatch on mutually exclusive top-level outcomes.
3. **Partial-directory cleanup failure** — emitted by the non-recursive partial-directory cleanup path when its single `rmdir` fails for any reason. Fires at most once per cleanup.

Because only one routine runs per cleanup attempt, and each routine emits at most one warning via mutually exclusive dispatch paths, a single cleanup emits at most one warning in total. A creation-failure cleanup with identity captured therefore may emit a category-1 or category-2 warning (from the full cleanup-safety routine) — never category 3; a creation-failure cleanup without identity captured may emit a category-3 warning — never categories 1 or 2. The set is closed: loopx does not emit any other cleanup warnings, and tests may assert that no warning outside these three categories is produced.

**Category tokens in warning text.** Each emitted cleanup warning must contain exactly one of the following bracketed category tokens, corresponding to its category, appearing exactly once in a loopx-controlled prefix position of the warning text:

- Category 1 (leave-in-place refusal): `[leave-in-place-refusal]`
- Category 2 (top-level cleanup failure): `[top-level-cleanup-failure]`
- Category 3 (partial-directory cleanup failure): `[partial-directory-cleanup-failure]`

The token is the observable category marker and is the only portion of the warning text this ADR fixes; the remainder of the message (human-readable prose, the affected path, underlying errno, etc.) is implementation-defined. Tests match on token presence to assert category identity and may assert exactly one token per warning, but must not assert on any other portion of the message text. Cleanup warnings without any of these tokens, or with more than one distinct category token, are non-conforming.

**Token placement and user-controlled text.** Because paths under `os.tmpdir()` are ultimately under OS-tempdir naming conventions and may be influenced by `TMPDIR` / `TEMP` environment values that the caller (or the caller's shell / CI runner) controls, a raw path included verbatim in warning text could in principle contain a substring matching one of the three category tokens — for example, a path ancestor named `[top-level-cleanup-failure]`. The normative rule is that **the implementation must ensure user-controlled text cannot contribute an additional plain-text occurrence of any cleanup category token.** Two techniques combine to satisfy this rule, and neither is sufficient by itself in every case:

- **Reserved prefix placement.** The category token appears in a loopx-controlled prefix position of the warning — for example, at the start of the message or immediately after a fixed loopx-controlled preamble — before any path, errno, or other user-controllable text is rendered. Prefix placement identifies *which* token is the loopx-owned marker and is what lets tests rely on "the loopx-emitted token" being present at all. Prefix placement alone is **not** sufficient when user-controlled text is also rendered elsewhere in the message: a path or errno string appearing after the prefix could still contribute a second matching token substring under a plain-text scan.
- **Rewriting the user-controlled text so it cannot match a token.** Implementations that render paths, errno strings, or other user-controllable text in the warning body must escape, encode, redact, or omit any substring of that user-controlled text that would otherwise match one of the three cleanup tokens. Acceptable renderings include backslash-escaping the `[` and `]` characters (rendering `[top-level-cleanup-failure]` in a path as `\[top-level-cleanup-failure\]`, whose literal character sequence no longer contains the 27-character token as a plain-text substring), substituting a replacement character for `[` or `]`, redacting the matched substring, or omitting the rendered path entirely. **Merely surrounding user-controlled text in quotes is not sufficient**, because a quoted string like `"[top-level-cleanup-failure]"` still contains `[top-level-cleanup-failure]` as a plain-text substring — the surrounding quote characters do not change what substrings the text contains. The token-matching substring itself must be changed, not just flanked.

The specific escaping / redaction convention is implementation-defined; the normative requirement is that the rendered warning, taken as a whole, contains exactly one of the three bracketed category tokens when interpreted as a plain-text substring match, regardless of what text the implementation chose to include in the message. An implementation may always omit the user-controlled text entirely (in which case prefix placement is sufficient on its own for that warning) and trade diagnostic richness for trivially-satisfied cardinality. Tests must remain compatible with any conforming implementation: they may match on token presence at any position in the warning and assert exactly one token per warning, but must not assume a specific escaping convention, a specific prefix format, or that the token appears at a specific byte offset. A warning that contains exactly one category token (after whatever escaping the implementation applies) and is tied to a real cleanup failure is conforming. Whatever warnings are emitted, the CLI exit code, generator outcome, and promise rejection reason are unchanged by cleanup warnings.

**Token requirement is scoped to `LOOPX_TMPDIR` cleanup warnings only.** The category-token set (`[leave-in-place-refusal]`, `[top-level-cleanup-failure]`, `[partial-directory-cleanup-failure]`), the cardinality rule, and the non-conforming-without-token status defined in this section apply *only* to warnings emitted by `LOOPX_TMPDIR` cleanup — i.e., warnings produced by the full cleanup-safety routine or the non-recursive partial-directory cleanup path described above. Other loopx warnings — including but not limited to version-check warnings (e.g., local-workflow loopx version does not match the running CLI), unreadable-`package.json` warnings, warnings about discovered but invalid workflow or script entries (SPEC §5.1), warnings about ignored invalid lines in env files (SPEC §8.1), and any future non-cleanup warning category — are outside this taxonomy and must **not** include any of these three tokens unless a later ADR adds a cleanup-warning-equivalent contract for them. A non-cleanup warning that happens to contain one of these tokens would be a violation of this scoping rule, not a conforming cleanup warning. Tests that assert "no cleanup warning outside the three categories is produced" must therefore filter for cleanup warnings before asserting token presence — for example, by matching on the set of tokens this ADR defines rather than on all stderr output. Tests must not require arbitrary stderr warnings to contain cleanup tokens (doing so would fail on legitimate non-cleanup warnings, such as a version-check or env-file invalid-line warning, which by this scoping rule must not contain the tokens). Tests may assert that known non-cleanup warnings (identified by their own category or message) do not contain any of the cleanup tokens, since a non-cleanup warning that contains one of these tokens would violate the scoping rule above.

**Stability of the cleanup-warning contract.** The category-token set above, the "at most one warning per cleanup attempt" cardinality rule, and the "tests may assert no cleanup warning outside these three categories is produced" license are a stable public testing contract. Future ADRs may extend this taxonomy (for example, add a new category and its own `[<kebab-case-token>]` marker) but must not retroactively change the existing tokens, relax the cardinality rule, or remove the non-conforming status of untokened cleanup warnings without explicitly revising this contract. The scoping rule above — that the token requirement applies only to `LOOPX_TMPDIR` cleanup warnings — is itself part of this stable contract and is not relaxed by future additions to other warning categories. Test suites written against the three tokens defined here and the cardinality rule are expected to remain valid across implementation changes.

#### Cleanup safety

Cleanup is path-based and best-effort — not a sandbox against actively racing same-user processes. The normative guarantee is that loopx only recursively removes the directory at the `LOOPX_TMPDIR` path when that path still names the directory loopx created at tmpdir-creation time. loopx therefore captures an identity fingerprint of the created directory at creation time, before any child is exposed to the directory, and verifies at cleanup time that the path still names the same directory. The POSIX implementation strategy is to record the directory's device/inode pair at creation time and compare it against an `lstat` of the cleanup-time path; the normative behavior is the identity-match check itself, not the specific fingerprint used. At cleanup time loopx `lstat`s the `LOOPX_TMPDIR` path and dispatches. If the top-level `lstat` itself fails for any reason other than ENOENT (ENOENT is case 1 below, a no-op), loopx emits a single category-2 warning and aborts further cleanup — no traversal, `unlink`, or removal is attempted. Otherwise, the outcome of the top-level `lstat` selects one of the following cases:

1. **Path no longer exists (ENOENT):** no-op.
2. **Path is a symlink:** unlink the symlink entry; do not traverse or follow the target. Unlinking a symlink affects only the symlink entry itself. If the `unlink` itself fails, loopx emits a single category-2 warning and leaves the symlink entry in place.
3. **Path is a regular file, FIFO, socket, or other non-directory non-symlink:** leave in place with a single category-1 stderr warning. Unlinking such replacements is unsafe because a hard link would decrement a shared inode's `nlink`, and data renamed into the path has `nlink == 1` — in both cases `unlink` would mutate unrelated data.
4. **Path is a directory whose identity matches the recorded identity:** recursively remove. Symlink entries encountered during the walk are unlinked but not traversed, so symlinks pointing outside the tmpdir do not collateral-delete their targets. If recursive removal fails, loopx emits a single category-2 warning (per-entry errors during the walk are subsumed by this single warning).
5. **Path is a directory whose identity does not match the recorded identity:** leave in place with a single category-1 stderr warning. loopx does not recursively remove a directory it did not create.

A script that removes or renames its tmpdir during the run defeats automatic cleanup of the moved directory; loopx does not chase renamed tmpdirs.

The hard-link refusal in case 3 applies to replacement of the top-level `LOOPX_TMPDIR` path only; recursive cleanup under case 4 of the identity-matched tmpdir may unlink hard-link entries placed *inside* that directory, as ordinary recursive deletion would. This is not a hard-link preservation guarantee for entries inside the tmpdir: such entries are treated like any other file during the recursive walk, and the shared-inode safety property applies only to the top-level path.

This guarantee covers script-introduced replacements that are quiescent by the time cleanup begins. It is not a race-resistant guarantee against a same-user process that mutates the path during cleanup itself; a stronger guarantee (fd-relative `openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`) is out of scope for v1.

The safety guarantee covers symlink and path-replacement behavior only. loopx does not detect or avoid mount points or bind mounts inside `$LOOPX_TMPDIR`, so the recursive walk under case 4 may cross filesystem boundaries if a script mounted a filesystem inside the tmpdir. Scripts that mount into `$LOOPX_TMPDIR` take responsibility for unmounting before cleanup begins, or use a location outside `$LOOPX_TMPDIR` for mounted filesystems.

#### Creation failure

If the underlying `mkdtemp`, identity-capture, or mode-securing operation fails (e.g., `EACCES`, `ENOSPC`, `EMFILE`), loopx does not spawn any child. The CLI exits `1` with a stderr error; `run()` throws on the first iteration; `runPromise()` rejects.

If `mkdtemp` itself fails, no partial directory exists and no post-failure cleanup is required. If `mkdtemp` succeeds but a subsequent step fails, the cleanup behavior depends on whether identity was recorded before the failure:

- **Identity captured before the failure** (e.g., mode-securing or another post-identity step fails): loopx applies the full "Cleanup safety" routine above. Because directory identity is recorded, recursive removal of an identity-matched directory is safe. The routine may emit a category-1 or category-2 warning exactly as it does under terminal-outcome cleanup; it does not emit category 3. The warning category reflects which routine ran, not whether cleanup arose from a terminal outcome or creation failure (see "Warning cardinality" above).
- **Identity capture itself failed** (the operation to record directory identity raised — for example, a `stat`/`lstat` on the freshly created directory returned an error): loopx attempts a single non-recursive `rmdir` on the path via the non-recursive partial-directory cleanup path. Non-recursive `rmdir` succeeds only on an empty directory — the expected state immediately after `mkdtemp` — and fails without side effects if the path has been replaced with a symlink, a non-directory, or a non-empty directory. Without a recorded identity fingerprint, however, this `rmdir` cannot distinguish the expected "still the original empty `mkdtemp` directory" case from a rare replacement case where a same-user process has swapped the original directory for a *different* empty directory between `mkdtemp` and cleanup: in the replacement case, `rmdir` would remove that unrelated empty directory. loopx accepts this narrow behavior as a deliberate trade-off — it remains outside the race-resistant guarantee, matching the "not a sandbox against actively racing same-user processes" posture that applies to the with-identity regime under "Cleanup safety" above, and is the cost of cleaning up the common identity-capture-failure case where no replacement has occurred. The stricter alternative (skip cleanup entirely when identity capture fails) would leak the partial directory every time identity capture raised, which is the case this path exists to handle. If the `rmdir` fails for any reason, loopx leaves the path in place and emits a single category-3 warning. loopx does not `unlink`, `lstat`-dispatch, or recursively remove in this case, because without recorded identity it cannot verify that the path still names the directory it created.

Failure of the best-effort cleanup emits at most one warning (category 1, 2, or 3 depending on which routine ran) but does not mask the original creation-failure error.

If a SIGINT/SIGTERM or `AbortSignal` abort arrives concurrently with a creation failure, the signal/abort wins — the creation failure is not surfaced as the terminal outcome — and any partial directory is cleaned up under the same rules.

#### Programmatic settlement and pre-iteration precedence

The rules in this subsection govern how loopx orders programmatic pre-iteration failures — abort, captured options-snapshot errors, target validation, discovery, env-file loading, target resolution, and tmpdir-creation — against each other, and how generator settlement interacts with these outcomes. They refine SPEC §9.1's "abort always produces an error, even if it occurs before the first `next()` call" rule with the precedence details this ADR makes observable. The compact precedence summary below states the normative ordering rules; the subsections that follow specify the full behavior for each case.

**Precedence summary (programmatic API).** The following rules apply to `run()` and `runPromise()`:

1. **Pre-first-`next()` consumer cancellation on `run()` wins over all captured pre-iteration state.** A `.return(value)` or `.throw(err)` call as the consumer's *first* interaction with a generator returned by `run()` settles the generator per standard JS async-generator semantics without observing captured signal state and without surfacing any captured call-time option snapshot error (invalid `options` values, invalid `options.signal` values, non-string `options.cwd` / `options.envFile`, invalid `options.maxIterations`, invalid `options.env` shape, throwing option-field getters, throwing `options.env` enumerable getters, throwing `options.env` proxy traps). This carve-out applies only to `run()`; `runPromise()` has no equivalent consumer interaction point, so captured pre-iteration errors and already-aborted signals still reject its promise.
2. **Once a usable `AbortSignal` has been captured, abort wins over all other pre-iteration failures on the same call.** A signal that was already aborted at `run()` / `runPromise()` call time, or that aborts at any point during the pre-iteration sequence before the first child spawn, displaces all other pre-iteration failure modes (captured options-snapshot errors read after `options.signal`, `target` argument validation, target syntax validation per SPEC §4.1, `.loopx/` discovery errors, env-file loading errors, target resolution errors, and tmpdir-creation errors). Version checking is not a failure mode per SPEC §3.2 and therefore not in this list. "Usable `AbortSignal`" means a signal loopx could actually read from `options.signal` — an invalid `options` value or a non-`AbortSignal`-compatible `options.signal` captures no signal and does not enter this pathway.
3. **Invalid `options` and invalid `options.signal` are ordinary captured option errors, not aborts.** An `options` value that is `null`, a non-object, or an array, and an `options.signal` value that is not `AbortSignal`-compatible, each prevent loopx from capturing a usable signal. They are surfaced through the standard pre-iteration error path on first `next()` (or via promise rejection for `runPromise()`) and do not participate in abort precedence. Rule 1 still suppresses them when the first interaction on a `run()` generator is `.return()` / `.throw()`.
4. **`options.signal` is read before other `RunOptions` fields.** This fixed first-read placement guarantees that an already-aborted usable signal, or a signal that aborts synchronously during the `options.signal` getter read itself, is captured before any other option-field read can produce a snapshot exception. A non-compatible `options.signal` captured at first-read therefore wins over any later option-field error from the same call (because no signal was captured, rule 2 does not apply, but the options-snapshot-ordering rule still fixes the surfaced error to be the signal-validation error rather than a later field's error). Beyond `options.signal`'s first-read placement, the order in which the remaining option fields (`options.env`, `options.cwd`, `options.envFile`, `options.maxIterations`) are read is implementation-defined.
5. **Relative priority between non-abort captured option errors and non-options pre-iteration errors is implementation-defined.** For the programmatic API only, when a captured options-snapshot error (invalid `options.env` shape, throwing getter on any option field, non-string `options.cwd` / `options.envFile`, etc.) combines with a non-options pre-iteration failure (missing `.loopx/`, env-file loading error, target resolution error, tmpdir-creation error), either error is an acceptable terminal outcome. Tests must not assume one surfaces before the other.
6. **The CLI retains SPEC §7.1's concrete pre-iteration ordering.** Rule 5 is scoped to the programmatic API; CLI pre-iteration ordering is unchanged. CLI tests may assert §7.1's documented order.

Subsections below cover abort precedence (rule 2), CLI signal precedence, options snapshot ordering (rule 4), invalid `options` / `options.signal` (rule 3), invalid `options.cwd` / `options.envFile` / `options.maxIterations`, and the non-abort priority rule (rule 5) in full.

**Abort precedence across pre-iteration.** Abort precedence applies only once loopx has successfully captured a usable `AbortSignal` from `options.signal`. An invalid `options` value that prevents `options.signal` from being read at all (for example, `options` is `null`, a non-object, or an array), a non-`AbortSignal`-compatible `options.signal` value (for example, a string, a number, a plain object missing `addEventListener`), and an exception thrown while reading `options.signal` itself (for example, a throwing `signal` getter on the options object) are each ordinary options-snapshot errors surfaced through the standard pre-iteration path, not aborts, because no signal has been captured to observe. See "Invalid `options` and invalid `options.signal`" below for the full treatment of those cases. Once a usable signal has been captured, an `AbortSignal` that was already aborted at `run()` / `runPromise()` call time, or that aborts at any point during the pre-iteration sequence before the first child spawn, wins over every other pre-iteration failure mode: later `RunOptions` field validation (including `RunOptions.env` shape and snapshot-time exceptions, non-string `options.cwd` / `options.envFile` values per "Invalid `options.cwd`, `options.envFile`, and `options.maxIterations` values" below, and throwing `options.env` / `options.cwd` / `options.envFile` / `options.maxIterations` getters read after `options.signal`), target argument validation (non-string `target` values such as `undefined`, `null`, `42`, or any other runtime-invalid type per SPEC §9.1's lazy-rejection rule for runtime-invalid `target` values), target syntax validation (empty string `""`, bare colon `":"`, leading colon `":script"`, trailing colon `"workflow:"`, multiple colons `"a:b:c"`, or name-restriction violations per SPEC §4.1 "Target validation"), `.loopx/` discovery errors, env-file loading errors, target resolution errors, and tmpdir-creation errors. Version checking is part of the pre-iteration sequence but is not a failure mode at runtime per SPEC §3.2 (workflow-level version mismatches, unreadable workflow `package.json`, invalid JSON, and invalid semver ranges produce non-fatal warnings, not errors, and do not by themselves terminate the run); an abort observed during or after the version-check step therefore wins over whatever pre-iteration failure mode would otherwise surface from the steps listed above, but it does not displace a version-check "error" because SPEC §3.2 defines no such runtime error category. Whether a version-check warning is still emitted to stderr before the abort is surfaced is timing-dependent and is not itself contractually constrained by this ADR. The abort outcome becomes the terminal error regardless of which of these other pre-iteration steps would otherwise have failed. If the abort is observed before tmpdir creation begins, no tmpdir is created; if tmpdir creation was already in flight when the abort was observed, any partial directory is cleaned up per the rules above before the abort error is surfaced. This extends SPEC §9.1's existing "abort always produces an abort error" rule through the full pre-iteration sequence once a usable signal has been captured, including target-argument and target-syntax validation (both of which SPEC §9.1 currently routes through the same lazy first-`next()` rejection path as the other pre-iteration failures). An abort observed against a captured usable signal also wins over any other options-snapshot exception from the same call (including exceptions thrown while reading `options.env`, invalid `options.env` shapes, non-string `options.cwd` / `options.envFile` values, and throwing `options.env` / `options.cwd` / `options.envFile` / `options.maxIterations` getters read after `options.signal`).

**CLI signal precedence across pre-iteration.** SIGINT / SIGTERM delivered to a CLI `loopx run` process during the pre-iteration phase — after loopx has begun discovery / env-file loading / target resolution / version check / tmpdir creation but before the first child has spawned — follows the same "signal wins over other pre-iteration failures" rule as programmatic `AbortSignal`: if the signal arrives while pre-iteration is in flight, loopx treats it as the terminal outcome, cleans up any partial tmpdir per the rules above, and exits with `128 + signal number` regardless of which non-signal pre-iteration step (target syntax validation per SPEC §4.1, `.loopx/` discovery error, env-file loading error, target resolution error, or tmpdir-creation error) would otherwise have surfaced.

**Scope of CLI signal-wins precedence.** Signal-wins precedence begins only *after* the `run` command has passed the SPEC §4.2 `run -h` / `--help` short-circuit and usage-level argument parsing, and loopx has entered the pre-iteration phase. Usage-level argument parsing includes — non-exhaustively — the SPEC §4.1 / §4.2 rejection rules for: no positional target, more than one positional after `run`, duplicate `-n` or `-e`, invalid `-n` value, missing `-e` value, unrecognized flags, and the "no `--` tail" rule this ADR adds under "Affected SPEC Sections" for `§4.1 / §4.2 — CLI Grammar" (see below). These remain parser-level failures: if they apply, they are surfaced as usage errors (exit `1` with a stderr usage message, per existing §4.1 / §4.2 semantics) even if a signal arrives during argument parsing, because loopx has not yet installed its pre-iteration signal handlers at that point — and signals that arrive before handler installation follow the standard POSIX default disposition (see below). Target *syntax* validation for a single accepted candidate target (empty string, bare colon, leading colon, trailing colon, multiple colons, name-restriction violations per SPEC §4.1 "Target validation") is a pre-iteration failure surfaced by SPEC §7.1 after usage parsing has accepted the single positional, and is therefore displaceable by signal-wins once pre-iteration handlers are installed — which is why it appears in the displaceable list above. "Target argument validation" as a failure mode applies only to the programmatic API (non-string `target` per SPEC §9.1's lazy-rejection path) and is not a CLI failure mode, because the CLI always receives string arguments from the shell. A later ADR may revisit whether to extend signal-wins across usage parsing; this ADR does not.

Version checking runs as part of pre-iteration but is not a failure mode per SPEC §3.2 (workflow-level version mismatches and broken workflow `package.json` produce non-fatal warnings, not errors); the signal still wins, and whether a pending version-check warning is emitted to stderr before the signal's exit code is surfaced is timing-dependent and not contractually constrained by this ADR. If no tmpdir has been created yet when the signal arrives, no cleanup is required and loopx exits with the signal code directly. Signals that arrive before loopx has installed its pre-iteration signal handlers — for example, between process startup and argument parsing, during argument parsing itself, or during `run -h` / `--help` short-circuit handling — follow the standard POSIX default disposition (SIGINT / SIGTERM terminate the process without running loopx's cleanup logic), because there is no partial tmpdir to clean up and no loopx-side state to settle at that point. This rule parallels and extends SPEC §7.3's existing signal-forwarding rule, which covers signals received while a child process group is active, and closes the previously-unspecified gap between the end of usage parsing and the first child spawn.

**Options snapshot ordering.** Implementations must read `options.signal` before reading the other `RunOptions` fields (`options.env`, `options.cwd`, `options.envFile`, `options.maxIterations`). This ordering guarantees that a signal passed in an already-aborted state — or a signal that aborts synchronously during the `options.signal` getter read itself — is captured before any other option-field read can produce a snapshot exception, which is what makes "once a signal has been captured, an abort wins over any other options-snapshot exception from the same call" reachable in practice: without this ordering, a throwing `env` getter read before `options.signal` could short-circuit the pre-iteration sequence before any signal was ever captured, turning a run with an already-aborted signal into an `env`-getter rejection. Beyond this fixed first-read placement for `options.signal`, the order in which the remaining option fields are read is implementation-defined; tests must not assume (for example) that `options.env` is read before `options.cwd`. Each option field, including `options.signal`, is read at most once per call. Implementations must not retry or intentionally re-read an option field, a property getter (e.g., the `env` getter on the options object, an enumerable string-keyed getter on `options.env`), or a proxy trap (e.g., `ownKeys` or `getOwnPropertyDescriptor` on an `options.env` proxy) after it has already produced a value or error. Normal one-pass enumeration is permitted and may invoke per-property traps as required by the JavaScript object enumeration algorithm — for example, `ownKeys` called once on an `options.env` proxy followed by one `getOwnPropertyDescriptor` / get-trap invocation per enumerated own string-keyed property; the rule forbids loopx from performing an **additional** enumeration pass, not from the per-property trap invocations a single enumeration requires. A throwing getter must not be re-invoked to retry, because retrying could produce observably different results on a getter with side effects.

**Invalid `options` and invalid `options.signal`.** This ADR extends the call-time-capture / pre-iteration-surfacing model already applied to `options.env` field validation to cover the `options` argument as a whole and the `options.signal` field specifically, so the runtime behavior of these two inputs is no longer left implicit.

- **Invalid `options` value.** `options` must be omitted, `undefined`, or a non-null, non-array object. Any other value — `null`, a number (`42`), a string, a boolean, an array (`[]`, `[{ env: {} }]`), a function, or any other non-object — is captured at call time and surfaced through the standard pre-iteration error path: `run()` returns a generator without throwing at the call site, and the captured error is raised on the first `next()`; `runPromise()` returns a promise without throwing at the call site, and the promise rejects. The rationale is the same as for `options.env` shape validation — the call-time capture preserves SPEC §9.1's "`run()` never throws at the call site" guarantee while keeping the validation error locally-originated and observable at the usual pre-iteration point. Because the invalid-`options` case makes no option field readable (there is no object to read from), the "Options snapshot ordering" rule above is vacuously satisfied: there is no `options.signal` to capture, so an already-aborted signal cannot win over an invalid-`options` rejection in this case; the pre-iteration error path surfaces the invalid-`options` error on first `next()`. This subcase therefore sits in the same pre-first-`next()` consumer-cancellation carve-out as the other captured pre-iteration errors (per "Pre-first-`next()` consumer cancellation wins over all captured pre-iteration state" below in §1's Programmatic API subsection).
- **Invalid `options.signal` value.** When `options.signal` is present and not `undefined`, it must be an `AbortSignal`-compatible object — specifically, an object exposing the abort-listener interface loopx relies on: a readable `aborted` property and an `addEventListener('abort', listener)` method that returns without throwing when loopx calls it to register its abort listener (equivalently, any `AbortSignal` or polyfill that duck-types those two shapes and accepts loopx's listener registration). A non-compatible value — a string (`"not a signal"`), a number, a boolean, a plain object missing `addEventListener` (`{ aborted: true }`, `{ aborted: false }`), an object whose `addEventListener` is not callable, an object whose `addEventListener` throws when loopx invokes it to register the abort listener, an object whose `aborted` getter throws, or any other shape that cannot serve as an `AbortSignal` — is rejected through the standard pre-iteration error path rather than being silently treated as "no signal": `run()` raises the error on first `next()`, `runPromise()` rejects. For the throwing-`addEventListener` subcase specifically, **no usable signal has been captured**; the thrown error is captured as an options-snapshot error and surfaced through the standard pre-iteration path, exactly as other non-compatible-`signal` rejections are — not as an abort, because no signal has been captured to observe. Because the `options.signal` read happens first (per "Options snapshot ordering" above), the non-compatible-`signal` rejection is captured *before* any other option-field read can produce a snapshot exception, so in a call that combines a non-compatible `options.signal` with an invalid `options.env` shape or a throwing `options.cwd` getter, the captured non-compatible-`signal` error is the pre-iteration terminal error — not an abort (no signal has been captured to observe for abort), and not the later option-field error. This preserves the asymmetry in the existing "Abort precedence across pre-iteration" rule: abort precedence requires that loopx has successfully captured a usable `AbortSignal`; a non-compatible `options.signal` captures no signal and therefore does not enter the abort-precedence pathway.
- **`removeEventListener` is not part of the compatibility contract.** An `AbortSignal`-compatible value as defined above requires `aborted` and `addEventListener`; `removeEventListener` is **not** required. loopx does not refuse a supplied signal for lacking `removeEventListener`, and whether loopx invokes `removeEventListener` to clean up its abort listener after run settlement is implementation-defined: an implementation may rely on standard `AbortSignal` semantics (the listener becomes no-op after the signal has fired and is unreferenced once the `run` / `runPromise` call is fully settled), and a supplied signal that does not expose `removeEventListener` does not affect a run's observable outcome. Implementations that do invoke `removeEventListener` on supplied signals must tolerate its absence and must not treat absence as a signal-compatibility violation. Tests therefore must not assume loopx removes its listener on settlement, must not assume loopx leaves its listener attached on settlement, and must not assert that `removeEventListener` is or is not called on a supplied signal.
- **Interaction with the pre-first-`next()` carve-out.** Both invalid-`options` and invalid-`options.signal` errors are "captured call-time option snapshot errors" as that phrase is used in the pre-first-`next()` consumer cancellation rule (§1's Programmatic API subsection). A pre-first-`next()` `.return()` / `.throw()` therefore settles the generator without surfacing either error, exactly as it settles without surfacing a throwing `options.env` getter, an invalid `env` shape, or an already-aborted signal. Once the consumer's first interaction is `next()`, the captured error is surfaced on that first `next()`.

**Invalid `options.cwd`, `options.envFile`, and `options.maxIterations` values.** Extending the explicit validation model above to the remaining `RunOptions` fields beyond `options.env` (see §2 for the `options.env` shape, snapshot-timing, and runtime-rejection model): `options.cwd`, when present and not `undefined`, must be a string. `options.envFile`, when present and not `undefined`, must be a string. Any non-string value (number, boolean, array, object, `null`, function, or any other non-string non-`undefined` value) is captured at call time and surfaced through the standard pre-iteration error path: `run()` returns a generator without throwing at the call site and raises the error on first `next()`; `runPromise()` returns a promise without throwing at the call site and the promise rejects. `options.maxIterations` retains its existing SPEC §9.5 validation (non-negative integer; negative values, non-integers, or NaN cause `run()` to throw on first `next()` and `runPromise()` to reject). A throwing getter on any of these fields (`options.cwd`, `options.envFile`, `options.maxIterations`) is captured at call time and surfaced through the same pre-iteration error path, per the "Options snapshot ordering" rule above. Because `options.signal` is read first, a usable signal captured before any of these option-field reads wins over the resulting snapshot error per "Abort precedence across pre-iteration" above. The pre-first-`next()` consumer-cancellation carve-out (§1's Programmatic API subsection) also suppresses surfacing of captured non-string `options.cwd` / `options.envFile` values, captured invalid `options.maxIterations` values, and captured throwing-getter errors on these fields for a `.return()` / `.throw()` as the consumer's first interaction, exactly as it suppresses other captured call-time option-snapshot errors.

**Non-abort pre-iteration error priority (programmatic API only).** This implementation-defined priority rule applies to the programmatic API (`run()` / `runPromise()`) only. The CLI retains SPEC §7.1's concrete pre-iteration ordering — target argument required, discovery, env-file loading, target resolution, `-n 0` early exit, version check, first execution — and this ADR does not loosen that ordering; a CLI invocation that combines two pre-iteration failures surfaces them in §7.1's documented order, and CLI tests may assert that order. For the programmatic API, relative priority among non-abort pre-iteration errors — `RunOptions` snapshot exceptions (including invalid-`options` values such as `null`, numbers, strings, and arrays; invalid-`options.signal` values such as non-`AbortSignal`-compatible objects; non-string `options.cwd` / `options.envFile` values; invalid `options.maxIterations` values such as negative numbers, non-integers, or NaN; `options.env` shape rejections; `options.env` snapshot-time throws from a `Proxy` `ownKeys` trap or throwing enumerable getter; exceptions thrown while reading `options.signal` / `options.env` / `options.cwd` / `options.envFile` / `options.maxIterations` getters on the options object), target argument validation (non-string `target`) and target syntax validation (per SPEC §4.1), `.loopx/` discovery errors, env-file loading errors, target resolution errors, and tmpdir-creation errors — is implementation-defined except where already specified in this ADR (the "Options snapshot ordering" rule fixes `options.signal`'s read position but does not fix the surfacing order between the captured `options.signal` error and a later non-options pre-iteration failure). Version checking runs as part of pre-iteration but is not in this list because it produces non-fatal warnings per SPEC §3.2, not errors, so it is not a competing terminal-outcome mode. For example, when `options.env` shape is invalid *and* `.loopx/` is missing, either error is an acceptable terminal outcome for a programmatic call; tests must not assume one surfaces before the other. Abort precedence (above) applies on top of this rule: any captured abort during pre-iteration wins regardless of which non-abort error would otherwise have surfaced.

#### Programmatic API (`run()` / `runPromise()`)

Tmpdir creation is lazy under both APIs. Neither creates a tmpdir at the call site of `run()` or `runPromise()`. The pre-iteration sequence (field validation, discovery, env loading, target resolution, version check) runs on the first `next()` call for `run()` and asynchronously after return for `runPromise()`; tmpdir creation follows that sequence immediately before the first child spawns.

Aside from the call-time option snapshot (see §2 "RunOptions.env"), a generator returned by `run()` that is never iterated (no `next()`, no `.return()`, no `.throw()`) performs no discovery, no env-file loading, no target resolution, no version check, no tmpdir creation, and no child execution. Pre-first-`next()` `generator.return(value)` or `generator.throw(err)` settles the generator immediately per standard JS async-generator semantics without creating a tmpdir: because the generator body has not yet been entered, the pre-iteration sequence does not run.

**Pre-first-`next()` consumer cancellation wins over all captured pre-iteration state — signals and option-snapshot errors alike.** Consumer-driven `.return(value)` or `.throw(err)` on a pre-first-`next()` generator settles the generator (`{ value, done: true }` for `.return(value)`; throws `err` for `.throw(err)`) **without observing `AbortSignal` state and without surfacing any call-time option snapshot error captured by `run()`**, even when the signal captured at call time was already aborted at `run()` call time or aborted between `run()` returning and the consumer's `.return()` / `.throw()` call, and even when the call-time option snapshot captured a deferred error that would otherwise have been surfaced through the pre-iteration path. The full set of deferred errors that this carve-out suppresses for a pre-first-`next()` `.return()` / `.throw()` is: invalid `options` values (non-object, `null`, array — see "Invalid `options` and invalid `options.signal`" below), invalid `options.signal` values (present but not `AbortSignal`-compatible — also per the same subsection), non-string `options.cwd` or `options.envFile` values (present, not `undefined`, and not strings — see "Invalid `options.cwd`, `options.envFile`, and `options.maxIterations` values" below), invalid `options.maxIterations` values (negative, non-integer, NaN — per the same subsection), exceptions thrown while reading `options.signal` / `options.env` / `options.cwd` / `options.envFile` / `options.maxIterations` getters on the options object itself, invalid `options.env` shapes (null, array, non-object, or an entry with a non-string value), and snapshot-time exceptions on exotic `options.env` values (a `Proxy` whose `ownKeys` or `getOwnPropertyDescriptor` trap throws, a throwing enumerable string-keyed getter on the supplied object, etc.). This is a deliberate carve-out from "Abort precedence across pre-iteration" above and from the standard pre-iteration error-surfacing path, justified by standard async-generator semantics: `.return()` / `.throw()` on a generator whose body has not yet been entered does not execute the body and therefore cannot observe any state (signal, options, environment, captured option-snapshot errors) the body would read. The carve-out is tight — it applies only to the *first* consumer interaction, and only when that interaction is `.return()` or `.throw()`. Once the consumer's first interaction is `next()` — which does enter the body — abort precedence and captured option-snapshot errors apply as usual: an already-aborted or mid-pre-iteration signal produces the abort error (abort wins over any captured option-snapshot error per "Abort precedence across pre-iteration"), and in the absence of an abort a captured option-snapshot error is surfaced on that first `next()`. A second `.return()` / `.throw()` after a first `next()` has already entered the body is no longer a pre-first-`next()` call and is handled by the existing consumer-driven-cancellation rules in the Cleanup section above. SPEC §9.1's existing "aborting the signal always produces an error, even if it occurs between iterations or before the first `next()` call" language is therefore qualified by this carve-out: abort wins whenever the consumer's first interaction is `next()`, but a consumer whose first interaction is `.return()` / `.throw()` settles without observing the signal and without surfacing any captured option-snapshot error. This carve-out applies only to `run()`; `runPromise()` has no equivalent consumer-driven pre-first-interaction API (the promise begins the pre-iteration sequence asynchronously after return without waiting for any consumer input), so an already-aborted signal passed to `runPromise()` still rejects the promise with the abort error, and a captured option-snapshot error still rejects the promise through the same asynchronous path.

Once created, the tmpdir is cleaned up whenever the generator settles terminally or the promise settles. Callers who drive `run()` via `for await (...)` or who use `runPromise()` observe cleanup automatically. For `run()`, a final `Output` produced by `stop: true` or `maxIterations` is yielded before the generator settles: cleanup runs before the subsequent `{ done: true }` is observed, before `.return()` / `.throw()` settles, and before an abort error or spawn error is surfaced to the caller. A caller that consumes the final yielded output and then abandons the generator without driving it to `{ done: true }` (or invoking `.return()` / `.throw()`) is therefore in the abandoned-iteration regime described next. Callers who manually consume `next()` and then abandon the generator without calling `.return()`, `.throw()`, or driving it to completion may leak the tmpdir indefinitely: JavaScript does not guarantee finalization of an abandoned async generator during the process lifetime, and loopx makes no cleanup guarantee in that case. Callers that want guaranteed cleanup under manual iteration must reach a terminal state explicitly — drive the generator to `{ done: true }`, `break` out of a `for await` (which invokes `.return()` implicitly), or call `.return()` / `.throw()` on the abandoned generator.

**Abort after the final yielded `Output` still wins until generator settlement.** If an `AbortSignal` aborts after the final `Output` has been yielded (by `stop: true` or `maxIterations` reached) but before the generator has settled via `{ done: true }` — i.e., in the window between the consumer's receipt of the final yielded `Output` and the consumer's next interaction that would otherwise drive the generator to settlement — the abort becomes the terminal outcome. Normal completion is fixed only at settlement, not at the final yield: yielding the final `Output` does not by itself commit the run to normal completion, because the generator has not yet settled. loopx terminates any active child process group per SPEC §9.1 (there is typically no child active at this point, since the final `Output` is produced after the last iteration's child has exited), runs tmpdir cleanup per the rules above, and the next generator interaction — whether it is the `g.next()` call that would otherwise have resolved `{ done: true }`, a consumer `.return(value)`, or a consumer `.throw(err)` — throws the abort error rather than resolving `{ done: true }` or settling on the consumer-supplied value. This extends the general "abort wins until generator settlement" rule through the narrow window between the final yielded `Output` and subsequent settlement, and is consistent with the broader principle that cleanup and abort-error surfacing are tied to generator settlement rather than to the final yield. Consumers who need to commit normal completion before an abort can intervene must drive the generator to settlement (`await g.next()` after observing the final `Output`, `break` out of a `for await` loop that has run to completion, or allow `for await` to complete naturally) — and even then, abort precedence applies only to aborts observed before that settling interaction completes; an abort observed after settlement is a no-op for the now-settled generator.

### 2. `RunOptions.env`

`RunOptions` gains an optional `env` field:

```typescript
interface RunOptions {
  maxIterations?: number;
  envFile?: string;
  signal?: AbortSignal;
  cwd?: string;
  env?: Record<string, string>;
}
```

**Timing-model terminology.** This ADR distinguishes three phases for the lifecycle of an `env` value (and, by extension, for the other `RunOptions` fields):

1. **Call-time capture.** The synchronous work that runs at the `run()` / `runPromise()` call site: loopx reads the `RunOptions` fields (in the order required by "Options snapshot ordering" in §1), iterates `options.env`'s own enumerable string-keyed properties, and stores the resulting name/value pairs as a shallow copy. Any exception produced during this phase — a throwing getter on the options object, a throwing `ownKeys` / `getOwnPropertyDescriptor` / per-property-get trap on an `options.env` proxy, a throwing enumerable getter on a non-proxy `options.env`, an invalid-shape detection (null, array, non-object, non-string entry value), an invalid `options` value, an invalid `options.signal` value — is **captured** but does not escape the call site (preserving SPEC §9.1's "`run()` never throws at the call site" guarantee).
2. **Pre-iteration surfacing.** The path through which captured call-time errors are raised to the caller. For `run()`, this is the first `next()` call on the returned generator (which runs the pre-iteration sequence: discovery, env-file loading, target resolution, version check, tmpdir creation); the captured call-time error is raised during that sequence. For `runPromise()`, it is the asynchronous continuation after return that drives the same pre-iteration sequence; the captured error rejects the promise. The pre-first-`next()` consumer-cancellation carve-out (§1) suppresses surfacing of captured call-time errors for `run()` when the consumer's first interaction is `.return()` / `.throw()`.
3. **Spawn-time runtime rejection.** Errors detected by the underlying runtime (Node, Bun, Deno) when the child process is spawned — most reliably, a child-environment-entry name or value containing an embedded NUL byte that the runtime's `execve` binding refuses. The rejection applies to the entry regardless of which env tier supplied it (inherited `process.env`, the global loopx env file, the local `-e` env file, `RunOptions.env`, or loopx-injected protocol variables): the runtime acts on the assembled child environment and does not distinguish tiers. `RunOptions.env` is the most likely practical source because env-file parsers typically reject such values at parse time and inherited `process.env` and loopx-injected values typically do not contain NUL bytes, but the contract is tier-agnostic. These surface as child launch / spawn failures (§1 "Cleanup" and §7.2 affected SPEC) and run only on iteration counts that actually spawn a child (i.e., not under `maxIterations: 0` / CLI `-n 0`).

The three phases are disjoint: a shape-invalid `env` (phase 1 → phase 2) is surfaced even under `maxIterations: 0` because it requires no spawn; a NUL-containing entry (phase 3) requires at least one spawn to surface. The bullets below use these terms consistently; in particular, "field validation", "snapshot rejection", and "call-time shallow copy" all belong to phase 1 capture; "rejected via the pre-iteration error path" / "throws on the first `next()`" / "rejects" all belong to phase 2 surfacing; "runtime-rejected entries surface at spawn time" belongs to phase 3.

- **Shape.** `env` must be omitted, `undefined`, or a non-null, non-array object whose own enumerable string-keyed entries all have string values. Invalid shapes (null, array, non-object, or an entry with a non-string value) are rejected via the standard pre-iteration error path: `run()` throws on the first `next()`; `runPromise()` rejects.
- **Enumeration algorithm.** `RunOptions.env` is snapshotted using `Object.entries`-equivalent semantics over own enumerable string-keyed properties: symbol keys are skipped, inherited keys are skipped, and non-enumerable own string keys are not forwarded. The probe traps this enumeration algorithm may invoke are observable and permitted — on a proxy, a single `ownKeys` call followed by `getOwnPropertyDescriptor` invocations as the enumeration algorithm requires (on a proxy, this typically involves one `getOwnPropertyDescriptor` invocation per own string key to determine enumerability, including on keys that are ultimately filtered out as non-enumerable), and one property read per enumerated own enumerable string key. A non-enumerable own string key therefore may still produce one observable `getOwnPropertyDescriptor` trap invocation before being filtered out of the snapshot; "ignored" in this context means the resulting snapshot contains no forwarded entry for the property, not that the enumeration algorithm avoids touching the property at all. A probe trap or value-producing getter that throws — `ownKeys` throwing, a `getOwnPropertyDescriptor` trap throwing while determining enumerability (including on a non-enumerable own string key that is eventually filtered out), or an enumerable getter throwing while reading a value — is captured as a snapshot error and surfaced through the standard pre-iteration error path, rather than escaping at the call site or being silently swallowed. Symbol keys are filtered at the `ownKeys` level and their descriptors are not read, so a throwing `getOwnPropertyDescriptor` trap on a symbol key does not participate in this rule. This algorithmic definition also pins down the observable one-pass nature of enumeration: the rule "each field is read at most once per call" in "Options snapshot ordering" below applies per-property, not just per-field, and is compatible with the per-property traps this algorithm invokes. See "Options snapshot ordering" for the retry / additional-pass prohibition.
- **Merge position.** `env` entries are merged into the child environment *after* global and local env-file loading and *before* loopx-injected protocol variables (`LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, `LOOPX_WORKFLOW_DIR`, `LOOPX_TMPDIR`). A `RunOptions.env` entry therefore overrides same-named values from `-e`, the global loopx env file, and inherited `process.env`, and is itself overridden by protocol variables.
- **Lifetime.** Entries are captured at call time as a shallow copy: loopx reads the supplied object's own enumerable string-keyed properties once and stores the resulting name/value pairs. The inspection is synchronous at call time — loopx does not defer the copy into the lazy pre-iteration sequence — but it never throws at the call site. Mutating the original object after `run()` / `runPromise()` returns has no effect on the running loop. Any exception raised during this snapshot — for example, a `Proxy` `ownKeys` trap that throws, or a throwing enumerable getter on the supplied object — is captured at call time and surfaced through the standard pre-iteration error path rather than escaping synchronously. The same capture-and-surface behavior applies if accessing `options.env` itself throws (for example, an `options` object with a throwing `env` getter): the exception is captured at call time and routed through the pre-iteration error path rather than escaping at the `run()` / `runPromise()` call site. `run()` still returns a generator without throwing, and the error is raised on the first `next()`; `runPromise()` still returns a promise and rejects. This preserves SPEC §9.1's guarantee that `run()` never throws at the call site while keeping the supplied object's contents pinned to call-time state, so callers may mutate or discard it between the call and the first `next()` without affecting the run.
- **Applies to every script in the run.** The starting target, `goto` destinations (intra- and cross-workflow), and loop resets to the starting target all receive the same `env` additions.
- **No CLI surface.** `env` is programmatic-only. CLI callers pass per-run values via the shell env prefix (`key=value loopx run …`), which flows through inherited `process.env`, or via `-e` / `loopx env set` for file-based config.
- **`LOOPX_DELEGATED` remains an internal delegation guard, outside this ADR's script-protocol-variable tier.** SPEC §13 reserves `LOOPX_DELEGATED`, and SPEC §3.2 sets it only on the inner loopx spawn during project-root delegation — not on every script spawn. This ADR does not add `LOOPX_DELEGATED` to the per-spawn protected protocol-variable tier (§8.3 as rewritten in "Affected SPEC Sections" below): loopx does not inject `LOOPX_DELEGATED` on script spawns, and `RunOptions.env`, env-file entries, and inherited `process.env` may therefore supply it to spawned scripts without a loopx-side per-spawn override. The SPEC §3.2 behavior of `LOOPX_DELEGATED` at loopx startup (used by the inner loopx to detect and skip recursive delegation) is unchanged by this ADR. A script that invokes `loopx run <subtarget>` as a subprocess observes whatever `LOOPX_DELEGATED` value its own environment held at spawn time — which is the existing SPEC §3.2 behavior. A later ADR may revisit whether to protect `LOOPX_DELEGATED` from script-level env overrides if evidence of need accumulates.
- **No name validation beyond "must be a string-to-string entry."** loopx performs no env-name validation beyond requiring own enumerable string-keyed entries with string values. It does not enforce the POSIX `[A-Za-z_][A-Za-z0-9_]*` env-name pattern, does not reject `LOOPX_*` keys, does not reject `=` characters in names, does not reject empty-string names, and does not reject embedded NUL bytes in names or values. Runtime-level rejections surface as child launch / spawn failures; the only rejection loopx requires runtimes to produce is for embedded NUL bytes in a name or value, which `execve`-style C spawn APIs cannot represent unambiguously. Other malformed-but-runtime-accepted names (for example, a name containing `=`, or an empty-string name) are forwarded to the runtime unchanged, and the child-observed behavior is runtime-dependent: a runtime that concatenates each name and value with `=` before `execve` may deliver `{ "A=B": "C" }` to the child as if the environment string were `A=B=C`, so the child may observe key `A` with value `B=C` rather than a spawn failure; empty-string names may be silently dropped on some runtimes. Tests must not require loopx to reject such entries unless a later ADR explicitly adds loopx-side name validation. A `LOOPX_*` key that collides with a protocol variable is silently overridden by protocol injection (which runs after `env`). Because loopx does not validate names, and runtime-level rejections only surface at spawn time, `maxIterations: 0` / CLI `-n 0` — which validates and exits before any spawn — produces no error for either runtime-rejected or runtime-accepted-but-malformed entries. This is intentional and matches the no-tmpdir-created behavior under the same condition; callers who want eager runtime-level validation must run at least one iteration, and even then only runtime-rejected entries surface as errors. **Shape validation is independent of spawn.** Invalid `env` shapes — `null`, arrays, non-objects, own enumerable string-keyed entries with non-string values, or objects whose ownership-probing operations throw (a `Proxy` `ownKeys` trap that throws, a throwing enumerable getter on an own string-keyed property, etc.) — do not require a child process to detect and are rejected via the standard pre-iteration error path under `maxIterations: 0` just as they are under any other iteration count. Symbol-keyed, non-enumerable, and inherited properties are ignored even if their values are non-string, so such properties do not make the `env` invalid. A future implementation must not interpret "no child spawn" as "no `env` validation."
- **Inherited parent `process.env` is snapshotted once per run.** This rule applies to every run, regardless of whether `RunOptions.env` is supplied. loopx reads the caller's `process.env` during the pre-iteration sequence (alongside env-file loading) and reuses that snapshot for every child spawn in the run — the starting target, intra- and cross-workflow `goto` destinations, and loop resets all observe the same inherited-env tier. Mutations to the caller's `process.env` between spawns do not propagate to later children. This parallels the existing "env files are loaded once at loop start" rule (SPEC §8.1), giving every run a single stable non-protocol environment baseline; only the loopx-injected protocol variables that refresh per spawn by design (`LOOPX_WORKFLOW`, `LOOPX_WORKFLOW_DIR`) vary across child spawns in the same run. The snapshot is a shallow copy by key/value; loopx does not retain a reference to the parent's live `process.env` object.
- **Global env file path resolution follows the inherited-env snapshot's per-API timing.** The global loopx env file path (`$XDG_CONFIG_HOME/loopx/env`, with the documented `HOME`-based fallback when `XDG_CONFIG_HOME` is unset — see SPEC §8) is resolved from the inherited environment alongside env-file loading. The resolution timing matches the inherited-env snapshot's timing under each API rather than being uniformly "not at call time". For `run()`, the resolution is lazy: `XDG_CONFIG_HOME` / `HOME` are read at the first `next()` call alongside the inherited-env snapshot, so a mutation to `process.env.XDG_CONFIG_HOME` or `process.env.HOME` between `run()` returning and the first `next()` does influence global-env-file path resolution. For `runPromise()`, the `XDG_CONFIG_HOME` / `HOME` read follows the same implementation-defined timing as the inherited `process.env` snapshot itself (may occur during synchronous call setup, during an asynchronous continuation after return, or at any other point no later than the first child spawn — see "Inherited-env snapshot timing" below); callers who synchronously mutate these variables after `runPromise()` returns therefore cannot rely on those mutations being observed or not observed by global-env-file path resolution, matching the inherited-env rule. `RunOptions.env` does not affect global-env-file path resolution because `RunOptions.env` is merged into the child environment *after* env-file loading has already run (see "Merge position" below); `XDG_CONFIG_HOME` / `HOME` values supplied via `RunOptions.env` therefore reach spawned scripts but do not redirect loopx's own env-file path lookup. Callers who need deterministic global-env-file path resolution must set `XDG_CONFIG_HOME` / `HOME` in the inherited environment before the pre-iteration sequence begins — for `run()` that means before first `next()`, and for `runPromise()` that means before `runPromise()` returns (given that capture may happen as early as synchronous call setup).
- **Inherited-env snapshot timing — per-API split, not uniformly "lazy".** This rule has a deliberate asymmetry with `RunOptions.env` and `RunOptions.cwd` (both captured at call time — the former to protect against caller-side mutation of the supplied object after return, the latter to pin the project root against caller `process.chdir` between the call and the first iteration) and, separately, a deliberate split between `run()` and `runPromise()`. Inherited `process.env` is **not** uniformly captured "lazy, after return" under both APIs; instead:
  - **For `run()`,** the inherited-env snapshot is lazy. It is taken on the first `next()` call, alongside the rest of the pre-iteration sequence (discovery, env-file loading, target resolution, version check, tmpdir creation). Mutations to `process.env` between `run()` returning and the first `next()` **are** observed by the running loop; mutations after the first `next()` returns (including between iterations) are not. This is normative and testable.
  - **For `runPromise()`,** the inherited-env snapshot timing relative to the `runPromise()` return point is implementation-defined, bounded only by "no later than the first child spawn." Implementations may capture the snapshot during synchronous call setup (before `runPromise()` returns), during an asynchronous continuation after return (at any microtask, macrotask, or specific continuation boundary an implementation chooses), or at any other point no later than the first child spawn. The resulting snapshot is then reused for every child spawn in the run. Callers who synchronously mutate `process.env` after `runPromise()` returns but before awaiting the promise therefore cannot rely on those mutations being observed or not observed; either outcome is conforming. This deliberate looseness avoids pinning loopx's inherited-env capture to a specific JS runtime scheduling point (synchronous setup vs. microtask vs. setImmediate vs. a post-argument-validation boundary), which would be hard to test reliably across Node / Bun / Deno and would add surface area without a clear use case. The word "lazy" therefore applies only to `run()`; under `runPromise()` the capture is neither strictly call-time nor strictly post-return but is instead freely scheduled by the implementation within the bounds above.
  - **Unified guidance for deterministic per-run values.** Callers who need a specific key pinned to call-time semantics — under either API, and irrespective of scheduling — should pass it through `RunOptions.env`, which sits above inherited env in the precedence tier (§8.3 as rewritten below) and is captured synchronously at call time regardless of when the inherited-env snapshot is taken. Under `run()` this preserves SPEC §9.1's "`run()` never throws at the call site" guarantee without special-casing env capture; under `runPromise()` this is also the only portable way to achieve a deterministic per-run value, because relying on inherited `process.env` mutations relative to the `runPromise()` return point is non-portable under this ADR.

### 3. Project-root cwd

Every child process loopx spawns for script execution uses `LOOPX_PROJECT_ROOT` as its working directory. This replaces SPEC §6.1's current rule that scripts run with the workflow directory as cwd.

- **CLI.** The CLI's invocation cwd is the project root (SPEC §3.2: "For loopx, the project root is always the invocation cwd"). The spawned script cwd becomes the same path — unchanged from the CLI user's terminal perspective, but different from what the child script observes, which was previously `.loopx/<workflow>/`.
- **Programmatic.** `RunOptions.cwd` (or `process.cwd()` at call time when omitted) specifies both the project root and the script execution cwd. When `RunOptions.cwd` is supplied and relative, loopx resolves it to an absolute path once at call time against the caller's `process.cwd()` (equivalent to `path.resolve(process.cwd(), options.cwd)`); no further `realpath` / canonicalization is applied. When supplied and already absolute, it is used unchanged. This keeps `LOOPX_PROJECT_ROOT` (and therefore `LOOPX_WORKFLOW_DIR`, see §4) an absolute path for every caller while preserving the "no extra canonicalization by loopx" rule (see below). These were previously distinguished: `RunOptions.cwd` set the project root (used for `.loopx/` resolution and `LOOPX_PROJECT_ROOT`) while scripts ran with the workflow directory as cwd. With this change they coincide, and SPEC §9.5's "cwd does not control script execution cwd" disclaimer is dropped.
- **Applies to every script in the run.** The starting target, intra-workflow `goto` destinations, cross-workflow `goto` destinations, and loop resets to the starting target all receive the same cwd. No per-target or per-workflow cwd variation.
- **Scripts remain free to change directory.** A script that needs to run in its own workflow dir (for example, to invoke a workflow-local `npm run …` whose `package.json` sits next to the script) does so with `cd "$LOOPX_WORKFLOW_DIR"` in bash or `process.chdir(process.env.LOOPX_WORKFLOW_DIR)` in JS/TS. Such changes are scoped to that child process and do not propagate to later script spawns — loopx always spawns the next script with project-root cwd regardless.
- **Script module resolution is unaffected by cwd.** Node's ESM resolver, tsx, and Bun resolve bare specifiers and relative imports *for the script module* from the importing module's file path, not cwd. SPEC §3.3's current claim that closer-`node_modules` precedence is "a natural consequence of running scripts with the workflow directory as cwd" is incorrect and is retired by this ADR; the responsibility lies with the resolver's file-path walk-up, independent of cwd. Other cwd-dependent behavior that scripts or the tools they invoke may exhibit — TypeScript `tsconfig.json` discovery, package-manager lookups, git repo discovery, user code that calls `path.resolve` without an explicit base — is not governed by this bullet and continues to depend on the child's cwd (which, under §3, is `LOOPX_PROJECT_ROOT`). Returning to the script module case: a workflow with its own `node_modules/loopx` takes precedence over the CLI-provided resolution whenever the resolver walks up from a path inside `.loopx/<workflow>/` — which holds when the absolute discovery-time entry path contains no symlinks and the workflow directory is traversed literally. When that entry path traverses any symlink — the `.loopx/<workflow>` directory entry, the entry script file itself, or any intermediate component (including an ancestor of the project root reached through a symlinked `RunOptions.cwd` value) — Node's default main-module handling canonicalizes the path, and the resolver walks up from the canonical target rather than from the symlink-preserving `.loopx/<workflow>/` spelling. In that case the `node_modules/loopx` that lives under `.loopx/<workflow>/` at its logical (symlink-preserving) spelling is not guaranteed to be the one found, and the CLI-provided resolution may apply instead. Authors who need the workflow-local `node_modules/loopx` to win must ensure no symlinks lie on the discovered entry path.
- **Custom resolve hook preserves standard precedence.** SPEC §3.3's Node.js/tsx strategy — a custom module resolve hook installed via `--import` + `module.register()` that makes `"loopx"` resolvable to the CLI's package exports — must preserve standard module resolution precedence; it must not unconditionally short-circuit bare `"loopx"` specifiers to the CLI exports. The hook must first allow or perform normal file-relative resolution from the importing module (by delegating to `defaultResolve`, or by running an equivalent file-path walk-up) and fall back to the effective (post-delegation) CLI package exports only when no `node_modules/loopx` is resolvable via standard file-relative resolution from the importer's path — whether that standard resolution would have found a workflow-local `node_modules/loopx`, a `.loopx/node_modules/loopx`, a project-root `node_modules/loopx`, an ancestor `node_modules/loopx`, or any other location the resolver would normally reach. The rule is "any standard file-relative resolution of `"loopx"` wins over the CLI fallback"; the "closer `node_modules/loopx` wins" phrasing elsewhere in this ADR and in §3.3 refers to any position standard resolution would reach, not to workflow-local `node_modules/loopx` specifically. A hook that intercepts `"loopx"` before delegation would bypass workflow-local resolution regardless of cwd or the symlink state of the entry path, defeating the "closer `node_modules/loopx` wins" rule retained by this ADR. The preceding bullet still applies: even with a correctly delegating hook, the symlink-canonicalization behavior of Node's default main-module handling determines which directory the resolver walks up from, and a workflow-local `node_modules/loopx` at the symlink-preserving logical path is not guaranteed to win when any component of the discovered absolute entry path is a symlink.
- **`$LOOPX_PROJECT_ROOT` remains available.** The env var is still injected into every script's environment. Scripts that `cd` elsewhere and later need an absolute-path anchor back to the project root continue to use `$LOOPX_PROJECT_ROOT`. Scripts that never change directory may simply use relative paths like `SPEC.md` or `adr/0004-script-execution-context.md`.
- **No extra canonicalization.** loopx does not apply `realpath` / canonicalization to the project-root path beyond what the runtime already provides. For the CLI, `LOOPX_PROJECT_ROOT` is exactly the string returned by loopx's own `process.cwd()` at invocation. loopx does not consult `$PWD` or any other source to recover a shell-side logical (symlink-preserving) spelling; whatever `process.cwd()` returns is what loopx stores. On POSIX systems where `getcwd(3)` walks from the real inode and returns a canonical path, a CLI invocation from a symlinked directory therefore yields a canonical `LOOPX_PROJECT_ROOT` — loopx never sees the shell's logical symlinked spelling, and there is no "symlinked CLI invocation cwd" asymmetry between `LOOPX_PROJECT_ROOT` and the child's `process.cwd()` to preserve. CLI users who require a specific non-canonical spelling must invoke loopx from a runtime whose `process.cwd()` preserves that spelling; there is no other lever on the CLI side. For the programmatic API, `LOOPX_PROJECT_ROOT` is derived from `RunOptions.cwd` when supplied (with the call-time `path.resolve(process.cwd(), options.cwd)` step from the preceding bullet applied only if `options.cwd` is relative) or from `process.cwd()` at call time otherwise. That `path.resolve` step is the only transformation loopx applies; in particular, no `realpath` follows, and an absolute `RunOptions.cwd` is used unchanged. Programmatic callers are therefore the only path through which `LOOPX_PROJECT_ROOT` can preserve a symlinked spelling loopx itself never canonicalizes. A symlink-preserving `LOOPX_PROJECT_ROOT` can arise whenever the resolved `RunOptions.cwd` string contains symlinked path components, whether `RunOptions.cwd` was supplied as an absolute path or as a relative path whose resolved form contains symlinked components. Because `path.resolve` is a pure string-normalization operation that does not consult the filesystem, any symlinked spelling present in the supplied `options.cwd` — including relative forms such as `options.cwd === "link-to-proj"` when `/tmp/link-to-proj` is a symlink on disk — is preserved verbatim through resolution (`path.resolve("/tmp", "link-to-proj") === "/tmp/link-to-proj"`). What a relative `RunOptions.cwd` cannot do is recover a symlinked spelling of the caller's *own* current directory that the caller's `process.cwd()` has already canonicalized (POSIX `getcwd(3)` typically canonicalizes), because `path.resolve(process.cwd(), options.cwd)` inherits whatever spelling `process.cwd()` produces for its own prefix. Symlinked components supplied *in* the relative cwd are still preserved. `LOOPX_WORKFLOW_DIR` (§4) is composed from the same project-root value plus the discovery-time `.loopx/<workflow>` suffix and inherits the same "no extra canonicalization by loopx" rule; the `.loopx/<workflow>` suffix itself is preserved as discovered (a symlinked workflow entry keeps the symlinked name).
- **Directory identity vs. string spelling.** loopx controls the child's *effective working directory* by spawning with `LOOPX_PROJECT_ROOT` as its cwd, so filesystem operations inside the child (`stat(".")`, `open("./SPEC.md")`, etc.) refer to the same directory named by `LOOPX_PROJECT_ROOT`. loopx does **not** control the *string spelling* that runtime cwd APIs report for that directory, and the handling differs by runtime. Node/tsx/Bun's `process.cwd()` returns whatever `getcwd(3)` returns (typically a canonical, symlink-free path on POSIX); when `LOOPX_PROJECT_ROOT` is itself a canonical, symlink-free absolute path, `process.cwd()` matches it byte-for-byte in practice, and when `LOOPX_PROJECT_ROOT` was reached through a symlink (which requires a `RunOptions.cwd` whose resolved string contains symlinked components — whether `RunOptions.cwd` was supplied as an absolute path or as a relative path with symlinked components, since `path.resolve` preserves symlinked spellings present in the input string; per "No extra canonicalization" above, CLI invocations cannot produce this asymmetry because loopx derives `LOOPX_PROJECT_ROOT` from its own `process.cwd()` without consulting `$PWD`), `process.cwd()` typically reports the canonical realpath while `LOOPX_PROJECT_ROOT` preserves the symlinked spelling; both still name the same directory, and `stat` / device-inode checks confirm filesystem identity. Bash's `$PWD` and bare `pwd`, by contrast, are shell-maintained and inherit `PWD` from loopx's own environment (loopx does not manage `PWD` — see below). Per POSIX, Bash retains an inherited `PWD` that names the current working directory — including when the inherited spelling differs from the kernel cwd through symlinks — and regenerates `PWD` from `getcwd()` only when the inherited value does *not* name the current directory. Consequently, when loopx itself was launched by a shell (or any process) whose `PWD` was a symlinked logical spelling that canonicalizes to the same directory loopx then passes as child cwd, Bash preserves that inherited spelling and its `$PWD` / bare `pwd` may differ byte-for-byte from both `LOOPX_PROJECT_ROOT` and Node's `process.cwd()` even when `LOOPX_PROJECT_ROOT` is itself canonical. **Byte-for-byte equality between Bash `$PWD` / bare `pwd` and `LOOPX_PROJECT_ROOT` is therefore not a loopx guarantee in any regime**, including the canonical `LOOPX_PROJECT_ROOT` case. Scripts that need the authoritative loopx-chosen spelling must read `LOOPX_PROJECT_ROOT`; scripts that need the runtime's canonical cwd spelling use `process.cwd()` (JS/TS) or `pwd -P` / `/bin/pwd` (Bash — both bypass `$PWD` and walk the kernel cwd via `getcwd(3)`); scripts that just need to reach project files can use project-root-relative paths (`cat SPEC.md`, `readFileSync("adr/…")`) and rely on the effective-cwd guarantee; scripts that need filesystem identity rather than any string spelling use `stat(".")` or an equivalent device/inode comparison. Callers who want `LOOPX_PROJECT_ROOT` to match Node/tsx/Bun `process.cwd()` byte-for-byte should pass an absolute, already-canonical `RunOptions.cwd`; Bash `$PWD` / bare `pwd` remain outside the byte-for-byte guarantee regardless of `RunOptions.cwd` spelling, so tests that compare a Bash cwd string to `LOOPX_PROJECT_ROOT` must use `pwd -P` or filesystem identity rather than `$PWD` / bare `pwd`. Callers who need a canonical realpath of an arbitrary input should compute it themselves before passing it in.
- **`PWD` is not a protocol variable.** loopx neither sets nor unsets `PWD` in the child environment. Whatever `PWD` value loopx's own process holds is inherited unchanged by the child. Bash's `$PWD` at child startup therefore reflects its own POSIX-specified inheritance rules, not loopx's: per POSIX, Bash retains an inherited `PWD` that names the current working directory (even when the inherited spelling differs from the kernel cwd through symlinks) and regenerates `PWD` from `getcwd()` only when the inherited value does *not* name the current directory. So when loopx's own `PWD` is itself a symlinked logical spelling that canonicalizes to the same directory as the kernel cwd Bash is spawned in (`LOOPX_PROJECT_ROOT`), Bash preserves that inherited spelling rather than regenerating — even when `LOOPX_PROJECT_ROOT` is itself canonical — and `$PWD` / bare `pwd` may differ byte-for-byte from `LOOPX_PROJECT_ROOT`. JS/TS children observe `process.env.PWD` as-inherited with no runtime regeneration; scripts that need a reliable cwd string must use `process.cwd()` or `LOOPX_PROJECT_ROOT` rather than `process.env.PWD`, and Bash scripts that need the canonical kernel cwd spelling must use `pwd -P` (or `/bin/pwd`) rather than `$PWD` or bare `pwd`. `PWD` is deliberately kept outside the protocol-variable tier (§8.3) and is not reserved (§13). A caller who wants a specific `PWD` value in the child can pass it via `RunOptions.env` or via the inherited-environment tier; it will reach the child, subject to Bash's standard POSIX handling above for Bash children.
- **`$LOOPX_TMPDIR` interaction.** Scripts writing scratch files have two ergonomic options: `$LOOPX_TMPDIR/…` (preferred for intra-run state; auto-cleaned) or project-root-relative paths (for files that should persist after the run). Previously, project-root-relative writes required the `$LOOPX_PROJECT_ROOT` prefix because relative paths resolved against the workflow dir.
- **Breaking change vs. current SPEC §6.1.** A script that previously relied on cwd to reach workflow-local files (`./helper.sh`, `readFileSync("./config.json")`) now resolves those relative paths against the project root, which almost certainly fails. Migration is by inspection of existing workflow scripts: switch to `"$LOOPX_WORKFLOW_DIR/helper.sh"` (bash) or `resolve(process.env.LOOPX_WORKFLOW_DIR, "config.json")` (JS/TS). This is acceptable at the current pre-1.0 stage; no automated migration tooling is provided.

### 4. `LOOPX_WORKFLOW_DIR` injection

loopx injects `LOOPX_WORKFLOW_DIR` into every script's environment, set to the absolute path of the workflow directory containing the currently-executing script. This is the companion to §3's project-root cwd: scripts reach project files via project-root-relative paths or `$LOOPX_PROJECT_ROOT`, and reach workflow-local assets via `$LOOPX_WORKFLOW_DIR` — no `$(dirname "$0")` or `import.meta.url` gymnastics required.

- **Value.** The absolute path of the workflow's own directory. In v1's layout, this is `"$LOOPX_PROJECT_ROOT/.loopx/$LOOPX_WORKFLOW"`, but the injected form is authoritative; scripts should prefer it over re-deriving the path so they remain stable if loopx's layout evolves.
- **Per-spawn refresh — cross-workflow `goto` semantics.** `LOOPX_WORKFLOW_DIR` tracks the currently-spawned script's containing workflow, not the starting target's workflow. This matches `LOOPX_WORKFLOW` and is the same refresh rule that already applies to `LOOPX_WORKFLOW` per SPEC §8.
  - Starting target spawn: `LOOPX_WORKFLOW_DIR` points at the starting workflow's directory.
  - Intra-workflow `goto` spawn: unchanged (same workflow, same directory).
  - Cross-workflow `goto` spawn: `LOOPX_WORKFLOW_DIR` updates to the destination workflow's directory before the child spawns. The destination script's `$LOOPX_WORKFLOW_DIR` points at its own containing workflow, not the caller's.
  - Deeper chains (A → B → C via successive cross-workflow gotos): each spawn observes its own workflow's directory.
  - Loop reset to the starting target: `LOOPX_WORKFLOW_DIR` restores the starting workflow's directory.
- **Stable across helper files within a single script execution.** A top-level workflow script and a sibling helper it sources or imports both read the same `LOOPX_WORKFLOW_DIR` value, because the env var is injected once per child spawn and inherited by the whole process. This is more robust than `import.meta.url` (which changes per file) or `$0` (which refers only to the top-level script).
- **Scripts are invoked by absolute discovery-time path.** loopx spawns each script using the absolute path by which it was discovered under `.loopx/` (e.g., `bash /project/.loopx/review-adr/index.sh`), not a path relative to any cwd. For bash scripts this makes `$0` an absolute path and, because bash does not canonicalize `$0`, `$(dirname "$0")` equals `LOOPX_WORKFLOW_DIR` byte-for-byte — regardless of whether symlinks appear anywhere in that absolute entry path (the `.loopx/<workflow>` directory entry, the entry script file itself, or any intermediate component such as a symlinked project-root ancestor reached via `RunOptions.cwd`). This is because loopx composes both values from the same components and bash never canonicalizes `$0`. This bash equality is normative.
- **JS/TS `import.meta.url` equality is not guaranteed whenever the absolute entry path traverses any symlink.** For JS/TS scripts the invocation path is likewise absolute, but the main-module URL exposed to the script is produced by the runtime. Node's default behavior for the main module applies `realpath` to the main module's file path and exposes the canonical result via `import.meta.url`. loopx therefore does not guarantee `dirname(fileURLToPath(import.meta.url))` equals `LOOPX_WORKFLOW_DIR` whenever **any** component of the discovered absolute entry path is a symlink — the `.loopx/<workflow>` directory entry, the entry script file, or any intermediate component up to and including the project-root ancestor chain. Bun's main-module symlink behavior is not specified to match Node's. loopx does not apply runtime flags (e.g., Node's `--preserve-symlinks-main`) to force symlink preservation for the main module. What loopx **does** guarantee is what loopx controls: the absolute discovery-time invocation path, and the `LOOPX_WORKFLOW_DIR` value computed from the project root plus the `.loopx/<workflow>` entry name. Consequently, **`LOOPX_WORKFLOW_DIR` is authoritative** for JS/TS code that needs to reach workflow-local assets; the `import.meta.url`-derived dirname equals `LOOPX_WORKFLOW_DIR` only when the absolute entry path is symlink-free, and that equality is non-normative beyond that case.
- **Symlink behavior — matches discovery-time workflow path.** `LOOPX_WORKFLOW_DIR` is derived from the project root plus the `.loopx/<workflow>` entry name as discovered, not from the script file's filesystem identity and not via any `realpath` step applied by loopx. If discovery reached the workflow through a symlinked `.loopx/<workflow>` directory entry, `LOOPX_WORKFLOW_DIR` preserves the symlinked spelling (`<project>/.loopx/<name>`), not the symlink's realpath target. If the entry script file is a symlink while the workflow directory is not, `LOOPX_WORKFLOW_DIR` is still `<project>/.loopx/<name>` because it is not derived from the script file's path. If the `.loopx` directory entry itself is a symlink (e.g., `<project>/.loopx` → `/elsewhere/.loopx-real`), the `.loopx` segment is treated as an intermediate component of the absolute entry path and its symlinked spelling is preserved — loopx does not `realpath` the `.loopx` entry, and `LOOPX_WORKFLOW_DIR` retains the `<project>/.loopx/<name>` spelling as discovered. If the project root itself was reached via a `RunOptions.cwd` with symlinked components, loopx preserves that spelling too (per §3's "no extra canonicalization" rule). In all cases, bash `$(dirname "$0")` equals `LOOPX_WORKFLOW_DIR` byte-for-byte because both are composed from the same components and bash does not canonicalize `$0`. JS/TS `import.meta.url` may canonicalize the discovered entry path per the runtime's main-module rules in any of these symlink cases (see preceding bullet), so a JS/TS script that needs the discovery-time spelling must read `LOOPX_WORKFLOW_DIR` rather than derive from `import.meta.url`.
- **Reserved — protocol-variable tier.** `LOOPX_WORKFLOW_DIR` joins `LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, and `LOOPX_TMPDIR` in the top precedence tier of SPEC §8.3. A user-supplied value in inherited env, `-e` local env file, global env file, or `RunOptions.env` is silently overridden by the injected value.
- **Relationship to cross-workflow state.** Scripts that need to hand data across a cross-workflow `goto` continue to use `$LOOPX_TMPDIR` (preferred) or a shared fixed location like `$LOOPX_PROJECT_ROOT/.loopx/shared/…`. `LOOPX_WORKFLOW_DIR` is deliberately not a cross-workflow rendezvous path — a script in workflow B cannot use `$LOOPX_WORKFLOW_DIR` to read a file written by workflow A because it points at B's own directory after the cross-workflow spawn.

## Consequences

- **Workflow directories stop accumulating temp files.** Existing workflows migrate `.feedback.tmp` / `.claude-output.tmp` / `.answer.tmp` / `.prompt.tmp` / `ralph`'s `.iteration.tmp` into `$LOOPX_TMPDIR/…` and drop their manual `rm -f` branches. Workflow directories contain only version-controlled scripts and static assets.
- **Cleanup is automatic on failure.** Loops that error out, are signaled, or are aborted no longer leak scratch files. Callers who relied on lingering `.tmp` files for post-mortem inspection must write those outside `$LOOPX_TMPDIR` (for example, under `$LOOPX_PROJECT_ROOT`).
- **Cross-run state is the caller's responsibility.** `LOOPX_TMPDIR` never persists across runs. A workflow that genuinely needs cross-run state (a long-lived counter, a cache surviving across invocations) uses `$LOOPX_PROJECT_ROOT`-relative storage. `ralph`'s `.iteration.tmp` is **not** an example of cross-run state — it is intra-run state leaking into the workflow directory, and should migrate to `$LOOPX_TMPDIR/iteration`.
- **Concurrent runs are safe.** Parallel `loopx run` invocations of the same workflow do not clobber each other's scratch files, which was previously possible with in-workflow `.tmp` files.
- **Scripts drop `$LOOPX_PROJECT_ROOT`-prefix boilerplate.** Existing workflows can replace `ROOT="$LOOPX_PROJECT_ROOT"; cat "$ROOT/SPEC.md"` with `cat SPEC.md`, and invoke sibling helpers as `.loopx/shared/resolve-adr.sh` instead of `"$ROOT/.loopx/shared/resolve-adr.sh"`. `$LOOPX_PROJECT_ROOT` remains injected for scripts that `cd` elsewhere and need an absolute anchor back, or for scripts constructing paths to hand to tools that don't inherit cwd.
- **Tools invoked from scripts get project-root context by default.** `git`, `claude`, `codex`, and similar tools that use cwd to locate their workspace now operate against the whole project repo without scripts having to `cd` first. Scripts that want workflow-scoped tool behavior `cd "$LOOPX_WORKFLOW_DIR"` explicitly.
- **`RunOptions.cwd` semantics simplify.** Previously, `RunOptions.cwd` specified the project root but explicitly did not control script execution cwd. Now it specifies both. The `RunOptions` type is unchanged; only the documented meaning changes.
- **Workflow-local cwd-relative references break.** Any script currently using `./foo` or `readFileSync("./foo")` to reach a file next to itself now resolves those paths against the project root. Migration is `"$LOOPX_WORKFLOW_DIR/foo"` (bash) or `resolve(process.env.LOOPX_WORKFLOW_DIR, "foo")` (JS/TS). In this repo, no existing workflow script uses this pattern — they all already prefix with `$LOOPX_PROJECT_ROOT` — so migration in-tree is a simplification, not a rewrite.
- **Workflow-local assets reachable in one token.** Workflows with bundled prompt files, schemas, or fixtures next to their scripts (`.loopx/review-adr/prompt.md`, `.loopx/spec-test-adr/schema.json`, etc.) read them as `cat "$LOOPX_WORKFLOW_DIR/prompt.md"` or `readFileSync(resolve(process.env.LOOPX_WORKFLOW_DIR, "schema.json"))`. The `$(dirname "$0")` / `import.meta.url` patterns continue to work but are not required.
- **Programmatic callers get per-run env without mutating `process.env`.** Previously the only way to inject a per-run value programmatically was to mutate `process.env` before calling `run()`, which is racy under concurrent calls. `RunOptions.env` is a clean shallow copy scoped to the call.
- **CLI parameterization uses existing shell idioms.** `adr=0003 loopx run review-adr` works today and continues to work. Values supplied via the shell env prefix live at the inherited-env precedence tier — the lowest user-controlled tier in SPEC §8.3 — and are therefore overridden by both env files and `RunOptions.env`. CLI users who want file-backed precedence above inherited env write a `.env` file and pass `-e <path>` (local-env-file tier, which sits above the global-env-file tier that `loopx env set` writes). loopx does not introduce a dedicated CLI-level override tier above `-e` for one-shot values.
- **`RunOptions.env` fills a different gap than the CLI's `-e`.** Programmatic callers already have `RunOptions.envFile` as an `-e` equivalent, but that still requires an on-disk file. `RunOptions.env` exists so programmatic callers can pass per-run values (a) without mutating `process.env` (racy under concurrent calls), (b) without writing a temporary env file, and (c) layered dynamically on top of `RunOptions.envFile` — same-named entries in `RunOptions.env` override entries in the supplied env file. Placing `RunOptions.env` above env files rather than at inherited-env precedence is what makes (c) work; keeping the CLI shell-prefix at inherited-env precedence is what preserves SPEC §8.3's existing "env files win over inherited env" rule. The asymmetry in precedence between the two surfaces is therefore a consequence of their different use cases, not a gap for programmatic callers to reach parity with the CLI.
- **No named-argument schema.** Workflows do not declare expected input names, types, defaults, or requiredness. Scripts validate their own inputs. A schema mechanism may be introduced in a later ADR if the unchecked-input model proves insufficient.
- **No first-class iteration counter.** Workflows maintain one in `$LOOPX_TMPDIR/iteration` if needed. A first-class counter may be introduced later.
- **Migration is manual but straightforward.** Existing `review-adr`, `apply-adr`, and `spec-test-adr` workflows (1) replace in-workflow `.tmp` paths with `$LOOPX_TMPDIR/…` and drop their `rm -f` cleanup, (2) take their per-run parameter via shell env prefix (e.g., `adr=0003 loopx run review-adr`), and (3) optionally drop the `ROOT="$LOOPX_PROJECT_ROOT"` header and `$ROOT/` prefixes, writing project-root-relative paths directly (e.g., `cat SPEC.md`, `cat adr/0004-script-execution-context.md`). No automated migration tooling is provided.

## Rejected / Deferred Alternatives

These alternatives were considered and not adopted by this ADR. Each is revisitable in a later ADR if evidence of need accumulates.

- **First-class CLI named-argument syntax (e.g., `loopx run <target> -- name=value`).** Deferred. The identified use case — passing a parameter like an ADR number into a script — is already served by the shell env-var prefix (`adr=0003 loopx run review-adr`), which flows through loopx's inherited-env tier. A dedicated CLI parser would add two-phase parsing, a name grammar, duplicate detection, `LOOPX_*` reservation, and a new precedence tier above `-e`, for no incremental value over the shell prefix at v1.
- **POSIX `--` end-of-options marker on `run` outside help.** Rejected. `--` is not supported in any position in the `run` argument list outside the `run -h` / `--help` short-circuit; wherever it appears, it is an unrecognized flag / usage error under the existing SPEC §4.1 / §4.2 rejection rules. Adding `--` support as an end-of-options marker before the target (e.g., accepting `loopx run -- ralph` as equivalent to `loopx run ralph`) would introduce parser surface — position-dependent tokenization, explicit distinction between the "named-argument tail" and "end-of-options marker" usages of `--` — without matching value, since `run` accepts at most a single positional target whose name is constrained by SPEC §4.1's `[a-zA-Z0-9_][a-zA-Z0-9_-]*` pattern and therefore cannot begin with `-`. A later ADR that adopts a first-class CLI named-argument surface may revisit both.
- **Named-argument schema.** Deferred. Workflows do not declare expected input names, types, defaults, or requiredness; each script validates its own inputs (`: "${adr:?need adr}"` in Bash; explicit checks in TS). A schema mechanism may be introduced later if the unchecked-input model proves insufficient.
- **First-class iteration counter.** Deferred. Scripts that need an iteration counter maintain one in `$LOOPX_TMPDIR/iteration` — a trivial read/increment/write against a file loopx guarantees to clean up on every terminal outcome.
- **Stronger race-resistant tmpdir cleanup (fd-relative `openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`).** Deferred. v1 uses path-based best-effort cleanup with device/inode identity matching, which handles script-introduced replacements that are quiescent by cleanup time. A race-resistant guarantee against a same-user process actively mutating the path during cleanup is out of scope.
- **Treating `PWD` as a protocol variable.** Rejected. Setting `PWD=LOOPX_PROJECT_ROOT` in every child would make `PWD` a de facto protocol variable, adding surface area to §8.3 and §13 for limited benefit. Scripts that need a reliable cwd string already have `process.cwd()` (JS/TS) and `pwd -P` (Bash); scripts that need the loopx-chosen spelling use `LOOPX_PROJECT_ROOT`.
- **`LOOPX_PROJECT_ROOT` canonicalization / `realpath`.** Rejected. Canonicalizing `LOOPX_PROJECT_ROOT` would break the "loopx preserves the runtime-provided spelling" rule used by the CLI (where `LOOPX_PROJECT_ROOT` is whatever `process.cwd()` returns at invocation, with no further transformation) and the "loopx preserves caller-provided spelling" rule used by `RunOptions.cwd` and the `LOOPX_WORKFLOW_DIR` composition. Callers that need a realpath compute one themselves before passing `RunOptions.cwd`. CLI users who want a specific non-canonical spelling must invoke loopx from a runtime whose `process.cwd()` already preserves it — loopx does not read `$PWD` to recover a shell-side logical spelling (see §3 "No extra canonicalization").

## Affected SPEC Sections

When this ADR is accepted, the following SPEC sections require updates:

- **§3.2 — Project Root.** Clarify the project-root derivation rules this ADR relies on. For the CLI, the project root is the absolute path returned by loopx's own `process.cwd()` at invocation time; loopx does not consult `$PWD` and does not apply an additional `realpath` / canonicalization step. On POSIX systems where `getcwd(3)` canonicalizes by walking from the real inode, a CLI invocation from a symlinked directory therefore yields a canonical `LOOPX_PROJECT_ROOT`, and loopx provides no lever to recover the shell-side logical (symlink-preserving) spelling from the CLI side — CLI users who need a specific non-canonical spelling must invoke loopx from a runtime whose `process.cwd()` already preserves it. For the programmatic API, the project root is `RunOptions.cwd` when supplied (resolved to an absolute path once at call time via `path.resolve(process.cwd(), options.cwd)` if relative, used unchanged if already absolute; no further `realpath`), or `process.cwd()` at call time when omitted. Cross-reference §6.1 for the script execution cwd change (scripts now run with `LOOPX_PROJECT_ROOT` as cwd, not the workflow directory) and §9.5 for the full `RunOptions.cwd` type and timing rules. The existing SPEC §3.2 startup-reserved `LOOPX_DELEGATED` handling (set on the inner loopx process during project-root delegation, used as a recursion guard) is unchanged by this ADR; see "§13 — Reserved Values" below for the distinction between startup-reserved and script-protocol-protected reserved names.
- **§7.1 — Pre-iteration sequence.** Insert `LOOPX_TMPDIR` creation between the starting workflow version check and the first child spawn. Under `-n 0` / `maxIterations: 0`, no tmpdir is created.
- **§3.3 — Module Resolution for Scripts.** Rewrite the sentence "This is a natural consequence of running scripts with the workflow directory as cwd (section 6.1)" to describe the actual mechanism: standard file-relative resolution of `"loopx"` — Node ESM / tsx / Bun walking up from the importing module's file path through enclosing `node_modules/loopx` directories — wins over the CLI fallback, independent of cwd. The winning `node_modules/loopx` may be workflow-local (`.loopx/<workflow>/node_modules/loopx`), `.loopx`-level (`.loopx/node_modules/loopx`), project-root (`<project>/node_modules/loopx`), or any ancestor the resolver walks into; the rule is simply "any standard file-relative resolution of `"loopx"` wins over the CLI fallback", not a workflow-local-only rule. Additionally, specify that the custom Node.js/tsx resolve hook described earlier in §3.3 must preserve this standard precedence: the hook must first allow or perform normal file-relative resolution from the importing module (via `defaultResolve` delegation or an equivalent file-path walk-up) and fall back to the effective CLI package exports only when no `node_modules/loopx` is resolvable via standard file-relative resolution from the importer's path. A hook that unconditionally short-circuits bare `"loopx"` to the CLI exports would violate this rule regardless of cwd or symlink state, and is non-conforming under this ADR. Note that SPEC §5.1 follows symlinks during discovery on the workflow directory, the entry script file, and any path component inherited from the project root, and that Node's default main-module handling canonicalizes the main module's file path via `realpath`. When the absolute discovery-time entry path traverses any symlink — the `.loopx/<workflow>` directory entry, the entry script file, or any intermediate component (including a project-root ancestor reached through a symlinked `RunOptions.cwd`) — the resolver walks up from the canonical target rather than the symlink-preserving `.loopx/<workflow>/` spelling, and `node_modules/loopx` locations that lie on the symlink-preserving logical path (commonly the workflow-local or `.loopx`-level positions) are not guaranteed to win even when the resolve hook delegates correctly. Authors who require a specific symlink-preserving-path `node_modules/loopx` to be selected must ensure no symlinks lie on the discovered entry path.
- **§4.1 / §4.2 — CLI Grammar.** Record that, **outside the existing `run -h` / `--help` short-circuit (§4.2)**, `loopx run` does not accept `--` in any position, nor any named-argument, key-equals-value, or per-run-parameter CLI syntax: `--` is treated as an unrecognized flag / usage error wherever it appears in the `run` argument list when `-h` / `--help` is not present (before the target, after the target, or interleaved with other flags). This includes the common POSIX end-of-options usages — `loopx run -- <target>` is rejected, `loopx run -n 1 -- <target>` is rejected, and `loopx run <target> -- name=value` is rejected — in addition to the named-argument tail usage. The sole supported surface for per-run values on the CLI is the standard shell env-var prefix (`key=value loopx run <target>`), which flows through inherited `process.env` at the inherited-environment precedence tier (§8.3). Callers who want file-backed precedence above inherited env use `-e <path>` or `loopx env set`. Record also that a `name=value` token appearing as the *sole* positional after `run` — `loopx run name=value` — is parsed as a target string under SPEC §4.1 and fails target validation because `=` is not part of the `[a-zA-Z0-9_][a-zA-Z0-9_-]*` workflow / script name pattern; it is not interpreted as a named argument. The rejected forms are therefore: `loopx run -- <target>` → rejected as unrecognized `--` (no end-of-options support); `loopx run -n 1 -- <target>` → rejected as unrecognized `--`; `loopx run <target> -- name=value` → rejected as unrecognized `--` / no supported `--` tail; `loopx run <target> name=value` → rejected as multiple positionals (existing §4.1 rule); `loopx run name=value` → rejected as target-validation failure (existing §4.1 rule). Only the shell-absorbed `name=value loopx run <target>` form succeeds, because the shell consumes the prefix before loopx sees its arguments. loopx's `run` grammar does not support `--` as an end-of-options marker and introduces no named-argument tail; any additional positional or flag after the target, and any `--` in any position, is rejected as a usage error under the existing §4.1 / §4.2 "more than one positional argument", "Target validation", and "unrecognized flags" rules. **The `run -h` / `--help` short-circuit is preserved exactly as written in SPEC §4.2** — when `-h` / `--help` is present anywhere in the `run` argument list (before the target, after the target, or interleaved with other flags), loopx shows run help, exits 0, and ignores all other run-level arguments unconditionally. Concretely, `loopx run ralph -- name=value -h`, `loopx run ralph -h extra`, `loopx run -h ralph -- foo=bar`, and `loopx run -h -- ralph` all show run help and exit 0; the `name=value` tokens, extra positionals, and `--` marker are not parsed or rejected under the help short-circuit, matching §4.2's existing list of ignored arguments. Only in the absence of `-h` / `--help` do this ADR's "no named-argument tail" and "no `--`" rules take effect, at which point any additional positional or flag after the target, any `--` in any position, and any unrecognized flag are usage errors per the existing §4.1 / §4.2 rejection rules. This decision is reversible only via a later ADR that adopts a first-class CLI named-argument surface or end-of-options support (see "Rejected / Deferred Alternatives").
- **§5.1 — Discovery.** Retarget the mid-loop removal/rename clause from "treated as a non-zero exit (section 7.2)" to "treated as a child launch / spawn failure (section 7.2)" so the reference matches §7.2's updated taxonomy (which now distinguishes script non-zero exit from spawn-path failure after tmpdir creation). The exit code remains `1` in either framing. Clarify the existing "symlinks are followed during discovery" rule so it is not misread as "resolved to `realpath`": discovery follows symlinks only to determine whether an entry is a directory or a supported script file, but the cached workflow and script paths preserve the discovered path spelling — loopx does not replace discovered paths with their `realpath`. This is the basis for the symlink-preserving `LOOPX_WORKFLOW_DIR`, the symlink-preserving absolute entry path used to spawn children (§6.1), and the symlink-preserving Bash `$0` / `$(dirname "$0")` equality (§6.2). Extend the symlink policy to cover the `.loopx` directory itself: if `.loopx` is a symlink (e.g., `<project>/.loopx` → `/elsewhere/.loopx-real`), discovery follows it to determine that it names a directory, but the cached workflow paths retain the `<project>/.loopx/<workflow>` spelling — the `.loopx` segment is treated as an intermediate component of the absolute entry path and its symlinked spelling is preserved in `LOOPX_WORKFLOW_DIR` and in the invocation path passed to the runtime. This parallels the existing treatment of symlinked `.loopx/<workflow>` directory entries, symlinked entry script files, and symlinked project-root ancestors.
- **§6.1 — Working Directory.** Replace "All scripts run with the workflow directory as their working directory" with "All scripts run with `LOOPX_PROJECT_ROOT` as their working directory". Drop the "ensures relative imports and `node_modules/` resolve naturally" rationale — it is inaccurate, since module resolution is file-relative rather than cwd-relative. Note that scripts remain free to `cd` / `process.chdir()` themselves, that such changes are scoped to the individual child process, and that later script spawns always start at project-root cwd. Record the normative rule — carried by §6.1 alone — that loopx invokes each script using its absolute discovery-time path and does not canonicalize that path: any symlinks in it (the `.loopx/<workflow>` directory entry, the entry script file, or intermediate components inherited from the project root) are preserved in the absolute path passed to the runtime. Cross-reference §6.2 and §6.3 for the language-specific consequences of that invocation rule.
- **§6.2 — Bash Scripts.** Note that the absolute discovery-time invocation path makes `$0` an absolute path, so `$(dirname "$0")` equals `LOOPX_WORKFLOW_DIR` byte-for-byte regardless of whether symlinks appear anywhere in that absolute path — the `.loopx/<workflow>` directory entry, the entry script file itself, or any intermediate component such as a symlinked project-root ancestor. Bash does not canonicalize `$0`, and loopx composes both values from the same components.
- **§6.3 — JS/TS Scripts.** Note that `dirname(fileURLToPath(import.meta.url))` equals `LOOPX_WORKFLOW_DIR` only when the absolute discovery-time entry path is symlink-free in every component. When any component is a symlink — the `.loopx/<workflow>` directory entry, the entry script file, or any intermediate path component — the runtime (Node by default; Bun behavior unspecified) may canonicalize the main module's URL and the equality does not hold. Scripts that need a portable, symlink-preserving workflow-directory handle must use `LOOPX_WORKFLOW_DIR`. loopx does not pass runtime-specific symlink-preservation flags (e.g., `--preserve-symlinks-main`).
- **§7.2 — Error Handling.** On `LOOPX_TMPDIR` creation failure (`mkdtemp`, identity-capture, or mode-securing fails), non-zero script exit, invalid / missing `goto` target, or child launch / spawn failure after tmpdir creation (discovered script removed or renamed before spawn per §5.1, runtime rejection of a child environment entry from any env tier — inherited `process.env`, global or local env file, or `RunOptions.env` — most reliably an entry whose name or value contains an embedded NUL byte, `exec` failure), the CLI exits `1` / `run()` throws on first iteration / `runPromise()` rejects. On tmpdir creation failure, no child is spawned; if a partial directory exists, best-effort cleanup runs per the rules in §1 ("Creation failure") before the original creation error is surfaced. On the other error paths, `LOOPX_TMPDIR` cleanup runs after the error is detected and before loopx exits `1`. Document the cleanup-warning category-token contract (§1 "Warning cardinality" and §1 "Token placement and user-controlled text"): each warning emitted by `LOOPX_TMPDIR` cleanup must contain exactly one of the three bracketed category tokens (`[leave-in-place-refusal]`, `[top-level-cleanup-failure]`, `[partial-directory-cleanup-failure]`) appearing exactly once in a loopx-controlled prefix position, corresponding to its category; the remainder of the warning message is implementation-defined. Because paths under `os.tmpdir()` are user-controllable via `TMPDIR` / `TEMP`, implementations that include user-controlled text such as paths in the warning body must render it so the user-controlled text cannot contribute an additional plain-text occurrence of any category token. Acceptable techniques include escaping `[` / `]` characters in user-controlled text (e.g., backslash-escaping: `[top-level-cleanup-failure]` → `\[top-level-cleanup-failure\]`), substituting a replacement character, redacting the matched substring, or omitting the rendered path entirely; reserved prefix placement alone is not sufficient if user-controlled text is also rendered elsewhere in the message, and merely surrounding user-controlled text in quotes does not satisfy the rule because the quoted string still contains the token as a plain-text substring. The specific convention is implementation-defined; the normative requirement is that the rendered warning, taken as a plain-text substring match, contains exactly one category token. Cleanup warnings without any token, or with more than one distinct category token, are non-conforming. This token requirement applies only to `LOOPX_TMPDIR` cleanup warnings — other loopx warnings (version-check, unreadable-`package.json`, invalid-entry, invalid env-file line, etc.) are outside this taxonomy and must not include these tokens.
- **§7.3 — Signal Handling.** On SIGINT / SIGTERM, `LOOPX_TMPDIR` cleanup runs after any active child process group has exited (per the existing grace period) and before loopx exits with the signal's code. When no child is active when the signal arrives, cleanup runs immediately. Signals that arrive before tmpdir creation require no cleanup. Document that SIGINT / SIGTERM delivered during the CLI pre-iteration phase (after loopx has begun discovery / env-file loading / target resolution / version check / tmpdir creation, before the first child spawns) follows the same "signal wins over pre-iteration failures" rule as programmatic `AbortSignal` precedence (§1 "CLI signal precedence across pre-iteration"): if a signal arrives while pre-iteration is in flight, loopx treats the signal as the terminal outcome, cleans up any partial tmpdir (per §1 "Creation failure" rules), and exits with `128 + signal number` regardless of which non-signal pre-iteration step (target syntax validation per SPEC §4.1 for an accepted single candidate target, `.loopx/` discovery error, env-file loading error, target resolution error, or tmpdir-creation error) would otherwise have failed. Signal-wins precedence begins only after the `run` command has passed the SPEC §4.2 `run -h` / `--help` short-circuit and usage-level argument parsing (no target, multiple positionals, duplicate `-n` / `-e`, invalid `-n`, missing `-e` value, unrecognized flags, and the "no `--` tail" rule from §4.1 / §4.2 below); those usage errors remain parser-level failures even if a signal arrives during argument parsing, because loopx has not yet installed its pre-iteration signal handlers at that point (§1 "Scope of CLI signal-wins precedence"). Version checking runs as part of pre-iteration but is not in the displaceable list because SPEC §3.2 defines no runtime version-check error category (workflow-level version mismatches and broken workflow `package.json` produce non-fatal warnings, not errors); the signal still wins, and whether a pending version-check warning reaches stderr before the signal's exit is timing-dependent and not contractually constrained by this ADR. Signals that arrive before loopx has installed its pre-iteration signal handlers (between process startup and argument parsing, during argument parsing, or during `run -h` / `--help` short-circuit handling) follow the standard POSIX default disposition.
- **§8 — Environment Variables.** Add `LOOPX_TMPDIR` and `LOOPX_WORKFLOW_DIR` to the injected-variables table. `LOOPX_WORKFLOW_DIR` is the absolute path of the currently-spawned script's workflow directory and refreshes per-spawn alongside `LOOPX_WORKFLOW` across intra-workflow `goto`, cross-workflow `goto`, and loop reset. Document that `RunOptions.env` merges after env files (global + local) and before loopx-injected protocol variables. Extend the existing "env files are loaded once at loop start and cached for the duration of the loop" rule (§8.1) to also cover the inherited parent environment: the parent `process.env` is snapshotted once during the pre-iteration sequence and reused for every child spawn in the run; mutations to the caller's `process.env` between spawns are not picked up. Rewrite the §8.3 precedence list (highest wins) to the five tiers:
  1. loopx-injected protocol variables (`LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, `LOOPX_WORKFLOW_DIR`, `LOOPX_TMPDIR`)
  2. `RunOptions.env`
  3. Local env file (`-e` / `RunOptions.envFile`)
  4. Global loopx env file (`$XDG_CONFIG_HOME/loopx/env`)
  5. Inherited system environment (snapshotted once per run)
- **§9.1 / §9.2 — Programmatic API.** Document the new `env` option: its call-time shallow-copy, its merge position, its rejection on invalid shape, and that it is ignored when omitted. Document that the caller's `process.env` is snapshotted once per run and is not re-read between child spawns, so mutations to `process.env` after the run has started iterating are not observed by later children. The snapshot timing is lazy for `run()` (captured at first `next()` alongside the rest of the pre-iteration sequence, so mutations between `run()` returning and first `next()` **are** observed). For `runPromise()`, the inherited-env snapshot timing relative to the `runPromise()` return point is implementation-defined (may be captured during synchronous call setup, during an asynchronous continuation after return, or at any other point no later than the first child spawn), so callers who synchronously mutate `process.env` after `runPromise()` returns cannot rely on those mutations being observed or not observed — either outcome is conforming (§1 "Inherited-env snapshot timing — per-API split, not uniformly 'lazy'"). Document `LOOPX_TMPDIR` creation timing (lazy — on first `next()` for `run()`, asynchronous after return for `runPromise()`) and cleanup under abort and consumer-driven cancellation. Document the invalid-`options` and invalid-`options.signal` validation rules (§1 "Invalid `options` and invalid `options.signal`"): `options` must be omitted, `undefined`, or a non-null non-array object — any other value is captured at call time and surfaced via the pre-iteration error path; `options.signal`, when present and not `undefined`, must be an `AbortSignal`-compatible object exposing readable `.aborted` and `addEventListener('abort', …)` — where `addEventListener` must be callable and must return without throwing when loopx invokes it to register its abort listener. Any non-compatible value (missing `addEventListener`, non-callable `addEventListener`, `addEventListener` that throws during listener registration, throwing `aborted` getter, or any other shape that cannot serve as an `AbortSignal`) is rejected via the same pre-iteration error path rather than being treated as "no signal". `removeEventListener` is not part of the compatibility contract: loopx does not require its presence, and whether loopx invokes `removeEventListener` to clean up its abort listener after run settlement is implementation-defined (§1 "`removeEventListener` is not part of the compatibility contract"). A non-compatible `options.signal` does not enter the abort-precedence pathway (no signal has been captured to observe for abort). Document the invalid-`options.cwd` / invalid-`options.envFile` validation rules (§1 "Invalid `options.cwd`, `options.envFile`, and `options.maxIterations` values"): `options.cwd` and `options.envFile`, when present and not `undefined`, must be strings; any non-string value is captured at call time and surfaced via the standard pre-iteration error path, exactly like other option-snapshot errors. `options.maxIterations` retains its existing SPEC §9.5 validation; throwing getters on any of these fields are captured at call time and surfaced through the same path. Document the options snapshot ordering rule (§1 "Options snapshot ordering"): `options.signal` must be read before `options.env`, `options.cwd`, `options.envFile`, and `options.maxIterations`, so that an already-aborted signal — or a signal that aborts synchronously during the `options.signal` getter read itself — is captured before any other option-field read can produce a snapshot exception. Beyond that first-read placement, the order of the remaining option fields is implementation-defined. Document that each option field (including `options.signal`) is read at most once per call, that the same one-read-per-field rule applies to any property getter or proxy trap reached through that read (a throwing getter must not be re-invoked to retry), and that this rule permits normal one-pass enumeration of an `options.env` proxy (a single `ownKeys` invocation followed by one `getOwnPropertyDescriptor` / get-trap invocation per enumerated own string-keyed property) but forbids loopx from performing an additional enumeration pass. Qualify SPEC §9.1's existing "aborting the signal always produces an error, even if it occurs before the first `next()` call" language with the pre-first-`next()` consumer-cancellation carve-out (§1 "Pre-first-`next()` consumer cancellation wins over all captured pre-iteration state — signals and option-snapshot errors alike"): a consumer whose *first* interaction with the generator returned by `run()` is `.return()` or `.throw()` settles the generator per standard JS async-generator semantics without observing signal state and without surfacing any captured call-time option snapshot error (throwing option-field getters on the options object, invalid `options` values, invalid `options.signal` values, invalid `options.env` shapes, throwing `options.env` enumerable getters or proxy traps, etc.), even when the captured signal was already aborted or the call-time snapshot captured a deferred error; once the consumer's first interaction is `next()`, abort precedence and captured option-snapshot errors apply as before. This carve-out is specific to `run()`; `runPromise()` has no equivalent consumer interaction point, so an already-aborted signal passed to `runPromise()` still rejects the promise, and a captured call-time option snapshot error still rejects the promise through the same asynchronous path.
- **§9.3 — Error Behavior.** Extend the rejection / throw surface to cover the new pre-iteration failure modes introduced by this ADR: `LOOPX_TMPDIR` creation failure (`mkdtemp`, identity-capture, or mode-securing fails); child launch / spawn failure after tmpdir creation (discovered script removed or renamed per §5.1, runtime rejection of a child environment entry from any env tier — inherited `process.env`, global or local env file, or `RunOptions.env` — most reliably an entry whose name or value contains an embedded NUL byte, `exec` failure); `RunOptions.env` shape or snapshot-time errors (invalid shape, throwing `ownKeys` trap, throwing enumerable getter, throwing `options.env` getter on the options object); the new invalid-`options` and invalid-`options.signal` rejections (§1 "Invalid `options` and invalid `options.signal`"); and the new non-string `options.cwd` / `options.envFile` and throwing `options.cwd` / `options.envFile` / `options.maxIterations` getter rejections (§1 "Invalid `options.cwd`, `options.envFile`, and `options.maxIterations` values"). Document that cleanup runs before the generator throws or the promise rejects (the abort outcome under `AbortSignal`, the spawn error under child launch / spawn failure, the creation error under tmpdir-creation failure), so cleanup ordering is observable from the caller. Document that cleanup warnings emitted during teardown (any of §1's three categories — leave-in-place refusal, top-level cleanup failure, or partial-directory cleanup failure) do not mask the original rejection reason: the error surfaced to the caller is the underlying failure, not the cleanup warning. Document abort precedence across pre-iteration (§1 "Abort precedence across pre-iteration"): once loopx has successfully captured a usable `AbortSignal`, an already-aborted or mid-pre-iteration abort wins over later `RunOptions` field validation (including `options.env` shape and snapshot-time exceptions, non-string `options.cwd` / `options.envFile` values, and throwing `options.env` / `options.cwd` / `options.envFile` / `options.maxIterations` getters read after `options.signal`), target argument validation (non-string `target`), target syntax validation (per SPEC §4.1), `.loopx/` discovery, env-file loading, target resolution, and tmpdir-creation errors, and becomes the terminal rejection reason. Invalid `options` values (non-object) and non-`AbortSignal`-compatible `options.signal` values capture no signal and therefore do not enter this pathway — they are ordinary options-snapshot errors surfaced through the standard pre-iteration path. Version checking is part of pre-iteration but produces non-fatal warnings rather than errors per SPEC §3.2, so it is not a competing terminal-rejection category; an abort observed during or after the version-check step still becomes the terminal rejection reason. Document that relative priority among non-abort pre-iteration errors is implementation-defined for the programmatic API only (§1 "Non-abort pre-iteration error priority (programmatic API only)") — the CLI retains SPEC §7.1's concrete ordering and this ADR does not loosen it. For example, when `options.env` shape is invalid and `.loopx/` is missing, either error is an acceptable terminal outcome for a programmatic call. Document the extended pre-first-`next()` consumer-cancellation carve-out (§1 "Pre-first-`next()` consumer cancellation wins over all captured pre-iteration state — signals and option-snapshot errors alike"): a `.return()` / `.throw()` call on a generator that has not yet had `next()` invoked settles the generator per standard JS async-generator semantics without observing `AbortSignal` state and without surfacing any captured call-time option snapshot error (throwing option-field getters, invalid `options` values, invalid `options.signal` values, invalid `options.env` shapes, throwing `options.env` enumerable getters or proxy traps, and equivalents for other fields), so abort precedence and captured option-snapshot surfacing apply only once the consumer's first interaction is `next()`. This carve-out applies only to `run()`; `runPromise()` has no equivalent consumer interaction point. Document that for `run()`, cleanup on normal completion (`stop: true` or `maxIterations` reached) is guaranteed only once the async generator is driven to settlement (`{ done: true }`, `.return()`, `.throw()`, or a `for await` loop that runs to completion); the final yielded `Output` is not itself settlement, so a caller that consumes the final output and abandons the generator without driving it further may leak the tmpdir. Document the abort-after-final-yield rule (§1 "Abort after the final yielded `Output` still wins until generator settlement"): an `AbortSignal` that aborts after the final yielded `Output` but before generator settlement produces the abort error on the next generator interaction — the final yield does not commit normal completion, which is fixed only at settlement.
- **§9.5 — Types.** Add `env?: Record<string, string>` to `RunOptions`. Update the `cwd?: string` prose: `cwd` now specifies both the project root (used for `.loopx/` resolution and `LOOPX_PROJECT_ROOT`) and the script execution cwd. Document explicit type requirements for `cwd` and `envFile`: when present and not `undefined`, each must be a string, and any non-string value (number, boolean, array, object, `null`, function, or any other non-string non-`undefined` value) is captured at call time and surfaced via the standard pre-iteration error path per §1 "Invalid `options.cwd`, `options.envFile`, and `options.maxIterations` values". Document that a relative `cwd` is resolved to an absolute path at call time against the caller's `process.cwd()` (equivalent to `path.resolve(process.cwd(), options.cwd)`) and that no further `realpath` is applied; an absolute `cwd` is used unchanged. Drop the prior disclaimer that `cwd` does not control script execution cwd. Preserve the existing relative-`envFile` rule, restated to match the new meaning of `cwd`: a relative `envFile` resolves against the resolved project root (equal to the child's cwd) when `cwd` is supplied, and against the caller's `process.cwd()` at call time otherwise. For example, `run("ralph", { cwd: "./project", envFile: ".env" })` from a caller whose `process.cwd()` is `/p` loads `/p/project/.env`, not `/p/.env`.
- **§12 — Exit Codes.** Extend the code-`1` enumeration to cover `LOOPX_TMPDIR` creation failure (`mkdtemp`, identity capture, or mode-securing fails) and child launch / spawn failure after tmpdir creation (discovered script removed or renamed between discovery and spawn per §5.1, runtime rejection of a child environment entry from any env tier — inherited `process.env`, global or local env file, or `RunOptions.env` — most reliably an entry whose name or value contains an embedded NUL byte, `exec` failure, or any other pre-first-line-of-user-code spawn-path error). Signal exit codes remain `128 + N`; `LOOPX_TMPDIR` cleanup runs before loopx exits with the signal's code (§7.3). Cleanup warnings emitted during tmpdir teardown — whether from the leave-in-place, top-level cleanup failure, or partial-directory cleanup failure category (§1 "Warning cardinality") — do not affect the exit code, the generator's outcome under `run()`, or the rejection reason under `runPromise()`.
- **§13 — Reserved Values.** Add `LOOPX_TMPDIR` and `LOOPX_WORKFLOW_DIR` as reserved env var names. Clarify that the §13 reserved list now covers two distinct roles — "startup-reserved" and "script-protocol-protected" — which this ADR makes explicit so the §13 table is not misread as a single "every reserved name is protected at script spawn" category:
  - **Script-protocol-protected** (loopx injects the variable on every child spawn and silently overrides any user-supplied value at the §8.3 top tier): `LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, `LOOPX_WORKFLOW_DIR`, `LOOPX_TMPDIR`.
  - **Startup-reserved only** (loopx inspects the variable on its own process startup — for example, as a delegation recursion guard per SPEC §3.2 — but does not inject or protect it on per-spawn script environments): `LOOPX_DELEGATED`. SPEC §3.2 sets `LOOPX_DELEGATED=1` only on the inner loopx process during project-root delegation, not on every script spawn. This ADR does not add `LOOPX_DELEGATED` to the script-protocol-protected tier; `RunOptions.env`, env-file entries, and inherited `process.env` may therefore supply `LOOPX_DELEGATED` to spawned scripts without a loopx-side per-spawn override, and a script that invokes `loopx run <subtarget>` as a subprocess observes whatever `LOOPX_DELEGATED` value its own environment held at spawn time (existing SPEC §3.2 behavior, unchanged). A later ADR may revisit whether to promote `LOOPX_DELEGATED` into the script-protocol-protected tier if evidence of need accumulates.
  
  The §13 table entry for `LOOPX_DELEGATED` should be updated to reflect the startup-reserved-only role explicitly, so that readers do not infer protection parity with `LOOPX_BIN` / `LOOPX_PROJECT_ROOT` / `LOOPX_WORKFLOW` / `LOOPX_WORKFLOW_DIR` / `LOOPX_TMPDIR`. `PWD` is **not** added to the reserved list under either role: loopx does not manage the child's `PWD` (see §3 "`PWD` is not a protocol variable").

## Test Recommendations

Non-exhaustive — these highlight edge cases that are easy to overlook.

### `LOOPX_TMPDIR`

- `LOOPX_TMPDIR` is injected with an absolute path; its basename begins with `loopx-` and lives under `os.tmpdir()`.
- Same value is observed across the starting target, intra-workflow `goto`, cross-workflow `goto`, and loop reset to the starting target.
- Files written by one script are visible to subsequent scripts in the same run.
- Parallel `loopx run` invocations receive distinct paths and distinct directories.
- Directory mode is `0700`.
- Cleanup runs on each terminal outcome: normal completion (`stop: true` and `maxIterations` reached), non-zero script exit, invalid `goto` target, child launch / spawn failure after tmpdir creation (discovered script removed or renamed before spawn, runtime rejection of a child environment entry from any env tier — inherited `process.env`, env file, or `RunOptions.env` — most reliably an entry whose name or value contains an embedded NUL byte, `exec` failure), SIGINT/SIGTERM (including between iterations and between tmpdir creation and first spawn), `AbortSignal` abort (including mid-iteration, between iterations, and after the final yield but before `{ done: true }`), `break` from `for await`, explicit `generator.return()` after first `next()`, explicit `generator.throw(err)` after first `next()`.
- Cleanup ordering across the final yield (`run()` generator, normal completion): a consumer that manually iterates `g.next()` and observes the final yielded `Output` (produced by `stop: true` or `maxIterations` reached) sees the tmpdir still present during its observation of that yielded `Output`; the subsequent `await g.next()` resolves with `{ done: true }`, and the tmpdir has been cleaned up by the time that `{ done: true }` resolve is observed. A consumer driving the generator via `for await (...)` observes cleanup before the `for await` loop exits, matching the general "cleanup runs before the generator settles terminally" rule. A consumer that receives the final yielded `Output` and then abandons the generator without calling `g.next()` again, `.return()`, `.throw()`, or driving it to `{ done: true }` may leak the tmpdir: loopx makes no cleanup guarantee in that case, so tests relying on cleanup under manual iteration must drive the generator to settlement explicitly.
- Cleanup of a symlink inside `$LOOPX_TMPDIR` pointing outside: symlink entry is unlinked, target is untouched.
- Cleanup when the tmpdir path has been replaced: symlink replacement is unlinked (target untouched); regular-file / FIFO / socket replacement is left in place with a warning; mismatched-identity directory replacement is left in place with a warning; renamed-away tmpdir is left at its new path without being chased.
- Hard-link safety: a script that creates a hard link at the tmpdir path to unrelated data leaves the link count of the shared inode unchanged after cleanup, and the unrelated target file is untouched.
- Rename-into-path safety: a script that renames an unrelated file into the tmpdir path leaves the file's data intact after cleanup.
- Warning cardinality, closed enumeration: a cleanup whose top-level `LOOPX_TMPDIR` path is a non-directory non-symlink replacement (case 3) or a mismatched-identity directory (case 5) emits a single category-1 warning for that top-level path; category 2 covers three mutually exclusive top-level dispatch-failure sub-cases — (a) top-level `lstat` failure other than ENOENT, (b) failed `unlink` of a top-level symlink replacement (case 2), and (c) failed recursive removal of an identity-matched directory (case 4, subsuming per-entry failures across many nested entries) — and emits a single category-2 warning for any of them; a creation-failure path running the non-recursive partial-directory cleanup path (identity-not-captured regime) whose `rmdir` fails emits a single category-3 warning. A creation-failure cleanup with identity captured runs the full cleanup-safety routine and may therefore emit a category-1 or category-2 warning exactly as terminal-outcome cleanup does — never category 3. Only one routine runs per cleanup attempt and each routine emits at most one warning, so a cleanup emits at most one warning in total. The set is closed: tests may assert that no cleanup warning outside categories 1, 2, and 3 is produced. The CLI exit code, generator outcome, and promise rejection reason are unchanged by cleanup warnings regardless of cardinality.
- Category tokens are observable: each emitted cleanup warning contains exactly one of `[leave-in-place-refusal]` (category 1), `[top-level-cleanup-failure]` (category 2), or `[partial-directory-cleanup-failure]` (category 3) as a bracketed substring. Tests assert category identity by matching on token presence rather than on the surrounding implementation-defined prose: for example, a fixture that places a regular file at the tmpdir path asserts exactly one stderr warning containing `[leave-in-place-refusal]` and zero warnings containing `[top-level-cleanup-failure]` or `[partial-directory-cleanup-failure]`; a fixture that forces `unlink` failure on a top-level symlink replacement asserts exactly one stderr warning containing `[top-level-cleanup-failure]`; a creation-failure fixture that forces identity capture to raise and then forces `rmdir` to fail asserts exactly one stderr warning containing `[partial-directory-cleanup-failure]`. Tests must not match on any text beyond the token (path, errno, surrounding prose are implementation-defined), must not assume a specific rewriting convention or prefix format (see §1 "Token placement and user-controlled text" — implementations must use prefix placement together with rewriting or omission of user-controlled text to guarantee the cardinality-of-one rule when rendered paths or errno strings could otherwise contribute additional token matches; tests must be compatible with any conforming approach), and must not assert that more than one distinct category token appears in a single warning.
- User-controlled-path token disambiguation: when the tmpdir path (influenced by `TMPDIR` / `TEMP` env values the caller controls) contains a substring that matches one of the three category tokens — for example, a test fixture that sets `TMPDIR` to a directory ancestor named `[top-level-cleanup-failure]` — a cleanup warning tied to a real cleanup failure for that path must still contain exactly one category token corresponding to the failure category, as interpreted by a plain-text substring match. Tests that construct this scenario must tolerate implementation-defined rewriting of the user-controlled text (e.g., backslash-escaping of `[` / `]`, substituting a replacement character, redacting the matched substring, or omitting the rendered path entirely). Surrounding the user-controlled text in quotes alone would **not** satisfy the rule — the quoted string still contains the token substring — so tests must not interpret a "quoted path still containing the token" implementation as conforming. A warning that contains the loopx-emitted token and a separately-rewritten rendering of the matching path text is conforming; a warning that contains two plain-text occurrences of any category token (from the same or different categories) is non-conforming. This test exercises the §1 "Token placement and user-controlled text" rule and is intentionally narrow — tests must not assert a specific rewriting convention.
- Token scoping is narrow — only cleanup warnings carry these tokens: existing non-cleanup loopx warnings (version-check warnings from SPEC §7.2's local-workflow version-mismatch path, warnings about unreadable `package.json` during version check, warnings about discovered but invalid workflow or script entries per SPEC §5.1, warnings about ignored invalid lines in env files per SPEC §8.1, and any other non-cleanup warning category loopx emits or may emit in the future) must not contain any of `[leave-in-place-refusal]`, `[top-level-cleanup-failure]`, or `[partial-directory-cleanup-failure]`. Tests that assert "no cleanup warning outside the three categories is produced" must filter for cleanup warnings — for example, by asserting that every stderr warning whose text contains any of the three bracketed tokens is a cleanup warning and conversely that no non-cleanup warning (as identified by its own category or message) contains any of those tokens. A run that produces both a non-cleanup warning (e.g., a version mismatch) and a cleanup warning (e.g., `[top-level-cleanup-failure]`) emits both, and the tokens identify the cleanup warning alone. A run that produces only a non-cleanup warning (e.g., an env-file invalid-line warning per SPEC §8.1 under `maxIterations: 0` — env-file loading still runs under `-n 0` / `maxIterations: 0`, while workflow-level version checking is skipped per SPEC §3.2 / §7.1 / §4.2 and no tmpdir is created for such a run, so no cleanup warning is emitted either) emits no tokens.
- Cleanup-internal failures emit category-2 warnings, not silent success. A simulated top-level `lstat` failure (non-ENOENT, e.g., via a path permission regime that denies `stat` without ENOENT) emits exactly one category-2 warning and leaves the top-level path entry in place (the dispatch aborts before any `unlink` or removal is attempted, so no entry at or below the path is modified). A simulated `unlink` failure on a top-level symlink replacement emits exactly one category-2 warning and leaves the symlink entry in place (only the top-level symlink entry is targeted by the `unlink`, and the target is never followed regardless of outcome). A simulated recursive-removal failure on an identity-matched directory emits exactly one category-2 warning; tests may assert that symlink targets outside the tmpdir remain untouched (since symlink entries encountered during the walk are unlinked but not traversed) and that no non-identity-matched top-level replacement was recursively removed, but must not assume the identity-matched tmpdir remains fully intact — recursive deletion can fail after partially deleting entries, so the top-level directory or any subset of its contents may remain depending on where the failure occurred, and this ADR does not require staged or atomic recursive removal. Filesystem identity of surrounding data (outside the tmpdir) is unchanged by these failed cleanups, and the CLI exit code / generator outcome / promise rejection reason remain those of the terminal outcome that triggered cleanup (cleanup warnings do not mask or alter the underlying rejection reason).
- Under `-n 0` / `maxIterations: 0`, no tmpdir is created.
- When pre-spawn validation fails (discovery error, env-file error, target-resolution error, invalid `RunOptions.env` shape, `RunOptions.env` snapshot-time exception from a throwing `ownKeys` trap or enumerable getter) or an `AbortSignal` is already aborted before tmpdir creation, no tmpdir is created.
- When tmpdir creation itself fails after producing a partial directory, best-effort cleanup runs on the partial directory without masking the original creation error.
- Partial-directory cleanup is dispatched by whether identity was captured: with identity recorded, the full "Cleanup safety" routine applies (identity-matched recursive removal, leave-in-place for mismatched identity, etc.) and may emit a category-1 or category-2 warning — never category 3 — under any of the category-2 sub-cases (top-level `lstat` failure other than ENOENT, failed `unlink` of a top-level symlink replacement, failed recursive removal of an identity-matched directory) or the category-1 dispatch cases (3 or 5), exactly as under terminal-outcome cleanup; with identity capture itself having failed, loopx attempts a single non-recursive `rmdir` on the path via the non-recursive partial-directory cleanup path and, if it fails for any reason, emits one category-3 warning and leaves the path in place — loopx does not `unlink`, `lstat`-dispatch, or recursively remove without recorded identity.
- Identity-capture-failure unrelated-empty-directory replacement is outside the race-resistant guarantee: in the identity-not-captured regime, if a same-user process replaces the original `mkdtemp` directory with a different empty directory between `mkdtemp` and the partial-directory `rmdir`, loopx's `rmdir` removes that unrelated empty directory without detection. Tests must not assume loopx detects or declines this replacement (it has no recorded identity to compare against) and must not include this scenario as a positive "loopx refuses to remove unrelated data" assertion. Tests that deliberately construct this race should assert only that the partial-directory cleanup path remains non-recursive (no `unlink` of symlink entries, no traversal into a non-empty directory, no removal of a non-directory) regardless of what the current inode at the path names. This matches the "not a sandbox against actively racing same-user processes" posture applied elsewhere.
- A user-supplied `LOOPX_TMPDIR` in inherited env, the `-e` local env file, the global env file, or `RunOptions.env` is overridden by the injected protocol value.
- CLI SIGINT / SIGTERM during pre-iteration wins over non-signal pre-iteration failures: a CLI `loopx run <target>` process sent SIGINT after usage-level argument parsing has completed but before the first child spawns exits with code `130` (SIGINT) or `143` (SIGTERM) regardless of whether a concurrent pre-iteration failure — missing `.loopx/`, invalid target syntax for the accepted single candidate target, env-file loading error, or tmpdir-creation error — would otherwise have surfaced. Workflow-level version mismatches are not on this list because they are non-fatal warnings per SPEC §3.2, not failure modes that compete for the terminal outcome; whether a pending version-check warning reaches stderr before the signal's exit is timing-dependent and not contractually constrained — tests that construct this race must not assert either that the warning is or is not emitted. If tmpdir creation had completed before the signal arrived, tmpdir cleanup runs before exit (observable via absence of `loopx-*` entries under `os.tmpdir()` after the process exits); if tmpdir creation had not yet begun or failed before the signal, no cleanup is required and the process exits directly with the signal code. Tests that construct this race must tolerate either observable outcome (signal exit vs. underlying pre-iteration-error exit) for signals that arrive *before* loopx's pre-iteration signal handlers are installed (during process startup, argument parsing, or `run -h` / `--help` short-circuit handling); signals after handler installation are required to produce the signal-wins outcome.
- CLI usage-level errors are not displaced by signal-wins precedence (§1 "Scope of CLI signal-wins precedence"): a CLI `loopx run` invocation with a parser-level usage error — no positional target (`loopx run`), multiple positionals (`loopx run a b`), duplicate `-n` or `-e`, an invalid `-n` value (e.g., `-n nope`, `-n -5`), a missing `-e` value (`loopx run -e` with no argument), an unrecognized flag (`loopx run --nope ralph`), or a `--` token outside `run -h` / `--help` (`loopx run -- ralph`, `loopx run -n 1 -- ralph`, `loopx run ralph -- name=value`) — exits `1` with the parser's usage error message regardless of whether a signal is delivered during argument parsing, because loopx has not yet installed its pre-iteration signal handlers at that point and signals before handler installation follow the standard POSIX default disposition. Tests that construct this race must tolerate either observable outcome (usage-error exit `1` vs. standard POSIX signal disposition) for the signal-during-parsing window and must not assert that the signal displaces the usage error.

### `RunOptions.env`

- `run(target, { env: { adr: "0003" } })` results in `process.env.adr === "0003"` inside the script, and `$adr == "0003"` in a Bash script.
- Same `env` applied to every iteration, every `goto` destination, and every loop reset.
- Overrides same-named entries from `-e` local env file, global loopx env file, and inherited `process.env`.
- Overridden by loopx protocol variables: `env: { LOOPX_TMPDIR: "/fake" }` is not observable in the child.
- `run(target, { env: undefined })`, `run(target, { env: {} })`, and `run(target, {})` all inject no additional entries.
- Invalid shapes reject on first iteration (for `run()`) or via promise rejection (for `runPromise()`): `env: null`, `env: []`, `env: "nope"`, `env: 42`, `env: { x: 42 }` (non-string value).
- Mutating the original `env` object after `run()` / `runPromise()` returns does not affect the running loop (call-time shallow copy).
- Symbol-keyed, non-enumerable, and inherited properties of `env` are ignored — neither validated nor forwarded.
- Snapshot-time exceptions on exotic `env` values (a `Proxy` whose `ownKeys` trap throws, a throwing enumerable getter, etc.) do not escape `run()` / `runPromise()` synchronously — they surface via the standard pre-iteration error path.
- Options snapshot ordering — `options.signal` read first: passing an already-aborted `AbortSignal` alongside a throwing `env` getter (`{ signal: alreadyAbortedSignal, get env() { throw new Error("env getter failed"); } }`) surfaces the abort as the terminal outcome, not the `env`-getter error; symmetrically, passing an already-aborted signal alongside an invalid `env` shape (e.g., `env: null`) or alongside a throwing `options.cwd` / `options.envFile` / `options.maxIterations` getter surfaces the abort. This exercises the rule that `options.signal` is read before other option fields, so the signal is captured before any other option-field read can produce a snapshot exception. A signal whose own getter throws (e.g., `get signal() { throw new Error("signal getter failed"); }`) surfaces as an ordinary options snapshot error, not as an abort, because no signal has been captured to observe.
- Each option field is read at most once per call: an `options.env` getter that increments a counter on each read, attached to a run that proceeds to completion, observes exactly one read. This applies equally to `options.signal`, `options.cwd`, `options.envFile`, and `options.maxIterations`; implementations must not re-read a getter for retry or any other reason.
- Non-abort pre-iteration error priority is implementation-defined: a call that combines two non-abort pre-iteration failures (for example, invalid `env` shape *and* a missing `.loopx/` directory, or a throwing `env` getter *and* a nonexistent `options.envFile`) may surface either error as the terminal outcome. Tests for such combinations assert only that the terminal outcome is one of the two expected errors and is not an unrelated error type; they must not assume a specific ordering between non-abort failure modes.
- Runtime-rejected entries — most reliably, a name or value containing an embedded NUL byte — surface only at spawn time. Under CLI `-n 0` or `maxIterations: 0`, which never spawns a child, such an entry produces no error; running the same options with `maxIterations: 1` surfaces the spawn failure. Entries that are malformed but runtime-accepted (for example, a name containing `=`, or an empty-string name) have runtime-dependent child-observed behavior — tests must not assert that loopx rejects them at spawn, since the runtime may forward them to the child unchanged (for example, a `{ "A=B": "C" }` entry may appear in the child as key `A` with value `B=C` rather than a spawn failure, and empty-string names may be silently dropped). Only loopx's shape rules and the runtime's own rejections are observable; loopx does not add its own name validation.
- Shape validation is independent of spawn: invalid `env` shapes (`env: null`, `env: []`, `env: "nope"`, `env: 42`, `env: { x: 42 }`) reject under `maxIterations: 0` / CLI `-n 0` via the standard pre-iteration error path, exactly as they do under any other iteration count — no child is required to detect a shape error.
- Inherited-env snapshot is per run, not per spawn (programmatic): setting `process.env.FOO = "before"` before calling `run()`, mutating `process.env.FOO = "after"` between iterations (e.g., from a `yield` consumer or a timer that fires during a script spawn), and asserting `FOO === "before"` in every child (starting target, intra- and cross-workflow `goto` destinations, and loop reset to the starting target). The CLI case is naturally fixed by POSIX process inheritance — another process cannot mutate the already-running loopx CLI process's environment, so `FOO=before loopx run <target>` observes `FOO=before` in every child without any loopx-side snapshot rule being observable. CLI inherited-env tests therefore focus on env-file caching and precedence (see next bullet and the existing SPEC §8.1 "env files loaded once at loop start" coverage), not on concurrent in-process env mutation, which is only meaningful under the programmatic API where the same JS process can mutate `process.env` between generator interactions.
- Inherited-env snapshot timing — lazy for `run()`, implementation-defined for `runPromise()`: for `run()`, set `process.env.FOO = "call-time"`, call `const g = run(target)` (no iteration yet), set `process.env.FOO = "after-return-before-first-next"`, then `await g.next()`. The first spawned child observes `FOO === "after-return-before-first-next"` — the mutation between return and first `next()` is captured because the pre-iteration sequence (including the inherited-env snapshot) runs at first `next()`. A subsequent mutation `process.env.FOO = "after-first-next"` between iterations is not observed by later children (they still see `"after-return-before-first-next"`). This distinguishes inherited env from `RunOptions.env` and `RunOptions.cwd`, both captured at call time: a parallel test that passes `FOO` via `RunOptions.env` at call time and mutates `process.env.FOO` before first `next()` must observe the call-time `RunOptions.env` value in every child. For `runPromise()`, the scheduling point at which inherited `process.env` is snapshotted is implementation-defined (§2 "Inherited-env snapshot timing — per-API split, not uniformly 'lazy'"); a test that calls `runPromise(target)`, synchronously mutates `process.env.FOO` before any `await`, and then awaits the promise must tolerate the child observing either the pre-mutation or post-mutation value — either outcome is conforming. Tests that need a deterministic per-run value under `runPromise()` must use `RunOptions.env` rather than relying on inherited-env mutation timing.
- Abort precedence covers target argument validation and target syntax validation: `run(undefined as any, { signal: alreadyAbortedSignal })`, `run(null as any, { signal: alreadyAbortedSignal })`, `run(42 as any, { signal: alreadyAbortedSignal })`, `run("" , { signal: alreadyAbortedSignal })`, `run(":", { signal: alreadyAbortedSignal })`, `run("workflow:", { signal: alreadyAbortedSignal })`, and `run("a:b:c", { signal: alreadyAbortedSignal })` each surface the abort error as the terminal outcome on first `next()`, not the target-validation error. Without the aborted signal, the same calls would surface the respective target-validation rejection via SPEC §9.1's lazy rejection path; adding an already-aborted signal replaces that rejection with the abort error, consistent with the unified abort-wins-over-pre-iteration-failures model.
- Pre-first-`next()` consumer cancellation wins over an already-aborted signal (carve-out from abort precedence): `const g = run(target, { signal: alreadyAbortedSignal }); await g.return("done");` settles with `{ value: "done", done: true }` — not an abort error — because `.return()` on a pre-first-`next()` generator does not enter the body and therefore does not observe signal state. Symmetrically, `const g = run(target, { signal: alreadyAbortedSignal }); await g.throw(new Error("consumer")).catch(e => e);` surfaces `Error("consumer")` — not the abort error. The carve-out is tight: `const g = run(target, { signal: alreadyAbortedSignal }); await g.next().catch(e => e);` surfaces the abort error (baseline — `next()` enters the body and observes the abort), and a subsequent `.return()` after a first `next()` follows the existing consumer-driven cancellation rules rather than this carve-out. `runPromise(target, { signal: alreadyAbortedSignal })` returns a promise that rejects with the abort error (no consumer-driven pre-first-interaction API exists for `runPromise()`).
- Pre-first-`next()` consumer cancellation also wins over captured call-time option snapshot errors (extending the carve-out beyond signals to all captured pre-iteration state): `const g = run(target, { get env() { throw new Error("env getter failed"); } }); await g.return("done");` settles with `{ value: "done", done: true }` — not the `env`-getter error — because `.return()` on a pre-first-`next()` generator does not enter the body and therefore does not surface any captured call-time option snapshot error. The same holds for other captured call-time errors: an invalid `env` shape (`run(target, { env: null as any })`, `run(target, { env: [] as any })`, `run(target, { env: { x: 42 as any } })`), a throwing `ownKeys` trap or throwing enumerable getter on an `options.env` proxy, an invalid `options` value (`run(target, null as any)`, `run(target, 42 as any)`, `run(target, [] as any)`), an invalid `options.signal` value (`run(target, { signal: "not a signal" as any })`, `run(target, { signal: { aborted: true } as any })`), a throwing `options.env` / `options.cwd` / `options.envFile` / `options.maxIterations` getter on the options object, and combinations of any of these with an already-aborted signal — in every such case, a pre-first-`next()` `.return(value)` settles `{ value, done: true }` without surfacing any of these errors, and a pre-first-`next()` `.throw(err)` throws `err` without surfacing them. The carve-out remains tight: the consumer's *first* interaction being `next()` surfaces the captured error (or the abort, when both are present — abort wins over any captured option-snapshot error per "Abort precedence across pre-iteration"), and a subsequent `.return()` after a first `next()` follows the existing consumer-driven cancellation rules. `runPromise()` still rejects with any of these captured call-time errors (no equivalent pre-first-interaction carve-out exists for `runPromise()`).
- Abort after the final yielded `Output` still wins until generator settlement: a consumer that manually iterates a run expected to complete normally (`stop: true` or `maxIterations`) observes the final yielded `Output`, then aborts the `AbortController` *before* calling `await g.next()` again; the subsequent `await g.next()` throws the abort error rather than resolving `{ done: true }`. Concretely: `const controller = new AbortController(); const g = run(target, { signal: controller.signal }); const { value: finalOutput } = await g.next(); /* ... assertions on finalOutput ... */ controller.abort(); await g.next().catch(e => e);` surfaces the abort error on the second `g.next()`, not `{ done: true }`. Tmpdir cleanup runs before the abort error is surfaced. A parallel test using `g.return("done")` after the abort is observed must also surface the abort error rather than `{ value: "done", done: true }`, because the final yield does not commit normal completion. The baseline case — consumer drives the generator to `{ done: true }` *before* aborting — settles normally; an abort observed after settlement is a no-op. This is the after-final-yield extension of the "abort wins until generator settlement" rule.
- Options snapshot one-pass enumeration (clarifying the "at most once per call" rule for proxies): an `options.env` value that is a `Proxy` with counters on `ownKeys`, `getOwnPropertyDescriptor`, and per-key `get` traps observes exactly one `ownKeys` invocation over the whole run and exactly one `getOwnPropertyDescriptor` (and, where applicable, one `get`) per enumerated own string-keyed property — not zero (loopx must actually read the keys to forward them) and not more than one (loopx must not perform an additional enumeration pass, and must not re-invoke a throwing trap to retry). This test exercises the refined "one-pass enumeration permitted, additional passes forbidden" rule (§1 "Options snapshot ordering").
- Invalid `options` value is captured at call time and surfaced via the pre-iteration error path (not at the call site): `run(target, null as any)`, `run(target, 42 as any)`, `run(target, "nope" as any)`, `run(target, true as any)`, `run(target, [] as any)`, `run(target, [{} as any] as any)`, and `run(target, (() => {}) as any)` each return a generator without throwing at the call site; the corresponding error is raised on the first `next()`. The parallel `runPromise(target, <invalid-options>)` calls return promises without throwing at the call site; the promises reject. No tmpdir is created in any of these cases (creation is gated on successful pre-iteration). A pre-first-`next()` `.return()` / `.throw()` on the returned generator suppresses the captured invalid-`options` error per the carve-out above.
- Invalid `options.signal` value is captured at call time and surfaced via the pre-iteration error path: `run(target, { signal: "not a signal" as any })`, `run(target, { signal: 42 as any })`, `run(target, { signal: { aborted: true } as any })` (missing `addEventListener`), `run(target, { signal: { aborted: false, addEventListener: "not callable" as any } as any })`, `run(target, { signal: { get aborted() { throw new Error("aborted getter failed"); }, addEventListener: () => {} } as any })`, and `run(target, { signal: { aborted: false, addEventListener: () => { throw new Error("addEventListener threw during registration"); } } as any })` (the throwing-`addEventListener` subcase — no usable signal is captured, and the thrown error is surfaced as an options-snapshot error) each return a generator without throwing at the call site; the corresponding error is raised on the first `next()`. `runPromise()` variants reject. Because `options.signal` is read first (per "Options snapshot ordering"), a non-compatible `signal` combined with an invalid `env` shape or a throwing `options.cwd` getter surfaces the non-compatible-`signal` error as the terminal pre-iteration outcome, not the later option-field error. A non-compatible `options.signal` does not enter the abort-precedence pathway — no signal has been captured to observe for abort — so an invalid-`options.signal` run with a deliberately thrown consumer error via `.throw()` on pre-first-`next()` still surfaces the consumer error per the carve-out, not the non-compatible-`signal` error. Tests must **not** assert whether loopx invokes `removeEventListener` on a supplied signal that does expose it (either outcome is conforming per "`removeEventListener` is not part of the compatibility contract" in §1), and must not treat absence of `removeEventListener` as a compatibility violation.
- Invalid `options.cwd` / `options.envFile` type (present, not `undefined`, non-string) is captured at call time and surfaced via the pre-iteration error path: `run(target, { cwd: 42 as any })`, `run(target, { cwd: null as any })`, `run(target, { cwd: {} as any })`, `run(target, { cwd: [] as any })`, `run(target, { cwd: true as any })`, `run(target, { cwd: (() => {}) as any })`, and the same shapes for `envFile` each return a generator without throwing at the call site; the corresponding error is raised on the first `next()`. `runPromise()` variants reject. No tmpdir is created. A pre-first-`next()` `.return(value)` settles the generator with `{ value, done: true }` without surfacing the captured type error; a pre-first-`next()` `.throw(consumerErr)` throws `consumerErr` without surfacing the captured type error. When combined with an already-aborted `options.signal` (`{ signal: alreadyAbortedSignal, cwd: 42 as any }`), the signal is captured first (per "Options snapshot ordering") so `.next()` surfaces the abort error; `.return()` / `.throw()` as the first consumer interaction still suppress both the captured type error and the abort per the carve-out. `run(target, { maxIterations: "nope" as any })`, `run(target, { maxIterations: -1 })`, and `run(target, { maxIterations: NaN })` follow the same pattern under SPEC §9.5's existing `maxIterations` validation.
- CLI `run -h` short-circuit preservation: this ADR's "no named-argument tail" and "no `--` outside help" rules do not alter SPEC §4.2's help short-circuit. `loopx run ralph -h`, `loopx run -h ralph`, `loopx run ralph -h extra`, `loopx run ralph -- name=value -h`, `loopx run -h ralph -- foo=bar`, `loopx run -h -- ralph`, `loopx run -h ralph -e missing.env`, and `loopx run -h ralph --unknown` all exit `0` with run help; `name=value` tokens, the `--` marker in any position, extra positionals, unrecognized flags, and invalid flag values are not rejected under the help short-circuit. These tests pair with the existing SPEC §4.2 coverage; they exist in this ADR's test surface to confirm that the new "no named-argument tail" and "no `--` outside help" rules do not regress help-short-circuit behavior. Absent `-h` / `--help`, all of the following forms exit `1` under the existing §4.1 / §4.2 rejection rules, but they split across two distinct exit-`1` categories that SPEC §12 treats as separate kinds of failure: the `--`-bearing forms and the multiple-positionals form are **parser-level usage errors** (surfaced at argument-parse time with a CLI usage diagnostic), while the sole-positional `name=value` form is a **target-validation failure** (surfaced in the pre-iteration phase from SPEC §4.1 target validation, at the same phase and with the same kind of diagnostic as other invalid target strings like `""`, `":"`, or `"a:b:c"`). Concretely: `loopx run -- ralph` → usage error (unrecognized `--` — no end-of-options support outside help); `loopx run -n 1 -- ralph` → usage error (unrecognized `--`); `loopx run ralph -- name=value` → usage error (unrecognized `--` / no supported `--` tail); `loopx run ralph name=value` → usage error (multiple positionals); `loopx run name=value` → target-validation failure (the single positional is parsed as a target string and fails SPEC §4.1 target validation because `=` is not part of the `[a-zA-Z0-9_][a-zA-Z0-9_-]*` name pattern). Tests must cover all five forms to confirm the grammar is unambiguous (none of them is interpreted as a named argument, and `--` is not interpreted as an end-of-options marker in any position); tests asserting diagnostic shape must distinguish the four usage-error forms (whose messages come from the CLI parser and are usage-style) from the target-validation form (whose message comes from SPEC §4.1 target validation at pre-iteration time, and is not required to be a usage-style diagnostic), and must not require a usage-style diagnostic for `loopx run name=value`. `name=value loopx run ralph` (shell env prefix) succeeds because the shell absorbs the prefix before loopx sees its arguments, leaving `ralph` as loopx's only run-level positional.

### Project-root cwd

**Equality conventions.** Tests in this section compare `LOOPX_PROJECT_ROOT` (the loopx-chosen spelling, fully under loopx's control) against runtime cwd APIs. The runtime APIs split into two regimes with different guarantees. Node/tsx/Bun's `process.cwd()` is kernel-sourced via `getcwd(3)` and is guaranteed to equal `LOOPX_PROJECT_ROOT` byte-for-byte when `LOOPX_PROJECT_ROOT` is itself a canonical, symlink-free absolute path — the common case for most test fixtures. Bash's `$PWD` and bare `pwd`, by contrast, are shell-maintained and inherit `PWD` from loopx's own environment (see §3 "`PWD` is not a protocol variable" and "Directory identity vs. string spelling"): per POSIX, Bash retains an inherited `PWD` that names the current working directory even when the spellings differ through symlinks, so an inherited `$PWD` from any ancestor shell whose logical cwd was a symlinked spelling of the same directory may leave the child with a `$PWD` that differs byte-for-byte from `LOOPX_PROJECT_ROOT` even under an otherwise canonical layout. **Byte-for-byte equality between Bash `$PWD` / bare `pwd` and `LOOPX_PROJECT_ROOT` is therefore not a loopx guarantee in any regime.** Bash tests that need to verify cwd must use `pwd -P` (or `/bin/pwd`), which bypasses `$PWD` and walks the kernel cwd via `getcwd(3)`, or filesystem identity via `stat` / device-inode comparison; they must not assert byte-for-byte equality between `$PWD` / bare `pwd` and `LOOPX_PROJECT_ROOT`. When a fixture deliberately involves a symlinked project root, Node/tsx/Bun's `process.cwd()` typically reports a canonicalized spelling while `LOOPX_PROJECT_ROOT` preserves the symlinked one; the two still name the same directory and tests in that regime must assert filesystem identity via `stat` device/inode comparison, or by comparing `fs.realpath()` results on both spellings, rather than byte-for-byte string equality. Bash `pwd -P` / `/bin/pwd` similarly reports the canonical spelling under a symlinked project root; tests use those, not `$PWD` / bare `pwd`, when a deterministic cwd assertion is required. Per §3 "No extra canonicalization", symlinked project roots are reachable only via a `RunOptions.cwd` whose resolved string contains symlinked components (programmatic API) — whether `RunOptions.cwd` was supplied as an absolute path or as a relative path with symlinked components (since `path.resolve` preserves symlinked spellings present in the input string without consulting the filesystem); a CLI invocation from a symlinked directory does not produce this `LOOPX_PROJECT_ROOT` asymmetry because loopx derives `LOOPX_PROJECT_ROOT` from its own `process.cwd()` without consulting `$PWD`, so on POSIX systems where `getcwd(3)` canonicalizes the CLI case yields a canonical `LOOPX_PROJECT_ROOT` directly — though Bash `$PWD` may still diverge in the child per the rule above, independently of how `LOOPX_PROJECT_ROOT` was produced. Tests must therefore not assume a symlinked CLI invocation cwd produces a symlink-preserving `LOOPX_PROJECT_ROOT`. Pure `path.resolve()` round-trips are not sufficient for the symlinked-project-root regime — `path.resolve` is a string-normalization operation that does not consult the filesystem, so it cannot prove that two spellings name the same directory. Tests that do not specify a symlink regime default to the symlink-free case.

- Under a canonical, symlink-free project root, `process.cwd()` inside a starting-target JS/TS script equals `LOOPX_PROJECT_ROOT` byte-for-byte. Under a symlinked project root, the two resolve to the same directory (same device/inode) but may differ in spelling.
- Bash `pwd -P` (or `/bin/pwd`) inside a starting-target script reports the kernel cwd via `getcwd(3)` and, under a canonical, symlink-free project root, equals `LOOPX_PROJECT_ROOT` byte-for-byte; under a symlinked project root the two name the same directory and tests assert filesystem identity rather than byte equality. Bash `$PWD` and bare `pwd`, by contrast, are not guaranteed to equal `LOOPX_PROJECT_ROOT` byte-for-byte in any regime (see §3 "Directory identity vs. string spelling" and "`PWD` is not a protocol variable"): `PWD` is inherited from loopx's own environment and Bash retains an inherited logical `PWD` that names the current directory through symlinks, so an ancestor shell with a symlinked logical cwd may leave the child with a `$PWD` that differs byte-for-byte from `LOOPX_PROJECT_ROOT` even under an otherwise canonical layout. Tests that compare a cwd string must therefore use `pwd -P` / `/bin/pwd` or filesystem identity, not `$PWD` / bare `pwd`. Scripts that need the loopx-chosen spelling read `LOOPX_PROJECT_ROOT`; scripts that need the canonical kernel-cwd spelling use `pwd -P`.
- The same effective working directory (same device/inode) is observed across the starting target, intra-workflow `goto` destinations, cross-workflow `goto` destinations, and loop reset to the starting target. Tests that check string-level cwd identity across spawns do so only under a canonical, symlink-free project root.
- A project-root-relative reference like `cat SPEC.md` or `readFileSync("adr/0001-adr-process.md")` resolves successfully from inside a script, with no `$LOOPX_PROJECT_ROOT` prefix required. This is a filesystem-identity test and does not depend on string spelling.
- A script that calls `cd /some/other/dir` (bash) or `process.chdir("/some/other/dir")` (JS/TS) does not affect the cwd of the next script in the run: the next spawn still starts at `LOOPX_PROJECT_ROOT` (same directory by filesystem identity).
- Programmatic, canonical `cwd`: `run("ralph", { cwd: projectDir })` where `projectDir` is an already-absolute, symlink-free path makes `process.cwd()` inside the script equal `projectDir` byte-for-byte and makes `LOOPX_PROJECT_ROOT` equal `projectDir` byte-for-byte.
- Programmatic, symlinked `cwd`: `run("ralph", { cwd: "/symlinked/proj" })` where `/symlinked/proj` is a symlink to `/real/proj` makes `LOOPX_PROJECT_ROOT === "/symlinked/proj"` (preserved spelling) and the child's `process.cwd()` equal to whatever the runtime returns (typically `/real/proj` after `getcwd()` canonicalizes); the two still refer to the same directory and `stat`/device-inode checks confirm identity.
- Programmatic: `run("ralph")` (no `cwd` option) uses `process.cwd()` at call time as both the project root and the script execution cwd. Mutating the caller's cwd after the call does not retroactively change the run's cwd (snapshot at call time, per existing §9.5 semantics).
- Script module resolution is independent of cwd: a workflow with its own `node_modules/loopx` resolves `import "loopx"` to the workflow-local version despite project-root cwd, because the resolver walks up from the script's file path. This assertion assumes the absolute discovery-time entry path is symlink-free in every component; when any component (workflow directory, entry script file, or intermediate ancestor) is a symlink, runtime main-module canonicalization may cause the resolver to walk up from the canonical target instead, and workflow-local visibility at the logical path is not guaranteed (see §3 module-resolution rationale). Mirror of T-INST-GLOBAL-01 that explicitly asserts cwd-vs-resolution independence.
- Bash: `$(dirname "$0")` computes the script's own workflow directory correctly under project-root cwd, and the computed value equals `LOOPX_WORKFLOW_DIR` byte-for-byte regardless of whether symlinks appear anywhere in the absolute entry path (the `.loopx/<workflow>` directory entry, the entry script file, or an intermediate component such as a symlinked project-root ancestor reached through `RunOptions.cwd`). This equality is loopx-controlled (loopx composes both values from the same components; bash does not canonicalize `$0`) and does not depend on whether the project root itself is canonical.
- JS/TS: `dirname(fileURLToPath(import.meta.url))` equals `LOOPX_WORKFLOW_DIR` only when the absolute discovery-time entry path is symlink-free in every component. When any component is a symlink — the `.loopx/<workflow>` directory entry, the entry script file, or any intermediate component — this equality is not required (the runtime's main-module symlink handling applies); tests that need a deterministic value across symlinked and non-symlinked layouts must read `LOOPX_WORKFLOW_DIR` directly rather than derive it from `import.meta.url`.
- Programmatic, relative `cwd`: `run("ralph", { cwd: "./sub" })` called from a process whose `process.cwd()` is a canonical, symlink-free `/home/user/proj` produces `LOOPX_PROJECT_ROOT === "/home/user/proj/sub"` byte-for-byte and `process.cwd()` inside the script equal to the same absolute path byte-for-byte. loopx applies `path.resolve(process.cwd(), options.cwd)` once at call time and no further `realpath`; a subsequent caller `process.chdir()` does not retroactively alter the resolved value. Passing an already-absolute, canonical `RunOptions.cwd` leaves the value unchanged byte-for-byte. Under a non-canonical or symlinked caller `process.cwd()`, `LOOPX_PROJECT_ROOT` preserves the resolved non-canonical spelling and the child's `process.cwd()` may canonicalize it — assert filesystem identity rather than byte equality in that regime.
- Programmatic, relative `cwd` with a symlinked component in the supplied path itself: `run("ralph", { cwd: "./link-to-proj" })` called from a canonical, symlink-free `/tmp` process where `/tmp/link-to-proj` is a symlink to `/real/proj` on disk produces `LOOPX_PROJECT_ROOT === "/tmp/link-to-proj"` byte-for-byte — the symlinked spelling of the supplied relative component is preserved because `path.resolve("/tmp", "./link-to-proj") === "/tmp/link-to-proj"` and loopx never applies `realpath` to the result. The child's runtime `process.cwd()` reports `/real/proj` (canonicalized by `getcwd(3)` on POSIX); both still name the same directory, and tests assert filesystem identity in this regime. This explicitly exercises the rule that symlink preservation in `LOOPX_PROJECT_ROOT` is not limited to symlinked absolute `RunOptions.cwd`; relative paths with symlinked components preserve the symlinked spelling just as absolute inputs do.

### `LOOPX_WORKFLOW_DIR`

- `LOOPX_WORKFLOW_DIR` is injected into every script's environment as an absolute path.
- The path exists and is a directory.
- For a top-level bash workflow script, `LOOPX_WORKFLOW_DIR` equals `$(dirname "$0")` regardless of whether symlinks appear anywhere in the absolute entry path (the `.loopx/<workflow>` directory entry, the entry script file, or any intermediate component such as a symlinked project-root ancestor).
- For a top-level JS/TS workflow script whose absolute discovery-time entry path is symlink-free in every component, `LOOPX_WORKFLOW_DIR` equals `dirname(fileURLToPath(import.meta.url))`. When any component of that entry path is a symlink — workflow directory, entry script file, or intermediate — this equality is not required (Node typically canonicalizes the main module's URL and Bun's behavior is not specified to match); portable tests must read `LOOPX_WORKFLOW_DIR` directly.
- For the `ralph` workflow under project root `/p`, `LOOPX_WORKFLOW_DIR` equals `/p/.loopx/ralph` in v1's layout.
- **Intra-workflow `goto`:** destination script observes the same `LOOPX_WORKFLOW_DIR` as the caller (same workflow).
- **Cross-workflow `goto` (A → B):** the B-side script observes `LOOPX_WORKFLOW_DIR` pointing at B's directory, not A's. A file written to `$LOOPX_WORKFLOW_DIR/foo` in A is not visible via `$LOOPX_WORKFLOW_DIR/foo` in B — the two paths are different directories.
- **Deep cross-workflow chains (A → B → C):** each spawn observes its own workflow's directory; B's spawn sees B's directory even though it was reached from A, and C's spawn sees C's directory.
- **Loop reset to starting target:** `LOOPX_WORKFLOW_DIR` restores the starting workflow's directory on the re-entry spawn.
- A helper file sourced from bash (`source "$LOOPX_WORKFLOW_DIR/lib/helpers.sh"`) or imported from JS/TS observes the same `LOOPX_WORKFLOW_DIR` as the top-level script that invoked it — the env var is inherited through the child process, not recomputed per source file.
- **Symlinked workflow directory:** a workflow reached via a symlink (`.loopx/foo` → `/elsewhere/foo`) has `LOOPX_WORKFLOW_DIR` equal to `<project>/.loopx/foo`, not `/elsewhere/foo` — the symlinked discovery-time path, matching bash `$(dirname "$0")` behavior. JS/TS `dirname(fileURLToPath(import.meta.url))` is not required to match `LOOPX_WORKFLOW_DIR` in this case (it typically reflects the realpath target under Node's default main-module handling); tests that observe JS/TS behavior under this layout must not assert equality.
- **Symlinked `.loopx` directory entry:** with `<project>/.loopx` a symlink to `/elsewhere/.loopx-real` and `ralph` a real directory inside `/elsewhere/.loopx-real`, `LOOPX_WORKFLOW_DIR` equals `<project>/.loopx/ralph` — the symlinked spelling of the `.loopx` segment is preserved because loopx treats `.loopx` as an intermediate component of the absolute entry path and does not apply `realpath`. Bash `$(dirname "$0")` equals `LOOPX_WORKFLOW_DIR` byte-for-byte. JS/TS `dirname(fileURLToPath(import.meta.url))` may canonicalize through the `.loopx` symlink per Node's default main-module handling; tests that observe JS/TS behavior under this layout must not assert equality.
- **Symlinked entry script file within a non-symlinked workflow directory:** with `.loopx/foo/` a real directory but the entry script file (e.g., `.loopx/foo/index.sh` → `/elsewhere/index.sh`, or `.loopx/foo/index.ts` → `/elsewhere/index.ts`) a symlink, `LOOPX_WORKFLOW_DIR` still equals `<project>/.loopx/foo` — derived from the workflow directory, not from the script's filesystem identity. For bash, `$(dirname "$0")` continues to equal `LOOPX_WORKFLOW_DIR` (bash does not canonicalize `$0`). For JS/TS, `dirname(fileURLToPath(import.meta.url))` is not required to match `LOOPX_WORKFLOW_DIR` (Node's default main-module handling canonicalizes symlinked entry scripts); tests that observe JS/TS behavior under this layout must not assert equality.
- **Symlink in an intermediate path component:** with the workflow directory and entry script file both non-symlinked but an ancestor of the project root reached through a symlink (e.g., `RunOptions.cwd = "/symlinked/proj"` whose realpath is `/real/proj`), `LOOPX_WORKFLOW_DIR` preserves the symlinked spelling `/symlinked/proj/.loopx/<workflow>` (per §3's "no extra canonicalization" rule). Bash `$(dirname "$0")` still equals `LOOPX_WORKFLOW_DIR` byte-for-byte. JS/TS `dirname(fileURLToPath(import.meta.url))` is not required to match — Node's main-module `realpath` canonicalizes the entry path, so the observed value may be `/real/proj/.loopx/<workflow>` instead.
- A user-supplied `LOOPX_WORKFLOW_DIR` in inherited env, the `-e` local env file, the global env file, or `RunOptions.env` is overridden by the injected protocol value.
