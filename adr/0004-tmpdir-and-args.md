# ADR-0004: Run-Scoped Temporary Directory, Named Run Arguments, and Pre-spawn Cancellation Semantics

**Status:** Proposed

---

## Decision Summary

Accepting this ADR decides, at minimum, the following. Each bullet is normative; the detailed rules appear under "Decision" below. This summary exists so that future readers do not underestimate what was frozen here.

- **`LOOPX_TMPDIR` (user-facing).** A per-`loopx run` temporary directory is created under `os.tmpdir()` with mode `0700`, injected into every script's environment, shared across the run, cleaned up on all terminal outcomes, and subject to a path-based best-effort cleanup with a documented threat model (not a sandbox).
- **Named run arguments (user-facing).** `loopx run [options] <target> [-- name=value...]` injects each `name=value` into every script's environment. Names must match `[A-Za-z_][A-Za-z0-9_]*`, `LOOPX_*` is reserved by the parser, values may contain any byte except NUL. Programmatic counterpart: `RunOptions.env`. Named arguments are intentionally a general run-scoped env overlay — any non-`LOOPX_*` name may be set, including `PATH`, `HOME`, `LANG`, etc.
- **Env precedence.** Five-tier ordering (highest → lowest): loopx-injected protocol vars, named run arguments, `-e` local env file, global loopx env file, inherited system env.
- **Programmatic `RunOptions` / `options` field validation.** `target` must be a string. `options` must be `undefined` or a non-null, non-array object. `env`, `maxIterations`, `envFile`, `cwd`, and `signal` are each field-validated at step 1; `env` is treated as record-like (own enumerable string-keyed entries; symbols / non-enumerable / inherited ignored).
- **Same-realm-only `AbortSignal`.** `RunOptions.signal` must be `instanceof AbortSignal` against the platform-global constructor. Cross-realm and duck-typed signals are rejected; cross-realm callers bridge through a same-realm `AbortController`.
- **Pre-spawn precedence.** Nine-step programmatic pre-iteration sequence (field validation → already-aborted check → discovery → env loading → target resolution → `maxIterations: 0` → version check → tmpdir creation → first spawn). In the absence of an abort, earliest-numbered step wins. A step-2 already-aborted check races cleanly with concurrent abort firing.
- **Abort dominance.** Once step 1 has succeeded with a valid signal, an abort observed at any time before terminal settlement wins over: pre-spawn outcomes from steps 3–7, in-flight step-8 tmpdir creation failures, pending non-zero script exits, pending invalid/missing-`goto` errors, consumer-initiated `generator.return()` / `break`, consumer-initiated `generator.throw(err)`, and even the post-final-yield / pre-`{ done: true }` window. Only step-1 field-validation errors and non-terminal cleanup warnings survive.
- **Tmpdir cleanup safety.** Device/inode identity captured at creation; recursive removal runs only if identity matches at cleanup time. Non-directory replacements at the path are unlinked as a single non-traversing entry; symlinks are unlinked but never followed. The guarantee covers quiescent script-introduced replacements within the same run; it is not a race-resistant sandbox against an actively racing same-user process.

A later ADR may tighten individual pieces (e.g., pinning the abort error's exact class/shape, adopting fd-relative cleanup syscalls, introducing a named-argument schema) without re-opening the decisions above.

---

## Context

As workflows in `.loopx/` have accumulated (`ralph`, `review-adr`, `apply-adr`, `spec-test-adr`), two recurring pain points have emerged:

1. **Passing intermediate data between scripts during a loop run.** `review-adr`, `apply-adr`, and `spec-test-adr` stash hand-off files like `.feedback.tmp`, `.claude-output.tmp`, `.answer.tmp`, and `.prompt.tmp` directly inside their workflow directories. Each script deletes files it consumes via `rm -f` along the happy path. There is no cleanup on script error, SIGINT/SIGTERM, or unexpected termination, so leaked files accumulate in the workflow directory — where they are indistinguishable from legitimate workflow content. The `ralph` workflow has a related issue: it maintains an iteration counter at `.loopx/.iteration.tmp` by hand, with cleanup only in the success branch. That counter is intra-run state — it carries no meaning across separate `loopx run` invocations — and has been leaking into the workflow directory because there is no per-run scratch space to hold it.

2. **Per-invocation parameterization.** Scripts have no mechanism to receive caller-supplied parameters that are scoped to a single `loopx run` invocation. `loopx env set` exists for persistent/global env vars, but it is the wrong tool for single-run parameterization: using it pollutes the user's global env with values that are only meaningful for one run, and removing them requires a matching `loopx env remove` call per name. Workflows that need to operate on a specific input (e.g., review ADR-0002 vs. ADR-0003) currently hardcode the input inside the script, forcing the user to edit the script or fork the workflow for each new input. The existing `review-adr`, `apply-adr`, and `spec-test-adr` workflows all hardcode `adr/0001-adr-process.md` and `adr/0002-run-subcommand.md` paths for exactly this reason. What is missing is a session-scoped counterpart to `loopx env set`: a way to pass named values on the `loopx run` command line that are injected into the child environment for that run only and leave no persistent state behind.

A generic session-state mechanism (e.g., a first-class iteration counter or a key-value store) was considered but intentionally deferred. Scripts that need their own state can maintain it inside the temporary directory introduced by this ADR — including an iteration counter, which is a trivial read/increment/write against a file in `$LOOPX_TMPDIR`. A dedicated state API may be added later if that pattern proves insufficient.

## Decision

Two independent mechanisms, both scoped to a single `loopx run` invocation.

**Scope note.** Although the title foregrounds the two user-facing additions (`LOOPX_TMPDIR` and named run arguments), accepting this ADR also accepts the surrounding pre-spawn validation and cancellation-precedence semantics of the programmatic API. Reviewers should read this ADR with that broader scope in mind: the lifecycle rules below are not incidental, they are what make the tmpdir cleanup contract and the named-argument validation timing observably well-defined, and without fixing them the rules stated below would be ambiguous about when validation runs, which outcome surfaces, and when cleanup is guaranteed. Concretely, the following are in scope and become normative when this ADR moves to Accepted: field-level `RunOptions` shape validation (the `options`-parameter shape, the `target` type, the `env` field's object/key/value rules, and the `signal`'s `instanceof AbortSignal` requirement); the already-aborted-signal check and listener-race contract; the broad `AbortSignal`-abort-wins-until-terminal-settlement rule; and abort dominance over consumer-driven `generator.return()`, `generator.throw()`, and `break`. These are decided here rather than in a separate ADR because they are tightly coupled to the tmpdir lifecycle; a later ADR may tighten individual pieces (for example, pinning the abort error's exact class and shape, or adopting fd-relative cleanup syscalls) without re-opening this ADR's core decisions.

### 1. `LOOPX_TMPDIR` — run-scoped temporary directory

For each `loopx run` invocation that proceeds to execution, loopx creates a unique temporary directory after pre-spawn validation and immediately before the first child process is spawned, and injects its absolute path into every script's environment as `LOOPX_TMPDIR`. The precise ordering of pre-spawn validation against tmpdir creation is specified under "Scope and lifecycle" below; pre-spawn outcomes that occur before the tmpdir-creation step do not create a tmpdir.

#### Location and naming

The directory is created under the OS temporary directory (Node's `os.tmpdir()`). The directory name begins with `loopx-` and contains a random component sufficient to guarantee uniqueness across concurrent runs on the same host (e.g., via `mkdtemp`). The full name format beyond the `loopx-` prefix is an implementation detail and must not be relied upon by scripts.

#### Mode

The directory is created with mode `0700` (owner read/write/execute only).

#### Scope and lifecycle

- **Created:** once per `loopx run` invocation, after the starting workflow's version check and immediately before the first iteration's child process is spawned. The pre-iteration sequence (extending SPEC §7.1) differs slightly between the CLI and the programmatic API, because the programmatic API exposes a `RunOptions` object whose fields must be validated before any runtime work begins.

  **CLI (`loopx run`) pre-iteration sequence:**
  1. Run-subcommand argument parsing (section 4.1), including `-n` / `-e` value validation, flag recognition, the `-h` / `--help` short-circuit, and the named-argument grammar check (section 2). Parser errors (unknown flags, missing flag values, duplicate flags, non-integer `-n`, malformed `name=value` tokens after `--`, invalid names, reserved `LOOPX_*` names, duplicate names, NUL-byte values, etc.) exit with code `1` before any runtime work begins. The `-h` / `--help` short-circuit exits the **run-execution parser path** before target consumption, flag validation, env loading for execution, tmpdir creation, or child spawning; it does **not** skip the run-help command's existing behavior described in SPEC §11.2. When `-h` / `--help` triggers run help, loopx hands control to the run-help command path, which may still perform its non-fatal discovery and validation to display available workflows/scripts and any discovery diagnostics before printing help and exiting `0`. The run-help command path does not create `LOOPX_TMPDIR`, does not load env files for execution, and does not spawn any child process — it performs only the non-fatal discovery behavior currently specified in SPEC §11.2.
  2. Discovery and name/collision validation.
  3. Environment variable loading (global + `-e` local).
  4. Starting-target resolution.
  5. `-n 0` early exit (if requested).
  6. Starting workflow version check (non-fatal per SPEC §3.2 — a mismatch emits a stderr warning and continues).
  7. **Create and secure `LOOPX_TMPDIR`** (this ADR).
  8. Spawn the first child process.

  **Programmatic (`run()` / `runPromise()`) pre-iteration sequence:**
  1. Field-level validation of the call-site inputs. The `target` parameter must be a string — non-string values (`undefined`, `null`, numbers, booleans, objects, etc.) are rejected here, independent of what the value would otherwise resolve to. The `options` parameter, if present, must be `undefined`, or a non-null, non-array object. Non-object values (`null`, strings, numbers, booleans, functions, etc.) and arrays are rejected at step 1 as invalid-options-shape errors before any `RunOptions` field is examined; an explicit `undefined` is treated identically to omitting the parameter. The `RunOptions` fields are then type-checked: `env` must be `undefined` or a non-null, non-array object (its prototype is not constrained — `env` is treated as a record of own enumerable string-keyed properties, not as a strictly plain object); loopx snapshots `env`'s own enumerable string-keyed entries via `Object.keys(env)` and requires each such entry to have a name matching `[A-Za-z_][A-Za-z0-9_]*`, a name that does **not** begin with `LOOPX_` (the `LOOPX_*` namespace is reserved for loopx protocol variables — see section 2), and a string value that does not contain a NUL byte (`\0`) — POSIX env strings cannot carry NUL, so a NUL-containing value is a step-1 field-validation error rather than a deferred child-spawn failure. Symbol-keyed, non-enumerable, and inherited properties of `env` are **ignored**: they are neither iterated by `Object.keys`, nor validated, nor forwarded to child processes. In particular, a symbol-keyed property is not a non-string-key validation error, it is simply not in the snapshot; the same applies to inherited and non-enumerable properties. `maxIterations` must be a non-negative integer or `undefined`; `envFile` and `cwd` must be strings or `undefined`; `signal` must be an `AbortSignal` or `undefined`. For validation purposes, "an `AbortSignal`" means `instanceof AbortSignal` against the platform global constructor of the runtime that called `run()` / `runPromise()` (Node.js ≥ 20.6 and Bun ≥ 1.0 both provide a global `AbortSignal`). Duck-typed or cross-realm objects that expose an `aborted` property and `addEventListener` but are not `instanceof AbortSignal` are rejected at step 1. **This is intentional.** loopx's signal handling assumes the standard event-target semantics that the platform-global constructor guarantees, and accepting duck-typed or cross-realm objects would expand the contract to objects whose `addEventListener` / `removeEventListener` / `.aborted` semantics loopx cannot rely on. Callers in cross-realm contexts (VM contexts, worker realms with separate globals, polyfilled environments) that need to forward an abort to loopx should bridge the cross-realm signal through a same-realm `AbortController` and pass that controller's `signal` to `RunOptions.signal`. Filesystem-level and semantic checks — whether `envFile` exists, whether `cwd` contains a valid `.loopx/`, whether the `target` *string* matches the workflow/script name pattern or resolves to an existing target — are deferred to later steps and are **not** step-1 concerns. **Order within step 1:** the `target` parameter is validated before the `options` parameter shape; the `options` parameter shape is validated before any individual `RunOptions` field; and the internal order among `RunOptions` field type-checks is implementation-defined. The externally-guaranteed contract is: a non-string `target` surfaces in preference to an invalid-options-shape error and to any invalid `RunOptions` field; an invalid-options-shape error surfaces in preference to any individual `RunOptions` field error; and any step-1 field-validation error surfaces in preference to all downstream steps (2 through 9 below).
  2. Already-aborted `AbortSignal` check and listener installation. If step 1 succeeded and `RunOptions.signal` was provided (and is therefore a valid `AbortSignal` per step 1), loopx inspects its `.aborted` property. If `.aborted` is already `true`, the run is aborted here: the generator throws an abort error (for `run()`) or the promise rejects with an abort error (for `runPromise()`), and no further pre-iteration work runs. This step is reached only after step 1 has succeeded — a field-validation error from step 1 wins over an already-aborted signal. An already-aborted signal at step 2 wins over every later step (discovery, env loading, target resolution, `maxIterations: 0`, version check, tmpdir creation, first spawn). **Implementation contract:** step 2 must install abort observation in a way that cannot lose an abort occurring between the `.aborted` inspection and listener registration. An implementation may register the listener before inspecting `.aborted`, re-inspect `.aborted` after registering the listener, or use any other ordering that closes the window. The broad abort-dominance guarantee described below depends on this: an abort that fires at step 2 — whether observed via the `.aborted` inspection or via the just-installed listener — must surface in preference to every later pre-spawn step.
  3. Discovery and name/collision validation.
  4. Environment variable loading (global + `envFile` local).
  5. Starting-target resolution: parse the `target` string's syntax and verify that the referenced workflow and script exist in the cached discovery results. (Non-string `target` values are already rejected at step 1; this step handles only the string-syntax and existence checks.)
  6. `maxIterations: 0` early exit (if requested).
  7. Starting workflow version check (non-fatal per SPEC §3.2).
  8. **Create and secure `LOOPX_TMPDIR`** (this ADR).
  9. Spawn the first child process.

  **Precedence among pre-spawn outcomes is the order shown above, with one exception: once a valid signal is registered, any abort observed before the first child spawns wins over later pre-spawn outcomes.** When multiple pre-spawn outcomes are present and no signal has aborted, the earliest-numbered step wins. Under the programmatic API, this means field validation (step 1 — non-string `target`, invalid options shape, invalid `RunOptions` field types) runs **before** the already-aborted signal check, discovery, env loading, target syntax/existence resolution, and `maxIterations: 0`. A non-string `target`, invalid options shape, or an invalid `env` / `maxIterations` / `envFile` / `cwd` / `signal` therefore surfaces in preference to an already-aborted signal, a missing `.loopx/`, a missing env file, a missing workflow, a missing script, or the `maxIterations: 0` early-exit branch.

  **Abort-dominance during pre-spawn (steps 2 through 7).** Once step 1 has succeeded with a valid `RunOptions.signal`, an aborted signal — whether already-aborted at step 2 or aborted at any point during steps 3 through 7 (discovery, env loading, target resolution, `maxIterations: 0`, version check) — wins over those pre-spawn outcomes and prevents any later step (tmpdir creation, first spawn) from running, but does **not** win over step 1. Concretely, if `RunOptions.signal` aborts after step 2's check passed but before step 8 (tmpdir creation) begins — for example, while discovery is still running, while an env file is being loaded, while target resolution is in progress, or while the non-fatal version check is in progress — loopx surfaces the abort error in preference to whichever pre-spawn condition would otherwise have surfaced from steps 3 through 7. This is consistent with the broader rule (documented under the programmatic API section below) that an `AbortSignal` abort wins until the generator has terminally settled. Aborts that fire during steps 3 through 7 require no cleanup, since step 8 has not yet begun and no tmpdir has been created. Aborts that fire during step 8 itself (tmpdir creation/securing in progress) are covered by a separate rule documented under "Abort or signal during tmpdir creation" below.

  **Other ordering rules (in the absence of an abort).** Env-file loading (step 4) runs before target syntax/existence resolution (step 5), so a missing or malformed env file surfaces in preference to a missing workflow or missing script. Under the CLI, argument parsing (step 1) runs before any of the runtime pre-iteration steps, so `-n` non-integer values, unknown flags, missing flag values, duplicate flags, and named-argument grammar errors surface in preference to project-level errors. The version check (step 6 in the CLI sequence and step 7 in the programmatic sequence) is non-fatal and never itself produces a pre-spawn failure: it produces at most a stderr warning and continues to tmpdir creation. There is no "version error" pre-spawn outcome — a version mismatch does not suppress tmpdir creation or change the run's terminal outcome on its own.

  **CLI parser-error precedence.** When multiple CLI parser usage errors could apply to the same `loopx run` invocation (for example, a non-integer `-n` value together with a missing target, or a missing `-n` value together with a missing target, or a malformed named-argument token together with a duplicate `-n`), Phase 2 reports the first parser error encountered as the scan proceeds left-to-right, with the exception that validation of an already-consumed flag value (e.g., the non-negative-integer check on `-n`'s value) may be reported before the missing-target check even when the flag value was consumed earlier in the scan. Diagnostic precedence among simultaneous parser usage errors beyond this rule is implementation-defined; tests may assert on the *kind* of usage error surfaced but should not over-constrain the exact diagnostic when multiple usage errors are present at the same left-to-right position.

  Pre-spawn outcomes that occur before the tmpdir-creation step — CLI parser errors (including named-argument grammar errors), programmatic field-validation errors (including non-string `target`, invalid options shape, and invalid `RunOptions` field types), already-aborted-signal aborts at step 2, aborts that fire during steps 3 through 7 of the programmatic sequence, discovery errors, env-file loading failures, starting-target resolution failures, and `-n 0` / `maxIterations: 0` early exits — do not create a `LOOPX_TMPDIR`, and no cleanup is needed for these cases. The `-n 0` / `maxIterations: 0` cases are early exits, not failures: when the options are valid, the loop completes normally with no outputs (CLI exit code `0`; `runPromise()` resolves with `[]`; `run()` completes with `{ done: true }` and no prior yields). The version check is non-fatal and therefore does not itself suppress tmpdir creation.
- **Shared:** a single directory serves the entire run. All scripts — the starting target, scripts reached via intra-workflow `goto`, scripts reached via cross-workflow `goto`, and re-executions of the starting target on loop reset — observe the same `LOOPX_TMPDIR` value.
- **Persisted within the run:** the directory is not cleared between iterations. Files written by one script remain visible to subsequent scripts for the remainder of the run.
- **Cleanup at loop end.** loopx guarantees cleanup of the original tmpdir *when it remains at the original path*, and guarantees that cleanup will not recurse into a symlink or unrelated replacement at that path. A script that removes or renames the tmpdir directory itself can defeat automatic cleanup of the moved directory; loopx does not chase renamed tmpdirs. This is a path-based best-effort cleanup, not a sandbox: the identity checks documented under "`LOOPX_TMPDIR` path replacement between creation and cleanup" below are what make cleanup safe, and the same checks are what leave a moved directory uncleaned. Subject to that caveat, loopx runs cleanup on the tmpdir in all of the following cases:
  - Normal completion via `stop: true`.
  - Normal completion via `-n` / `maxIterations` reached.
  - Error exit: non-zero script exit, invalid `goto` target, missing workflow or script in a `goto` resolution.
  - SIGINT / SIGTERM: if a child process group is active, cleanup runs after the process group has exited (per the grace period in SPEC §7.3, including `SIGKILL` escalation to the process group if required) and before loopx itself exits with the signal's exit code. If no child process is active when the signal arrives — including (a) the window after `LOOPX_TMPDIR` has been created but before the first child has spawned, and (b) the window between a child exiting and the next child spawning (for example, between a script exit and the next `goto`-resolved child starting) — cleanup runs immediately, and loopx then exits with the signal's exit code. Pre-tmpdir-creation arrivals — including signals that arrive before step 8 begins — are covered by SPEC §7.3's "between iterations" rule and require no cleanup. Signals that arrive during step 8 itself (tmpdir creation/securing in progress) are covered by the "Abort or signal during tmpdir creation" rule below: loopx waits for the in-flight operation to settle, performs best-effort cleanup of any partial directory, and then exits with the signal's exit code.
  - Programmatic `AbortSignal` abort: if a child process group is active, loopx first terminates it using the cancellation behavior defined in SPEC §9.1; cleanup runs only after the child process group has exited or been killed, and before the generator throws or the promise rejects. If the signal is aborted while no child process is active — including (a) while the generator is suspended between `next()` calls, (a') the window after one child has exited and before the next child has spawned within a single ongoing `next()` call (for example, during `goto` target validation/resolution, during first-entry version checking for a target workflow reached via cross-workflow `goto`, or after iteration N's output has been prepared but before iteration N+1's child has been spawned), and (b) during the window after `LOOPX_TMPDIR` has been created but before the first child has spawned — loopx's signal listener performs cleanup immediately when the abort event fires; the next generator interaction (or the in-flight first `next()` call, in the post-tmpdir / pre-spawn case) settles with the abort error as soon as cleanup completes, and the `runPromise()` promise (if outstanding) rejects with the abort error as soon as cleanup completes. The caller is not required to resume or explicitly close the generator to observe cleanup in this case. (c) Aborts that fire during steps 3 through 7 of the programmatic pre-iteration sequence — after step 2's already-aborted check has passed but before step 8 (tmpdir creation) begins — require no cleanup, since no tmpdir has been created; the run aborts with the abort error per the abort-dominance rule documented in the precedence section above. (d) Aborts that fire during step 8 itself — while tmpdir creation or securing is in progress — are covered by a separate rule documented under "Abort or signal during tmpdir creation" below: loopx waits for the in-flight operation to settle enough to determine whether a directory was created, performs best-effort cleanup if one was created, and otherwise surfaces the abort without cleanup.
  - Programmatic consumer-driven cancellation (`break` out of a `for await`, explicit `generator.return()`, or explicit `generator.throw(err)`): if a child process group is active, loopx first terminates it using the cancellation behavior defined in SPEC §9.1; cleanup runs only after the child process group has exited or been killed, and before the generator settles. In the absence of an `AbortSignal` abort observed before terminal settlement, `break` and `generator.return()` settle the generator with `{ done: true }`, and `generator.throw(err)` settles by throwing `err` after cleanup. If an `AbortSignal` abort is observed before the generator has settled terminally — even after the consumer has initiated `break` / `.return()` / `.throw(err)` and even while loopx is in the middle of cleanup for the consumer-initiated cancellation — the broad abort-wins rule documented under the programmatic API section below applies: the generator settles by throwing the abort error rather than completing normally or throwing the consumer's `err`.
- **Cleanup failure:** if the cleanup operation fails (e.g., `EBUSY`, `EACCES`, or an error during the single-entry `unlink` that cleanup performs when a symlink or other non-directory replacement is found at the original path), loopx prints a single warning to stderr and proceeds. Cleanup is not always recursive: in several cases it is a no-op (path no longer exists), a single non-traversing `unlink` (symlink or non-directory at the path), or an identity-gated recursive removal (path resolves to the directory whose identity was recorded at creation). The "cleanup failure" rule covers a failure of whichever of these operations cleanup attempted for a given run. The loop's exit code is not changed by a cleanup failure. Under the programmatic API, cleanup failure does not change the generator or promise outcome; the original success, error, or abort result is preserved.
- **Symlinks inside `LOOPX_TMPDIR`:** recursive removal removes symlink entries found inside `LOOPX_TMPDIR` — the symlink itself is unlinked — but does **not** traverse or follow symlinks out of `LOOPX_TMPDIR` to delete their targets. A script that creates a symlink inside its tmpdir pointing at a file or directory outside the tmpdir will observe the symlink removed at cleanup; the target is left untouched. This preserves the invariant that cleanup affects only the run's scratch space and does not collateral-delete workflow state such as `.loopx/`, `node_modules/`, or any other location on the filesystem reachable only through a symlink.
- **`LOOPX_TMPDIR` path replacement between creation and cleanup:** if the `LOOPX_TMPDIR` path has been replaced between creation and cleanup (for example, a script removes the tmpdir and creates a symlink of the same name pointing elsewhere, removes the tmpdir and creates an unrelated directory at the original path, removes the tmpdir and creates a regular file at the original path, or renames the tmpdir and creates a symlink at the original path), cleanup is designed not to traverse into or recursively delete any directory, symlink target, or other location that loopx did not originally create as this run's scratch space. A non-directory replacement at the original path (a symlink, regular file, FIFO, or socket) may be unlinked as a single non-traversing entry — the basename was unique to this run, so unlinking a single non-directory entry at the original path does not affect any other location on the filesystem — but loopx never follows it or recursively deletes anything behind it. A directory at the original path whose device/inode identity does not match the recorded identity is not recursively removed; loopx prints a single cleanup warning and leaves that directory intact. The per-case dispatch below makes these high-level guarantees precise.

  **Threat model.** This guarantee covers script-introduced replacements that are *quiescent* by the time cleanup begins — symlinks, substituted directories, or substituted files left behind after the child process group has exited and before cleanup starts. It is **not** a sandbox boundary against an actively racing same-user process that mutates the tmpdir path *during* cleanup itself. The mechanism described below performs `lstat` on the path and then acts on the result; between those operations, an actively racing same-user process could in principle substitute the entry, and loopx does not defend against that case. The intended scope is misbehaving or careless scripts within the same run, not unrelated concurrent workloads on the host. A stronger race-resistant guarantee (e.g., fd-relative traversal via `openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`) is out of scope for v1 and would require a separate ADR if pursued later.

  **Observable behavior.** Within the threat model above, the only directory contents loopx recursively removes are the contents of the directory whose device/inode identity matches the original tmpdir; any other entry at the tmpdir path is at most unlinked as a single non-traversing entry. Because loopx v1 is POSIX-only (SPEC §1 Platform support), the device/inode pair is always available via the platform's stat-family syscalls, and loopx captures it during step 8 — after the directory has been created and before the directory is exposed to any child process as `LOOPX_TMPDIR` (the specific syscall used to capture it — `lstat`, `stat`, or equivalent — is an implementation detail; what is normative is that the pair is captured and stored before any child can see the directory). At cleanup time, loopx re-inspects the `LOOPX_TMPDIR` path (via `lstat` or equivalent) and dispatches on the result:

  1. If the path no longer exists (`ENOENT` / `ENOTDIR`), cleanup is a no-op.
  2. If the path is a symlink, loopx unlinks the symlink entry and does **not** traverse it. The symlink's target is untouched, regardless of whether that target is a file, a directory, or a dangling path.
  3. If the path is a non-directory, non-symlink entry (regular file, FIFO, socket, etc.), loopx unlinks that entry. This is safe within the threat model above because the `LOOPX_TMPDIR` path basename is unique to this run, and `unlink` on a non-directory entry does not traverse into any other location on the filesystem.
  4. If the path is a directory and its recorded device/inode identity matches the directory observed now, loopx recursively removes it, applying the "symlinks inside `LOOPX_TMPDIR`" rule above: symlink entries found during the walk are unlinked but never traversed.
  5. If the path is a directory but its device/inode identity does not match the recorded identity (e.g., the original tmpdir was removed and an unrelated directory was created in its place), loopx does **not** recursively remove it. Loopx prints a single cleanup warning to stderr identifying the path and the mismatch, and proceeds. The unrelated directory is left intact.

  The recursive-removal branch (case 4) runs only when the re-inspection (via `lstat` or equivalent) confirms the path resolves to the directory whose identity was recorded at creation. Symlink-safety applies equally to entries inside the tmpdir and to the tmpdir path itself: within the threat model above, cleanup does not walk into a symlink's target, and does not recurse into a directory other than the one whose identity was recorded at creation, regardless of whether the replacement was introduced inside the tmpdir or in place of the tmpdir. Workflow state such as `.loopx/`, `node_modules/`, or any location reachable only through a symlink or a quiescent post-creation substitution at the tmpdir path is not touched by cleanup. The device/inode comparison is normative for the contract; the choice to use `lstat` (rather than fd-relative dispatch) is an implementation detail that matches the v1 threat model and may be tightened in a future ADR without re-opening the contract.
- **SIGKILL to loopx itself, and host crash:** no cleanup is performed. Leaked directories in `os.tmpdir()` are expected to be reaped by OS temp-cleaning policy (e.g., `systemd-tmpfiles`, tmpfs reboot). loopx does not attempt to reap stale tmpdirs at startup. This case is distinct from loopx sending `SIGKILL` to a child process group after the `SIGTERM` grace period (SPEC §7.3): in that case, loopx itself is still alive and performs cleanup after the process group exits.

#### Creation failure

Step 8 (tmpdir creation and securing) has three observable preconditions, **all** of which must hold before the directory is exposed to any child as `LOOPX_TMPDIR`:

1. A directory has been created under `os.tmpdir()` with the `loopx-` prefix (as by `mkdtemp` or an equivalent uniqueness-guaranteeing mechanism).
2. Its device/inode identity has been captured and recorded (so cleanup can later verify that the path still resolves to the same inode).
3. Its final mode is `0700`.

The order in which loopx performs (2) and (3) relative to each other is not normative, and the specific mechanisms — `mkdtemp`, `chmod`, `lstat`, or equivalents — are implementation details. The spec does not require any particular syscall sequence; an implementation may create the directory with mode `0700` directly (as POSIX `mkdtemp` does), call `chmod` afterward, verify the mode set by creation, or use an equivalent mechanism. What is normative is that all three preconditions hold before the directory is exposed.

If loopx fails to meet any of these preconditions (e.g., the underlying create call fails with `EACCES`, `ENOSPC`, or `EMFILE`; the identity-capture call fails; or the mode-securing or mode-verification step fails), **no child process is spawned**. Under the CLI, loopx prints an error to stderr and exits with code `1`. Under `run()`, the generator throws on the first iteration; under `runPromise()`, the promise rejects. If a directory was partially created before the failure, loopx attempts best-effort cleanup of the partial directory subject to the same identity-based safety rules as normal cleanup (see "Abort or signal during tmpdir creation" below for the identity-recording gap); if that best-effort cleanup itself fails, loopx prints a single additional warning to stderr but the original creation-failure error is the one surfaced — cleanup-of-partial-directory failure does not mask the tmpdir-creation error and does not change the exit code, thrown error, or promise rejection reason.

Because tmpdir creation is a required runtime facility for all iterations, a creation failure prevents loopx from faithfully executing any accepted spec behavior that depends on `LOOPX_TMPDIR` being present in the script environment. Scripts must not be spawned in a state where that invariant is violated.

Pre-spawn outcomes that precede tmpdir creation — non-string `target`, invalid `options` shape (non-null, non-array object required), invalid `RunOptions.env` (value is not a non-null, non-array object; or a snapshotted own-enumerable string-keyed entry has a name that fails the pattern, begins with `LOOPX_`, has a non-string value, or has a NUL-byte-containing value), invalid `RunOptions.maxIterations`, invalid `RunOptions.envFile`, invalid `RunOptions.cwd`, invalid `RunOptions.signal`, already-aborted `AbortSignal` at step 2, aborts that fire during steps 3 through 7 (discovery, env loading, target resolution, `maxIterations: 0`, version check), discovery errors, env-file loading failures, starting-target resolution failures (invalid target syntax, missing workflow, missing script), and `-n 0` / `maxIterations: 0` early exits — do **not** create a tmpdir, and do not reach the tmpdir-creation step. This is consistent with the "no tmpdir for pre-spawn outcomes" rule stated under Scope and lifecycle above.

#### Abort or signal during tmpdir creation

Step 8 is not atomic: as documented under "Creation failure" above, it requires loopx to create the directory, capture its device/inode identity, and ensure its final mode is `0700`, and a SIGINT/SIGTERM signal or an `AbortSignal` abort may be observed at any point during that sequence. The following rule covers the window in which step 8 has begun but has not yet reached a terminal outcome.

If SIGINT or SIGTERM is observed, or `AbortSignal` aborts, while step 8 is in progress, loopx does **not** spawn a child. It waits for the in-flight operation to settle enough to determine two observable preconditions: **(1)** whether a directory was created on-disk, and **(2)** whether that directory's device/inode identity was recorded. The cleanup behavior depends on those two preconditions:

- **A directory was created and its identity was recorded.** loopx performs best-effort cleanup of the directory before surfacing the signal/abort outcome. Best-effort cleanup applies the same device/inode-identity comparison and symlink-safety rules documented under "`LOOPX_TMPDIR` path replacement between creation and cleanup" above: recursive removal proceeds only if the directory observed at cleanup time still has the recorded identity (otherwise loopx prints the single cleanup warning and leaves the replacement directory intact), and symlinks are not traversed.
- **A directory was created but its identity was not recorded** (the signal/abort arrived after the directory was created on-disk but before identity capture had completed). loopx does **not** recursively remove the path. Without a recorded identity, loopx cannot distinguish the directory it just created from one substituted by a racing process at the same path, so the safe action is to leave any directory at the path intact. Cleanup may still perform a non-traversing action: loopx re-inspects the path (via `lstat` or equivalent), and if the path is a symlink or non-directory entry (regular file, FIFO, socket) loopx may unlink that single entry — the basename was uniquely chosen for this run, so unlinking a non-directory entry at the path cannot affect any other location on the filesystem. If the path resolves to a directory, loopx prints a single cleanup warning to stderr identifying the path and proceeds; the directory is left in place. This is intentional: identity capture is what gates recursive removal, and if capture did not complete, recursive removal cannot be performed safely. Whether mode `0700` was achieved is irrelevant to this case — the child was never exposed to the directory, so the 0700 contract was not violated, and the cleanup decision turns solely on whether identity was recorded.
- **No directory was created** (the underlying create operation itself failed before producing anything on-disk). No cleanup is needed.

If best-effort cleanup itself fails in any of these cases, loopx prints a single additional stderr warning; that warning does not mask or change the surfaced signal/abort outcome.

In all of the above cases the terminal outcome is the signal/abort outcome: under SIGINT/SIGTERM, loopx exits with the signal's exit code after cleanup (if any) completes; under `AbortSignal` abort, the generator throws the abort error (for `run()`) or the promise rejects with the abort error (for `runPromise()`) after cleanup completes.

If a signal or abort is observed concurrently with a tmpdir-creation failure (e.g., the underlying create call is about to reject with `ENOSPC` when the signal arrives), the signal/abort wins over the creation failure. This is consistent with the broad rule that an `AbortSignal` abort wins until the generator has terminally settled, and with SPEC §7.3's rule that SIGINT/SIGTERM dominates the run's terminal exit code. The creation failure is not surfaced as the terminal outcome in that case; any partial directory is still cleaned up per the rules above.

In the absence of a signal or abort, a tmpdir-creation failure surfaces as the creation-failure outcome defined under "Creation failure" above.

#### Concurrent runs

Each `loopx run` invocation receives its own distinct `LOOPX_TMPDIR`. Two concurrent runs — including two runs of the same workflow — do not share temporary state.

#### Env precedence

`LOOPX_TMPDIR` is a loopx-injected variable and participates in the injection precedence defined in SPEC §8.3: it overrides any value of the same name in named run arguments (section 2), the inherited system environment, the global loopx env file, or a local env file (`-e`). The full precedence ordering after this ADR is documented in section 2 under "Env precedence."

#### `-n 0` behavior

With `-n 0`, no iterations run and no child process is spawned. loopx does not create a `LOOPX_TMPDIR` under `-n 0`, and does not inject the variable into any environment. This is consistent with `-n 0` skipping workflow-level version checks (SPEC §3.2): no runtime work is performed beyond target validation and env loading.

#### Programmatic API

Tmpdir creation is lazy under both `run()` and `runPromise()`. Neither API creates a tmpdir at the call site of `run()` or `runPromise()`. The precise timing differs between the two, because each drives the loop differently:

- **`run()`** returns an `AsyncGenerator<Output>` that the caller drives. The programmatic pre-iteration sequence documented above (field validation through version check) is executed on the **first `next()` call**, and if it succeeds, `LOOPX_TMPDIR` is created immediately before the first iteration's child process is spawned, still within that first `next()`. If a generator returned by `run()` is never iterated (no `next()`, no `.return()`, no `.throw()`), none of the pre-iteration sequence runs and no tmpdir is created. An `AbortSignal` that was aborted before the first `next()` is surfaced via the step-2 already-aborted check documented above: field validation (step 1) still runs first, and if it passes, the already-aborted signal produces the abort error on the first `next()` call before any later pre-spawn step (discovery, env loading, target resolution, `maxIterations: 0`, version check, tmpdir creation, spawn) runs. If field validation fails, the field-validation error surfaces in preference to the already-aborted signal, because step 1 precedes step 2.
- **`runPromise()`** returns a `Promise<Output[]>` and internally drives the loop after returning. The same pre-iteration sequence is executed asynchronously on its internal generator. No tmpdir is created at the call site of `runPromise()`, but one **may** be created asynchronously once pre-spawn validation has succeeded and the first child is about to spawn. If pre-spawn validation fails (any step preceding tmpdir creation), or if the `AbortSignal` is already aborted when the step-2 check runs, no tmpdir is created and the promise rejects without any cleanup step being necessary.

In both cases, "lazy" means the tmpdir is created only when pre-spawn validation has fully succeeded and a child is about to be spawned; it is never created speculatively. Once created, the tmpdir is cleaned up when the generator completes (normally, via error, or via consumer-driven cancellation) or when the promise settles.

**Generator cleanup is guaranteed when the generator is driven to completion, throws, is explicitly closed, or is aborted via `AbortSignal`.** For `run()`, this includes: normal completion of a `for await` loop, `break` out of a `for await` (which closes the generator via `.return()`), explicit `generator.return()`, explicit `generator.throw(err)` (treated as consumer-driven cancellation — loopx terminates any active child, runs cleanup, then settles the generator by throwing `err`), an error raised during iteration, and `AbortSignal` abort. For `AbortSignal` abort specifically, cleanup runs eagerly from loopx's signal listener even when the generator is suspended between `next()` calls, so abort cleanup does not depend on the consumer resuming the generator; loopx surfaces the abort as a throw from the generator on its next interaction (or rejects the outstanding `runPromise()` promise immediately upon cleanup completing). In each of these cases, cleanup runs before the generator settles.

**`AbortSignal` abort wins until the generator has settled terminally.** This is an intentional rule, chosen so that there is a single predicate that determines the outcome of a `run()` or `runPromise()` — "is the generator terminally settled?" — rather than a side-table of per-state exceptions.

**The rule is intentionally broad.** An abort observed before terminal settlement wins over **any** other not-yet-settled outcome, including:

- pending normal completion (the final-yield window discussed below);
- a pending non-zero script exit, invalid `goto` target, or missing-`goto`-workflow / missing-`goto`-script error that has been detected by loopx but has not yet been surfaced to the generator;
- a pending pre-spawn outcome during steps 3 through 7 of the programmatic pre-iteration sequence (discovery, env loading, target resolution, `maxIterations: 0`, version check — the version check being non-fatal, a warning rather than an error), per the abort-dominance rule documented in the precedence section above;
- an in-flight tmpdir creation or securing operation at step 8 that would otherwise have surfaced a tmpdir-creation failure — the abort wins per "Abort or signal during tmpdir creation" above, and any partial directory is cleaned up before the abort is surfaced, subject to the identity-recording gap rule (recursive cleanup requires a recorded identity; a partial directory whose identity was not recorded is left in place with only non-traversing cleanup applied);
- an in-flight `generator.return()` triggered by the consumer (`break` from `for await`, explicit `.return()` call) — abort wins over the consumer-initiated normal completion, so the post-cleanup settlement is the abort error rather than `{ done: true }`;
- an in-flight `generator.throw(err)` from the consumer — abort wins over the consumer-supplied `err`, so the post-cleanup settlement is the abort error rather than `err`;
- an in-flight cleanup arising from any of the above consumer-driven cancellations.

The only outcomes that survive an abort observed before terminal settlement are:

- step-1 field-validation errors (which precede signal registration entirely, and so are surfaced before the signal is ever inspected); and
- cleanup-failure warnings (which are non-terminal and never change the terminal outcome reason regardless of which terminal outcome reaches the consumer).

**Abort error shape is implementation-defined.** The exact class, name, message, code, and cause of the abort error surfaced by loopx (whether thrown from the generator or used to reject the promise) are not fixed by this ADR. Tests should assert that the abort outcome wins over competing outcomes — i.e., that an abort error (as opposed to normal completion, a script-failure error, a tmpdir-creation failure, a consumer-supplied `err`, etc.) is what reaches the consumer — but should not over-constrain the error's observable shape. An implementation may choose to surface a standard `DOMException` with name `"AbortError"`, an `Error` subclass with a specific `code`, the `RunOptions.signal.reason` value verbatim, or any other form; a future ADR may tighten this to a stable public API. The same principle applies to the cleanup-failure stderr warning's exact wording: the existence of the warning (and its non-terminal nature) is normative, but its exact text is not.

The narrow final-yield window is one specific application of this broad rule. The rule continues to apply in the window after the final iteration's output has been yielded (`{ value: Output, done: false }`) but before the consumer has resumed the generator to observe its terminal `{ done: true }` transition. If the signal aborts in that window, loopx's signal listener performs cleanup, the next generator interaction settles with the abort error (not `{ done: true }`), and any outstanding `runPromise()` promise rejects with the abort error. Yielding the final iteration's output does not make normal completion irrevocable: the generator is considered to have "settled" as normally-complete only at its terminal `{ done: true }` transition, and an `AbortSignal` abort observed before that transition takes precedence over normal completion. This is consistent with SPEC §9.1's rule that aborting the signal always produces an error, and it applies regardless of whether the final iteration reached its final state via `stop: true` or via `maxIterations`. In practice the final-yield window is narrow: consumers driving `run()` via `for await (...)` automatically advance the generator past the final yield to `{ done: true }`, and `runPromise()` drives its internal generator to termination before resolving. The rule is observable in that window only by consumers that retain a reference to a `run()` generator, consume `next()` manually, pause after receiving the final output, and then abort before calling `next()` again.

**Manual abandonment is an exception.** If a caller manually consumes one or more `next()` results and then abandons the generator without calling `.return()` or driving it to completion, the generator is left suspended and cleanup is not reliably observable until the generator is garbage-collected (if ever). This is a limitation of JavaScript async generators, not a behavior loopx can enforce. This covers the final-iteration case as well: if the consumer calls `next()` and receives the final iteration's output (`{ value: Output, done: false }`) but does not call `next()` again to drive the generator to `{ done: true }` (or call `.return()`, call `.throw()`, or abort the signal), the generator remains suspended at the pre-completion yield point and cleanup is not observed. Callers that consume `run()` via `for await (...)` (which drives the generator past the final yield to `{ done: true }` automatically on normal completion and calls `.return()` on `break`) or who always pair manual `next()` calls with a final `.return()` will observe cleanup as specified; callers that keep references to suspended generators without resuming or closing them may observe a leaked tmpdir. This caveat does **not** apply to `runPromise()`, which always drives its underlying generator to completion, nor to `AbortSignal` abort, which cleans up eagerly from the signal listener regardless of suspension state.

### 2. Named run arguments

`loopx run` accepts named arguments of the form `name=value` after a `--` separator. Each named argument is injected into every child process's environment as an env var whose name is `name` and whose value is `value`. Named arguments are session-scoped (this run only) and function as the per-invocation counterpart to `loopx env set` (persistent). They are the sole caller-supplied parameterization channel for scripts; loopx does not append anything to `argv` and does not expose any `LOOPX_ARG_*` namespace.

**Scope of the feature: general env overlay, not a workflow-parameter namespace.** Named arguments are intentionally a general run-scoped environment overlay: any name that passes the name pattern and is not in the reserved `LOOPX_*` namespace may be bound, including names that collide with well-known POSIX env vars such as `PATH`, `HOME`, `LANG`, `LD_PRELOAD`, `LD_LIBRARY_PATH`, `NODE_OPTIONS`, `TMPDIR`, etc. loopx does not curate a list of env vars that are unsafe to override; it is the caller's responsibility to avoid shadowing names whose values the child process needs. This is the same contract that `loopx env set` / `-e` already provide at the file layer, extended here to the command-line layer. Authors who want a narrower "workflow parameter" namespace should adopt a naming convention in their scripts (e.g., prefix consumed names with the workflow slug) or validate received names explicitly; loopx does not enforce such a convention in v1.

#### Grammar

```
loopx run [options] <target> [-- [name=value]...]
```

The `--` separator is optional; a run with no named arguments may omit it entirely. If `--` is present, every token after it must match the `name=value` shape defined below; tokens before `--` are either run-scoped flags (`-n`, `-e`, `-h`, `--help`) or the target. No token between the target and the `--` separator may be a named argument — named arguments appear only after `--`.

#### Token shape

After `--`, each token must contain at least one `=` character. The name is the substring before the first `=`; the value is the substring after the first `=`. Additional `=` characters within the value are part of the value: `foo=a=b` binds name `foo` to value `a=b`.

- A token containing no `=` character (e.g., `foo`) is a parser usage error.
- A token beginning with `=` (empty name, e.g., `=value`) is a parser usage error (also fails the name pattern below).
- A token ending with `=` (empty value, e.g., `foo=`) is **valid**; it injects the name with the empty string as its value. Scripts that wish to distinguish "unset" from "set-to-empty" should check `printenv foo` / `"foo" in process.env` rather than `$foo` / `process.env.foo`.

#### Name rule

The name portion of each `name=value` token must match the pattern `[A-Za-z_][A-Za-z0-9_]*` — identical to `loopx env set`'s name rule (SPEC §4.3). A name that fails this pattern (e.g., `1abc`, `a-b`, `a.b`, `a b`, the empty string) is a parser usage error.

Names beginning with `LOOPX_` are **reserved** for the loopx protocol and are rejected as a parser usage error regardless of whether the specific name is currently injected by loopx. The reservation is the full `LOOPX_*` namespace, not just the names loopx currently uses (`LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, `LOOPX_TMPDIR`, `LOOPX_DELEGATED`). This reserves the namespace against future protocol additions and prevents authors from passing a value that would be silently overridden by loopx-injected precedence. The rejection is defense in depth alongside the precedence ordering in "Env precedence" below.

#### Value rule

The value portion of each `name=value` token may be any byte string (including the empty string) that does not contain a NUL byte (`\0`). NUL-containing values are rejected as a parser usage error. This is a POSIX constraint: `execve`-family syscalls cannot carry NUL in an env-string, so a NUL-containing value could not be faithfully delivered to the child even if loopx accepted it — rejecting at parse time is clearer than deferring to a child-spawn failure. **End-to-end delivery caveat.** POSIX `argv` strings themselves cannot contain NUL bytes either: an ordinary shell invocation (including `bash`'s `$'x\0y'`) cannot deliver a NUL-containing token to loopx's parser, so this rule is primarily a parser invariant rather than a rule that real-world CLI users are likely to trip over. It is still worth specifying because loopx's parser is also reachable by programmatic callers that construct CLI-style token arrays directly (for example, tests, embedding code, or IPC bridges), and for those callers a NUL-containing value is a well-defined parse-time usage error. The programmatic `RunOptions.env` NUL rejection, by contrast, is directly observable end-to-end.

Other control characters — newlines (`\n`), carriage returns (`\r`), tabs, bells, etc. — are **accepted**. This is more permissive than `loopx env set`, which rejects `\n` and `\r` because its on-disk env-file format cannot round-trip them. The runtime-injection path has no such file-format constraint: a value containing a newline is passed to the child's environment verbatim. Scripts that receive multiline values should parse them accordingly.

#### Duplicate names

If the same name appears more than once in a single `loopx run` invocation (e.g., `loopx run foo -- adr=0003 adr=0004`), this is a parser usage error. This is consistent with the existing duplicate-run-scoped-flag rule (`-n 5 -n 10` is a usage error).

Duplicate detection is case-sensitive — `adr` and `ADR` are different names and do not conflict.

#### Parsing algorithm

Run-level CLI parsing is normatively defined by the following two-phase algorithm. The Parsing rules that follow describe derived cases in human-readable form; where the rules and the algorithm appear to conflict, the algorithm is authoritative.

**Phase 1 — help short-circuit scan.** Scan the run-level tokens left-to-right, one token at a time. Flag parsing is initially enabled. At each position, apply the first matching rule:

1. **Help trigger.** If flag parsing is enabled and the current token is `-h` or `--help`, **run help is triggered**: loopx exits the run-execution parser path and hands control to the run-help command path, which prints run help and performs the non-fatal discovery behavior described in SPEC §11.2 before exiting with code `0` (see "Run-help discovery is preserved" below). This rule always takes precedence, including over any "consume as flag value" intent from a preceding `-n` or `-e`. **A `-h` or `--help` token is never consumed as the value of `-n` or `-e`;** it always triggers help when it is encountered before the first unconsumed `--` and while flag parsing is still enabled.
2. **Value-taking flag.** If flag parsing is enabled and the current token is `-n` or `-e`:
   - If the **next** token is `-h` or `--help`, do **not** mark it as a consumed flag value. Advance past only the current token; the next iteration will reach the `-h` / `--help` token and rule §1 will fire.
   - Otherwise, mark the next token (if any) as "consumed as a flag value" and advance past both. If no next token exists, do not raise an error in this phase — Phase 2 will report the missing value.
3. **Unconsumed `--`.** If flag parsing is enabled and the current token is `--` and has **not** been marked as a consumed flag value by a preceding `-n` or `-e`, this is the named-argument separator: flag parsing is now disabled. Continue scanning, but subsequent tokens can no longer trigger help.
4. **After `--`.** If flag parsing is disabled, the current token is a named-argument candidate for Phase 2 and has no help-trigger behavior.
5. **Otherwise.** Advance to the next token.

If the scan ends (end of input) without encountering `-h` or `--help` while flag parsing was enabled, proceed to Phase 2.

**Phase 2 — main parse.** Scan left-to-right again. Flag parsing is initially enabled.

While flag parsing is enabled:

- `-n` and `-e` consume the immediately following token as their value, regardless of that token's content (including `--`, a named-argument-looking `foo=bar` token, or a would-be target). If no following token exists, this is a usage error (missing flag value).
- A `--` token that is not consumed as a flag value disables flag parsing and is itself discarded (not injected into the child environment as a named argument).
- A bare `-` (a single hyphen with no other characters) is treated as a non-flag token (see below).
- Any other token that begins with `-` and is not a recognized flag is an unrecognized-flag usage error.
- A non-flag token is the target if no target has been consumed yet; otherwise it is an unrecognized-token usage error (no positional arguments exist between the target and the `--` separator — named arguments appear only after `--`).
- If a value-taking flag (`-n`, `-e`) is specified more than once, this is a duplicate-flag usage error.

After flag parsing is disabled (by `--`):

- Each remaining token must match the `name=value` shape (at least one `=`, name matching `[A-Za-z_][A-Za-z0-9_]*`, name not in `LOOPX_*`, value not containing NUL). A token that fails any of these checks is a parser usage error.
- A name that has already been consumed in this invocation is a duplicate-named-argument usage error.

If the first unconsumed `--` appears before any target has been consumed (e.g., `loopx run -- adr=0003`), this is a missing-target usage error.

**Worked examples.** Concrete outcomes that follow from the algorithm:

- `loopx run review-adr -- adr=0003 mode=strict` → target `review-adr`, env additions `{adr: "0003", mode: "strict"}`.
- `loopx run ralph -n 10 -- target=src/foo.ts` → `-n = 10`, target `ralph`, env additions `{target: "src/foo.ts"}`.
- `loopx run foo` → target `foo`, no named arguments.
- `loopx run foo --` → target `foo`, no named arguments (bare `--` with no following tokens is valid).
- `loopx run foo -- a=1 b=2` → two named arguments.
- `loopx run foo bar` → usage error (unrecognized token `bar` after target; use `--` to introduce named arguments).
- `loopx run foo -- 0003` → usage error (token does not match `name=value`).
- `loopx run foo -- =value` → usage error (empty name; fails name pattern).
- `loopx run foo -- foo` → usage error (no `=`).
- `loopx run foo -- foo=` → target `foo`, env additions `{foo: ""}` (empty value is valid).
- `loopx run foo -- foo=a=b` → target `foo`, env additions `{foo: "a=b"}` (value may contain `=`).
- `loopx run foo -- adr=0003 adr=0004` → duplicate-named-argument usage error.
- `loopx run foo -- adr=0003 ADR=0004` → target `foo`, env additions `{adr: "0003", ADR: "0004"}` (case-sensitive; not a duplicate).
- `loopx run foo -- 1abc=x` → usage error (name starts with digit, fails pattern).
- `loopx run foo -- a-b=x` → usage error (hyphen not in name pattern).
- `loopx run foo -- LOOPX_BIN=/tmp/fake` → usage error (reserved `LOOPX_*` name).
- `loopx run foo -- LOOPX_ANYTHING=x` → usage error (full `LOOPX_*` namespace reserved, not just currently-injected names).
- `loopx run -- adr=0003` → missing-target usage error (`--` before any target).
- `loopx run foo -h` → run help, exit 0.
- `loopx run foo -- -h` → usage error (`-h` does not match `name=value`; after `--`, `-h` is neither a help trigger nor a valid named argument).
- `loopx run foo -- -n 5` → usage error on `-n` (does not match `name=value`); flag parsing is disabled after `--`.
- `loopx run foo -e -- adr=0003` → `-e` consumes `--` as its value (binding `-e = "--"`); the separator has been absorbed. `adr=0003` is therefore a non-flag positional token after the target with flag parsing still enabled, which is an unrecognized-token usage error. Authors who want to combine `-e` and named arguments must ensure `-e`'s value is explicit: `loopx run foo -e ./env.local -- adr=0003`.
- `loopx run --unknown -h` → run help, exit `0`. Phase 1 encounters `-h` before Phase 2 ever runs, so `--unknown` is never rejected.
- `loopx run foo -e -h` → run help, exit `0`. Phase 1 does not consume `-h` as `-e`'s value.
- `loopx run foo -n -- 5` → parser-time usage error. Phase 1: `-n` marks `--` as consumed; `5` is neither help nor `--`; scan ends without help. Phase 2: `-n = "--"` fails `-n`'s non-negative-integer validation.
- `loopx run foo -n` → missing-value usage error (Phase 2).
- `loopx run foo -e` → missing-value usage error (Phase 2).

#### Parsing rules

The following rules restate and summarize the algorithm above; where a rule and the algorithm appear to conflict, the algorithm governs.

1. **Flags may appear anywhere before an explicit `--` separator.** Recognized run-scoped flags (`-n`, `-e`, `-h`, `--help`) are consumed as flags regardless of whether they appear before or after the target. This preserves the current SPEC's "options and the target may appear in any order" rule.
2. **The first non-flag token is the target.** Unchanged from the current SPEC. The target must appear before the first explicit `--`. If `--` appears before any target token (e.g., `loopx run -- adr=0003`), no target is consumed and the command is a missing-target usage error.
3. **No positional arguments between the target and `--`.** The current SPEC's "more than one positional argument is a usage error" rule continues to hold: any non-flag, non-consumed token after the target and before the first unconsumed `--` is a usage error. This ADR does not introduce positional arguments; the sole caller-supplied parameterization channel is named arguments after `--`.
4. **The `--` separator ends flag parsing and opens the named-argument region.** Every token after `--` must match `name=value` (see "Token shape," "Name rule," and "Value rule" above). The `--` token itself is consumed and not passed to any script as a named argument. A literal value of `--` can be passed as the value of a named argument by embedding it in the value: `loopx run foo -- arg=--`.
5. **Unrecognized flags before `--` are usage errors** (exit code 1). There is no way to pass a `-`-prefixed token to a script as a named-argument name — names must match `[A-Za-z_][A-Za-z0-9_]*`, which disallows a leading `-`.
6. **Duplicate run-scoped flags remain usage errors.** `loopx run -n 5 -n 10 foo -- adr=0003` is a usage error for duplicate `-n`, independent of whether named arguments are present.
7. **Duplicate named-argument names are usage errors.** `loopx run foo -- adr=0003 adr=0004` is a usage error. Case-sensitive (`adr` and `ADR` are different names).
8. **The `-h` / `--help` short-circuit is preserved, is position-sensitive, and takes precedence over all other run-level parsing.** When `-h` or `--help` appears **before** an explicit, unconsumed `--` separator (at any position), it triggers run help and exits 0, ignoring all other run-level tokens (target, other flags, would-be named arguments, and the would-be values of value-taking flags). The help scan takes precedence over value-taking flag consumption (rule §10): `loopx run foo -e -h`, `loopx run foo -n --help`, and `loopx run foo -e --help` all trigger run help; `-h` / `--help` is **not** bound as the value of `-e` / `-n`. When `-h` or `--help` appears **after** `--`, it is a named-argument candidate token; because `-h` does not match `name=value`, it is a parser usage error rather than a help trigger.
9. **Bare `-`** (a single hyphen with no other characters) is a non-flag token. It is treated as the target if no target has been consumed yet, otherwise as an unrecognized-token usage error. (In practice, `-` fails the workflow/script name restriction pattern and would cause a target-validation error if used as a target.)
10. **Run-scoped flags that take values** (`-n`, `-e`) consume the next token as their value, with one exception: if the next token is `-h` or `--help`, the help short-circuit (rule §8) takes precedence and that token is not consumed as a flag value. Outside that exception, value capture can absorb `--`: `loopx run foo -e -- bar` binds `-e = "--"`; `--` is consumed as the env-file path, not as the separator. Parsing then continues with `bar` as a non-flag token after the target with flag parsing still enabled, which is an unrecognized-token usage error under rule §3. The `-e = "--"` value is validated/loaded like any other `-e` value (a file literally named `--`, if it exists and is a valid env file, would load normally; otherwise the failure surfaces as a runtime env-file error). The `--` token acts as the named-argument separator only when it is not consumed as a flag value.

#### Exposure to scripts

Named arguments are injected into every child process's environment. Scripts read them using the same mechanisms they use for any other env var:

- **Bash scripts:** `$name`, `${name}`, `printenv name`. Names are injected verbatim (case-sensitive). Example: `loopx run foo -- adr=0003` makes `$adr` equal to `0003` inside the script.
- **JS/TS scripts:** `process.env.name`. Behavior is identical under `tsx` and Bun.

Named arguments are **not** exposed via `argv`. `process.argv.slice(2)` is empty for JS/TS scripts, and `$#` is `0` / `$1` is unset / `$@` is empty for Bash scripts, regardless of how many named arguments were passed. Scripts that need to propagate a named argument to sub-processes (e.g., `claude`, `codex`) read it from the env and pass it along explicitly — typically by re-exporting it or by passing it as a flag to the sub-process.

#### Propagation

For the CLI, the named-argument map is captured when `loopx run` parses its input. For the programmatic API, `RunOptions.env` is snapshotted when `run()` or `runPromise()` is called (a shallow copy of the object's own enumerable string-keyed entries). The captured map is immutable for the duration of the run.

- **Starting target:** receives the full named-argument map as env additions.
- **Scripts reached via `goto`** (intra-workflow or cross-workflow): receive the same map.
- **Loop reset to the starting target:** the starting target receives the same map on each reset.

There is no v1 mechanism for a `goto` to override, append to, or suppress the named-argument map for a downstream script. A script that needs to pass different data to a downstream script should use `result` (piped via stdin on `goto`), write to `$LOOPX_TMPDIR`, or spawn its own child with an explicit environment.

Changes a script makes to its own environment (e.g., `export FOO=bar` inside a Bash script, or mutating `process.env` inside a JS/TS script) do not propagate across iteration boundaries. Each iteration spawns a fresh child process with the same injected environment derived from the captured named arguments, `-e`, global env, and inherited env.

#### Env precedence

Named run arguments participate in the injection precedence defined in SPEC §8.3. The full precedence ordering after this ADR is (highest → lowest):

1. **loopx-injected protocol variables** (`LOOPX_BIN`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`, `LOOPX_TMPDIR`).
2. **Named run arguments** (this ADR — CLI `-- name=value` or programmatic `RunOptions.env`).
3. **Local env file** (`-e`).
4. **Global loopx env file** (`loopx env set` / `$XDG_CONFIG_HOME/loopx/env`).
5. **Inherited system environment.**

Named run arguments therefore override any same-named value from `-e`, the global loopx env, or the inherited system environment, and are themselves overridden only by loopx protocol variables. The protocol-variable override is enforced both by this precedence order and by the parser's rejection of `LOOPX_*` named arguments (defense in depth): a user cannot set `LOOPX_BIN` via `--`, because the parser rejects `LOOPX_*` names at step 1 before injection occurs.

**`LOOPX_*` reservation is asymmetric across layers, by design.** The `LOOPX_*` namespace is rejected at the named-arguments layer (CLI `--` tokens and programmatic `RunOptions.env`) as a parser/field-validation usage error, not because `LOOPX_*` names are "reserved across all injection layers." `-e` local env files, the global loopx env file (via `loopx env set`), and the inherited system environment may still contain `LOOPX_*` entries — there is no error at those layers — and loopx does not refuse to load them. Those entries are simply overridden by loopx-injected protocol variables at precedence tier 1: a user who sets `LOOPX_BIN=/tmp/fake` via `loopx env set` will not observe that value inside scripts, because tier-1 injection replaces it, but the set itself succeeds. The asymmetry is intentional: the named-arguments layer is the one place where a user might reasonably believe a value they supply will flow through, so a parser-time rejection is the clearest signal. Env-file layers rely on precedence alone because the broader env-file model (used for arbitrary variables) does not otherwise reserve names.

The rationale for placing named arguments above `-e`: named arguments are a per-invocation, command-line-level specification; `-e` is a per-invocation file reference. When both are used together on the same run, the command-line value is the more specific override and wins. Authors who want a value that the user cannot override from the command line should not rely on precedence for that guarantee — `-e` and `loopx env set` are not security boundaries; a determined caller can simply use a different command-line value.

#### No declaration or validation of semantics

Workflows do not declare expected named-argument names, types, defaults, or requiredness in `package.json` or elsewhere. loopx does not enforce that required named arguments are present, does not type-check values, and does not supply defaults. Scripts validate and parse their own named arguments — for example, a script that reads `$adr` may treat an empty value as "not supplied" or may exit non-zero if it is missing.

This keeps the contract minimal in v1. A first-class schema mechanism (e.g., a `named-args.schema.json` per workflow) may be introduced in a later ADR if script authors find the unchecked-input model insufficient.

#### Programmatic API

`RunOptions` gains an optional `env` field:

```typescript
interface RunOptions {
  maxIterations?: number;
  envFile?: string;
  signal?: AbortSignal;
  cwd?: string;
  env?: Record<string, string>;
}
```

- **Omitted, `undefined`, or an empty object:** scripts receive no additional env entries beyond the loopx-injected protocol variables, `-e`, global loopx env, and inherited system env (equivalent to CLI `loopx run <target>` with no `--` separator).
- **Type:** a non-null, non-array object treated as a record of own enumerable string-keyed properties. `env` is **not** required to be a structurally "plain" object in the strict sense — its prototype is not constrained and may be `Object.prototype` (literal `{...}`), `null` (`Object.create(null)`), or any other prototype (class instances, `Object.create(otherObject)`, etc.). What is required is that each entry in the `Object.keys(env)` snapshot — that is, each own enumerable string-keyed property — has a name matching `[A-Za-z_][A-Za-z0-9_]*`, a name that does not begin with `LOOPX_`, and a string value that does not contain a NUL byte (`\0`). The following are all step-1 field-validation errors, surfaced lazily per SPEC §9.1 (on first iteration for `run()`, as a rejection for `runPromise()`): non-object values (`null`, strings, numbers, booleans, functions); arrays; any snapshotted entry whose name fails the pattern, begins with `LOOPX_`, or whose value is not a string or contains a NUL byte. **Symbol-keyed, non-enumerable, and inherited properties of `env` are ignored** — they are not iterated by `Object.keys`, not validated, and not forwarded to child processes. In particular, a symbol-keyed property is **not** a validation error (even if the symbol's description would fail the name pattern), because it is simply not in the snapshot; the same applies to inherited and non-enumerable own properties. An `env` object constructed via `Object.create(someProto)` is accepted; a `Map`, `Set`, `Date`, or other object with no own enumerable string-keyed entries is also accepted but contributes no env injections.
- **Snapshotting:** if `env` is a non-null, non-array object, loopx captures its `Object.keys`-enumerable own string-keyed entries (and the corresponding values at each key) at call time. Mutating the original object after `run()` or `runPromise()` returns — adding keys, removing keys, or changing values — has no effect on the running loop, and the captured snapshot is what is later validated and forwarded to scripts. If `env` is not a non-null, non-array object (for example, `null`, a string, a number, a boolean, a function, or an array), the invalid raw value itself is captured at call time and is what is surfaced when lazy validation runs — mutating the original value after the call does not change the rejection reason. The snapshot/capture is the same call-time mechanism used for other `RunOptions` fields.
- **Duplicate names are impossible under the programmatic API,** because the names in `Object.keys(env)` are unique by construction (JavaScript does not permit duplicate own string keys on a single object). A CLI author who passes the same name twice on the command line surfaces as a parser usage error; the programmatic API has no analogous case. `env: { adr: "0003" }` and `env: { adr: "0004" }` are two separate calls with different sole values; there is no way to express "both" except by CLI or by an explicit override mechanism the script implements.
- **Validation runs regardless of `maxIterations: 0`.** `env` validation is a step-1 field-level check (see the programmatic pre-iteration sequence above). Step 1 runs before the `maxIterations: 0` early exit (step 6), so invalid `env` is rejected whether or not the loop would actually spawn a child. In particular, `run(target, { maxIterations: 0, env: "nope" as any })` still throws on first iteration and `runPromise(target, { maxIterations: 0, env: { "1abc": "x" } })` still returns a rejected promise — the options are rejected before the `maxIterations: 0` early return takes effect, and no tmpdir is created. The same precedence rule applies to every other step-1 field (`maxIterations`, `envFile`, `cwd`, `signal`) and to a non-string `target`.

### 3. Programmatic validation and cancellation semantics

The two preceding subsections introduce the user-facing additions (`LOOPX_TMPDIR` and named run arguments). Accepting this ADR also accepts the surrounding programmatic-API contracts that govern when `RunOptions` is validated, which pre-spawn outcome surfaces, and how cancellations interact with each other. The substantive rules are stated above (because they are tightly coupled to the tmpdir lifecycle and to the precise meaning of "cleanup runs at loop end" — without them, the rules in §1 and §2 would be ambiguous about when validation runs, which outcome surfaces, and when cleanup is guaranteed). This subsection enumerates them so that a reader scanning the table of contents does not miss what is being frozen here. Each item below is normative when this ADR moves to Accepted; the cross-references point to the section where the substantive rule is stated.

- **Programmatic `RunOptions` / `options` parameter shape validation.** The `target` parameter must be a string. The `options` parameter, if present, must be `undefined`, or a non-null, non-array object — non-object values (`null`, strings, numbers, booleans, functions) and arrays are rejected as invalid-options-shape errors before any individual `RunOptions` field is examined. Within `RunOptions`, `env`, `maxIterations`, `envFile`, `cwd`, and `signal` are each field-validated at step 1; `env` is treated as record-like (own enumerable string-keyed entries iterated via `Object.keys`; symbol-keyed, non-enumerable, and inherited properties are ignored — neither validated nor forwarded to child processes). Defined under §1, "Programmatic (`run()` / `runPromise()`) pre-iteration sequence," step 1.
- **Same-realm-only `AbortSignal`.** `RunOptions.signal` must be `instanceof AbortSignal` against the platform-global constructor. Cross-realm and duck-typed signals are rejected at step 1; cross-realm callers should bridge through a same-realm `AbortController`. Defined under §1, "Programmatic … pre-iteration sequence," step 1.
- **Order within step 1 (`target` / `options` / individual fields).** A non-string `target` surfaces in preference to an invalid-options-shape error; an invalid-options-shape error surfaces in preference to any individual `RunOptions` field error; the internal order among `RunOptions` field type-checks is implementation-defined. Step-1 errors as a whole surface in preference to all downstream steps (2 through 9). Defined under §1, "Programmatic … pre-iteration sequence," step 1.
- **Nine-step pre-spawn sequence and earliest-step-wins precedence (no-abort case).** Field validation → already-aborted check → discovery → env loading → target resolution → `maxIterations: 0` → version check → tmpdir creation → first spawn. When multiple pre-spawn outcomes are present and no signal has aborted, the earliest-numbered step wins. Defined under §1, "Programmatic … pre-iteration sequence," and the precedence paragraph immediately following it.
- **Step-2 listener-race contract.** The step-2 already-aborted check must install abort observation in a way that cannot lose an abort fired between the `.aborted` inspection and listener registration. The implementation may register the listener before inspecting `.aborted`, re-inspect after registering, or use any equivalent ordering; the externally guaranteed outcome is that an abort that fires at step 2 still surfaces in preference to every later pre-spawn step. Defined under §1, "Programmatic … pre-iteration sequence," step 2.
- **Abort dominance during pre-spawn (steps 2 through 7).** Once step 1 has succeeded with a valid `RunOptions.signal`, an abort observed before step 8 begins wins over any pre-spawn outcome from steps 3 through 7 (discovery, env loading, target resolution, `maxIterations: 0`, non-fatal version check) but does not win over step 1. Aborts during steps 3 through 7 require no cleanup (no tmpdir has yet been created). Defined under §1, "Abort-dominance during pre-spawn (steps 2 through 7)."
- **Abort or signal during step 8 (tmpdir creation in progress).** Dispatched on two observable preconditions — whether a directory was created on-disk and whether its device/inode identity was recorded. Recursive cleanup of a partial directory requires a recorded identity; if identity capture did not complete, cleanup is non-traversing only. The abort/signal wins over a concurrent tmpdir-creation failure. Defined under §1, "Abort or signal during tmpdir creation."
- **Broad `AbortSignal`-wins-until-terminal-settlement rule.** An abort observed before the generator has settled terminally wins over **any** other not-yet-settled outcome, including: pending normal completion (the final-yield window — the generator is considered to have settled as normally-complete only at its terminal `{ done: true }` transition, not when the final iteration's output is yielded), pending non-zero script exits, pending invalid / missing-`goto` errors not yet surfaced, in-flight `generator.return()` from the consumer (`break` from `for await` or explicit `.return()`), in-flight `generator.throw(err)` from the consumer (the consumer's `err` is shadowed by the abort error), and in-flight cleanup arising from any of those consumer-driven cancellations. Only step-1 field-validation errors and non-terminal cleanup-failure warnings survive. Defined under §1, "Programmatic API," "`AbortSignal` abort wins until the generator has settled terminally."
- **Env-object call-time snapshot semantics.** `RunOptions.env`'s own enumerable string-keyed entries (and the corresponding values at each key) are captured at call time via `Object.keys(env)`. Mutating the original object after `run()` or `runPromise()` returns — adding keys, removing keys, or changing values — has no effect on the running loop. Defined under §2, "Programmatic API," "Snapshotting."
- **Abort error shape is implementation-defined in v1.** The exact class, name, message, code, and `cause` of the abort error are not fixed by this ADR; tests should assert that the abort outcome wins over competing outcomes, not that the error has a particular shape. A future ADR may pin this. Defined under §1, "Programmatic API," "Abort error shape is implementation-defined."

A later ADR may tighten any individual item above (for example, pinning the abort error's exact class/shape, adopting fd-relative cleanup syscalls via `openat` / `unlinkat` with `AT_SYMLINK_NOFOLLOW`, or introducing a named-argument schema) without re-opening the decisions in §1, §2, or this subsection.

## Consequences

- **Workflow directories stop accumulating temp files.** Existing workflows can migrate `.feedback.tmp`, `.claude-output.tmp`, `.answer.tmp`, `.prompt.tmp`, and similar hand-off files to `$LOOPX_TMPDIR/feedback`, etc. Manual `rm -f` calls along happy paths can be removed. Workflow directories contain only version-controlled scripts and static assets.
- **Cleanup is automatic on failure.** Loops that abort via signals, error exits, or aborted `AbortSignal` no longer leak temp files. Users who previously relied on lingering `.tmp` files for post-mortem inspection must write those artifacts outside `$LOOPX_TMPDIR` (for example, under `$LOOPX_PROJECT_ROOT`).
- **Cross-run state is the caller's responsibility.** `LOOPX_TMPDIR` never persists across runs. A workflow that genuinely needs cross-run state (e.g., a long-lived counter or cache that must survive across separate `loopx run` invocations) must use `$LOOPX_PROJECT_ROOT`-relative storage. The existing `ralph` `.iteration.tmp` is **not** an example of cross-run state — it is intra-run state that has been leaking into the workflow directory, and it should migrate to `$LOOPX_TMPDIR` (see migration note below).
- **Concurrent runs are safe.** Parallel `loopx run` invocations of the same workflow do not clobber each other's temp files, which was previously possible with in-workflow `.tmp` files.
- **Per-invocation parameterization has a first-class, non-polluting channel.** Named run arguments are the session-scoped counterpart to `loopx env set`: persistent/global values go in `loopx env set`; single-run values go on the command line after `--`. Users no longer need to pollute the global env with values that are only meaningful for one run (and then clean up afterward with `loopx env remove`), and no longer need to fork per-input workflows.
- **One-off per-input workflows collapse into parameterized ones.** The current per-ADR duplication (`review-adr` implicitly scoped to ADR-0002, etc.) can be replaced by a single parameterized workflow: `loopx run review-adr -- adr=0003`. Script authors decide the named-argument contract.
- **Named-argument consumption is idiomatic env usage.** Scripts read named arguments the same way they read any other env var — `$adr` in Bash, `process.env.adr` in JS/TS — rather than via `$1` / `process.argv[2]` sentinels. This is consistent with every other loopx injection (`LOOPX_BIN`, `LOOPX_TMPDIR`, `-e`-loaded values, global env) and removes a distinct parsing surface for authors.
- **Multi-positional tokens between target and `--` remain usage errors.** `loopx run foo bar` is still a usage error (consistent with the existing SPEC's "more than one positional argument is a usage error" rule). This ADR does not introduce positional arguments, so there is no change to existing SPEC behavior on this point.
- **`goto` scripts see the same named-argument map.** Scripts reached via `goto` receive the starting target's captured named-argument map in their env. Scripts that previously assumed a clean env and reference a variable name that a caller later supplies via `--` may observe an unexpected value. Existing workflows in this repository do not depend on env names that overlap with plausible CLI names and are unaffected.
- **`-h` / `--help` remains a run-level short-circuit, but only before the first unconsumed `--`.** To pass `-h` / `--help` as a named-argument *value*, users must place it on the right-hand side: `loopx run foo -- h=-h` delivers `-h` as the value of the named argument `h`. `loopx run foo -- -h` is a usage error (`-h` does not match `name=value`). `loopx run foo -h` still triggers run help and exits.
- **`-e` immediately followed by `--` absorbs the separator.** `loopx run foo -e -- adr=0003` binds `-e = "--"` and then treats `adr=0003` as a non-flag positional token after the target (with flag parsing still enabled), which is an unrecognized-token usage error. Authors who want to combine `-e` with named arguments must ensure `-e`'s value is explicit: `loopx run foo -e ./env.local -- adr=0003`. This is a natural consequence of "value-taking flags consume the next token literally" and matches standard CLI conventions.
- **Named arguments are not a secret channel.** Named-argument values are injected as env vars and are visible to any same-user process that reads `/proc/<pid>/environ` (on Linux) or the platform equivalent, and to any sub-process the script spawns that inherits the parent's environment (unless the script filters its environment). Named-argument values are **not** forwarded to child scripts via `argv`, so scripts do not observe them in `process.argv` / `$@`, and a child-process command line does not include them unless the script explicitly re-forwards them. However, the `name=value` tokens are still present on the original `loopx run ... -- name=value` command line: they can be observed by same-user processes via `ps`, `/proc/<loopx-pid>/cmdline`, shell history, terminal scrollback, or equivalent platform mechanisms while (and sometimes after) loopx is running, and when startup delegation to a local `loopx` binary occurs (per SPEC §3.2, delegation is a startup/project-root behavior that happens before command parsing — not a cross-workflow `goto` behavior), the delegated process command line may contain them as well. Named arguments are therefore **not** a secret channel: the "not placed on child argv" property is an internal propagation detail of how loopx injects named arguments (as env vars rather than as positional argv), not a confidentiality property, and it does not hide values from a same-user observer of the loopx process itself. Users should not pass secrets through named arguments; env files (global loopx env or `-e` local) or another protected mechanism with appropriate file permissions should be used instead. This applies equally to `RunOptions.env` under the programmatic API, since `env` entries are forwarded verbatim to child processes — and while `RunOptions.env` values are never placed on any process's command line, they are still observable via `/proc/<pid>/environ` of any child that inherits them (subject to the same same-user process-visibility caveats above).
- **No first-class iteration counter.** Workflows that need an iteration counter maintain one themselves in `$LOOPX_TMPDIR/iteration` (or similar). A first-class counter may be introduced in a later ADR if this pattern proves insufficient.
- **No schema for named arguments.** Workflows do not declare expected named-argument names, types, defaults, or requiredness; scripts validate their own input. A first-class schema mechanism may be introduced in a later ADR if the unchecked-input model proves insufficient.
- **Migration of existing workflows is manual but straightforward.** No automated migration tooling is provided. The existing `review-adr`, `apply-adr`, and `spec-test-adr` workflows can be refactored to (a) replace in-workflow `.tmp` paths with `$LOOPX_TMPDIR/…`, (b) remove manual `rm -f` cleanup, and (c) parameterize ADR references as a named argument (e.g., read `$adr` and invoke as `loopx run review-adr -- adr=0003`). The `ralph` workflow's `.loopx/.iteration.tmp` counter is intra-run leaked state and should migrate to `$LOOPX_TMPDIR/iteration` (or similar), allowing its manual cleanup branch to be removed.

The programmatic API's pre-iteration sequence additionally begins with a step-1 field-validation check (target parameter type, `options` parameter shape, plus `RunOptions` field types for `env` — including the object/key-pattern/`LOOPX_*`-reserved/value-NUL checks — `maxIterations`, `envFile`, `cwd`, `signal` — including the `instanceof AbortSignal` requirement), followed by a step-2 already-aborted `AbortSignal` check that runs after field validation succeeds but before discovery; full ordering and precedence among pre-spawn outcomes (including abort-dominance during steps 3 through 7) are documented in §9.1 / §9.2. Add: on normal loop completion — `stop: true` (current step 9) or `-n` / `maxIterations` reached (current step 10) — `LOOPX_TMPDIR` cleanup runs before loopx exits with code 0. For the programmatic API, the final iteration's output is yielded to the consumer before the generator settles; `LOOPX_TMPDIR` cleanup runs after that final yield and before the generator's terminal `{ done: true }` transition (and, for `runPromise()`, before the outstanding promise resolves), **except** when an `AbortSignal` abort intervenes in that window — abort continues to take precedence until the generator has settled terminally, per the abort-after-final-yield rule documented in §9.1 / §9.2. Subject to the manual-abandonment caveat documented under §9.1 / §9.2, consumers who drive `run()` via `for await (...)` or who use `runPromise()` observe cleanup as part of normal completion. The full `LOOPX_TMPDIR` lifecycle across normal completion, error exit, signals, and programmatic cancellation is defined in the new §8.4.
- **7.2 (Error Handling)** — Note that on each error case listed (non-zero script exit, invalid `goto` target, missing workflow / missing script during a `goto` resolution), `LOOPX_TMPDIR` cleanup runs after the error is detected and before loopx exits with code 1. The full lifecycle definition lives in the new §8.4.
- **7.3 (Signal Handling)** — Add: on SIGINT / SIGTERM, `LOOPX_TMPDIR` cleanup runs before loopx itself exits with the signal's exit code. If a child process group is active when the signal arrives, cleanup runs after the process group has exited (which may be immediately, or after the grace period including the `SIGKILL` escalation to the process group if required). If no child process is active when the signal arrives — including the window after `LOOPX_TMPDIR` has been created but before the first child has spawned, and the window between a child exiting and the next child spawning — cleanup runs immediately. Signal arrivals that precede `LOOPX_TMPDIR` creation (e.g., during discovery, env loading, target resolution, or `-n 0` early exit) require no cleanup and are governed by the existing §7.3 "between iterations" rule. Signals that arrive while tmpdir creation/securing is in progress (step 8 has begun but has not yet reached a terminal outcome) are covered by the "Abort or signal during tmpdir creation" rule in the new §8.4, which dispatches on two observable preconditions (whether a directory was created on-disk, and whether its device/inode identity was recorded): when identity was recorded, loopx performs identity-gated recursive cleanup of the partial directory; when a directory was created but identity was not recorded, loopx performs only non-traversing cleanup (unlink of a symlink or non-directory replacement, otherwise a warning with the directory left in place); when no directory was created, no cleanup is needed. loopx then exits with the signal's exit code, and the signal wins over a concurrent tmpdir-creation failure.

Environment Variables** — Add a new subsection (e.g., "8.4 Temporary Directory") documenting `LOOPX_TMPDIR`: creation timing (including non-creation for pre-spawn outcomes such as failures, early exits, and aborts during steps 2 through 7, and separately the in-progress step-8 abort/signal rule, which dispatches on whether a directory was created on-disk and whether its device/inode identity was recorded — recursive cleanup of any partial directory requires a recorded identity; if identity recording did not complete, cleanup is non-traversing only), location, mode, scope, lifecycle (including the path-based best-effort guarantee that cleanup applies to the original tmpdir when it remains at the original path and does not chase renamed tmpdirs, and the explicit policy for non-directory replacements at the original path — at most unlinked as a single non-traversing entry, never followed), cleanup cases (including the threat-model scope of replacement-safety and the device/inode identity check that gates recursive removal, with identity recording normative and the specific stat-family syscall used to capture it an implementation detail), cleanup-failure handling, concurrency, and `-n 0` behavior. Document `env`'s object-model contract as record-like rather than strictly plain: `env` is accepted as any non-null, non-array object; loopx iterates its own enumerable string-keyed entries via `Object.keys(env)` and ignores symbol-keyed, non-enumerable, and inherited properties (these are neither validated nor forwarded to child processes). Document `env` snapshotting: if `env` is a non-null, non-array object, loopx captures its `Object.keys`-enumerable own string-keyed entries at call time and mutating the original object after the call has no effect on the running loop; if `env` is not a non-null, non-array object (null, strings, numbers, booleans, functions, arrays), or if a snapshotted entry has a key that fails the name pattern, begins with `LOOPX_`, has a non-string value, or has a value with an embedded NUL byte (`\0`), the invalid raw value (or invalid entry) is captured at call time and surfaced as the rejection reason during lazy validation. Document the programmatic pre-iteration sequence and the resulting precedence among pre-spawn outcomes: (1) field-level validation — `target` must be a string; the `options` parameter, if present, must be `undefined`, or a non-null, non-array object (non-object values, including `null`, strings, numbers, booleans, functions, and arrays, are invalid-options-shape errors); `RunOptions` fields are then type-checked (`env` — non-null, non-array object whose own enumerable string-keyed entries have names matching `[A-Za-z_][A-Za-z0-9_]*` not beginning with `LOOPX_` and string values without NUL, with symbol-keyed / non-enumerable / inherited properties ignored; `maxIterations` — non-negative integer; `envFile` and `cwd` — strings; `signal` — `instanceof` the platform-global `AbortSignal`, with cross-realm and duck-typed signals rejected and an explicit recommendation to bridge cross-realm signals through a same-realm `AbortController`), with `target` validated before the `options` shape, the `options` shape validated before any individual `RunOptions` field, and the internal order among `RunOptions` field checks implementation-defined; (2) already-aborted `AbortSignal` check — if `RunOptions.signal` was provided (and passed step 1 as a valid `AbortSignal`) and `.aborted` is already `true`, the run aborts here; (3) discovery and global validation; (4) env loading (global + `envFile` local); (5) starting-target resolution (string syntax, workflow existence, script existence); (6) `maxIterations: 0` early exit; (7) starting workflow version check; (8) `LOOPX_TMPDIR` creation; (9) first spawn. When multiple pre-spawn outcomes are present and no signal has aborted, the earliest-numbered step wins: in particular, field validation (step 1, including non-string `target`, invalid options shape, and invalid `env` / `maxIterations` / `envFile` / `cwd` / `signal`) runs before the already-aborted signal check (step 2), discovery (step 3), env loading (step 4), target resolution (step 5), and the `maxIterations: 0` early exit (step 6). Env loading (step 4) runs before target resolution (step 5), so a missing or malformed env file surfaces in preference to a missing workflow or missing script in the absence of an abort. `env` validation accordingly runs regardless of `maxIterations: 0`, as a consequence of the precedence rule. **Abort-dominance during pre-spawn:** once step 1 has succeeded with a valid `RunOptions.signal`, any abort observed before the first child spawns — whether already-aborted at step 2 or aborted during steps 3 through 7 (discovery, env loading, target resolution, `maxIterations: 0`, non-fatal version check) — wins over those pre-spawn outcomes and prevents any later step (tmpdir creation, first spawn) from running, but does not win over step 1. Aborts during steps 3 through 7 require no cleanup since step 8 has not begun and no tmpdir has been created. Aborts that fire during step 8 itself (tmpdir creation/securing in progress) are covered by a separate rule: loopx waits for the in-flight operation to settle, dispatches on whether a directory was created on-disk and whether its device/inode identity was recorded (recursive cleanup requires recorded identity; non-traversing cleanup otherwise), and then surfaces the abort error; the abort wins over a concurrent tmpdir-creation failure. **Listener race:** the step-2 already-aborted check must install abort observation in a way that cannot lose an abort occurring between the `.aborted` inspection and listener registration (for example, register the listener before inspecting `.aborted`, or re-inspect after registering); the broad abort-dominance guarantee depends on this. Document tmpdir creation timing by API: for `run()`, the pre-iteration sequence runs on the first `next()` call and `LOOPX_TMPDIR` (if creation is reached) is created within that first `next()` — a `run()` generator that is never iterated executes no pre-iteration work and creates no tmpdir. For `runPromise()`, the pre-iteration sequence runs asynchronously after the promise is returned and `LOOPX_TMPDIR` (if creation is reached) is created asynchronously before the first child spawn — a `runPromise()` whose pre-spawn outcome falls at or before step 7 (any of step 1's failures, an already-aborted signal at step 2, an abort observed during steps 3 through 7, a discovery error, an env-file loading failure, a target-resolution failure, or `maxIterations: 0`) creates no tmpdir and settles without a cleanup step; a pre-spawn outcome at step 8 (tmpdir creation failure, or abort/signal observed while step 8 is in progress) may have created a partial directory and is handled by the "Abort or signal during tmpdir creation" / "Creation failure" rules above. Also document `LOOPX_TMPDIR` cleanup behavior under programmatic cancellation: for `AbortSignal` abort and consumer-driven cancellation (`break`, `generator.return()`, and explicit `generator.throw(err)`), if a child process group is active, loopx first terminates it per §9.1's cancellation semantics; cleanup runs after the child process group has exited or been killed and before the generator settles (completes, throws the abort error, or throws `err`) or the promise settles. `generator.throw(err)` is treated like consumer-driven cancellation for child termination and cleanup; the generator settles by throwing `err` after cleanup. For `AbortSignal` abort, cleanup runs eagerly from loopx's signal listener even when the generator is suspended between `next()` calls (no child process active); the next `next()` call settles with the abort error, and `runPromise()`'s outstanding promise rejects with the abort error as soon as cleanup completes — the caller is not required to resume or explicitly close the generator. Document the broad **`AbortSignal` abort wins until terminal settlement** rule: an abort observed before the generator has settled terminally wins over **any** other not-yet-settled outcome — pending normal completion (including the narrow final-yield window described next), pending non-zero script exits, pending invalid / missing `goto` errors not yet surfaced, pending pre-spawn outcomes from steps 3 through 7, in-flight `generator.return()` from the consumer (`break` from `for await` or explicit `.return()`), in-flight `generator.throw(err)` from the consumer (the consumer's `err` is shadowed by the abort error), and in-flight cleanup from any of those consumer-driven cancellations. The only outcomes that survive are step-1 field-validation errors (which precede signal registration entirely) and cleanup-failure warnings (which are non-terminal and never change the terminal outcome reason). Document the `AbortSignal`-after-final-yield rule as a specific application of the broad rule: if the signal fires after the final iteration's `Output` has been yielded (`{ value: Output, done: false }`) but before the consumer has resumed the generator to observe `{ done: true }`, the next generator interaction throws the abort error (not `{ done: true }`), cleanup runs before the throw, and any outstanding `runPromise()` promise rejects with the abort error. Yielding the final output does not make normal completion irrevocable. This holds regardless of whether the final iteration reached its final state via `stop: true` or via `maxIterations`. Cleanup failure prints a warning to stderr and does not change the generator or promise outcome. Document the manual-abandonment caveat: cleanup is guaranteed when the generator is driven to completion, throws, is explicitly closed with `generator.return()` (including `for await` completion and `break` from `for await`), is closed with `generator.throw()`, or is aborted via `AbortSignal`; a caller that manually consumes `next()` results and then abandons the generator without calling `.return()` — including a caller that stops after receiving the final iteration's yielded output without driving the generator to `{ done: true }` — may leave cleanup unobserved. Document tmpdir creation-failure behavior: `run()` throws on first iteration and `runPromise()` rejects; no script is spawned.

## Test Recommendations

These highlight edge cases that are easy to overlook. They are not an exhaustive test plan.

### Temporary directory

- Verify the same `LOOPX_TMPDIR` value is observed across the starting target, scripts reached via intra-workflow `goto`, scripts reached via cross-workflow `goto`, and the starting target on loop reset.
- Verify the directory is created with mode `0700`.
- Verify the directory is removed after SIGINT/SIGTERM once the active child process group has exited, including the case where loopx escalates to SIGKILL after the 5-second grace period defined in SPEC §7.3.
- Verify SIGINT/SIGTERM received **between iterations** — after `LOOPX_TMPDIR` has been created but while no child process is active (e.g., between a script exit and the next child spawn, including the window after a `goto` target has been resolved but before the resolved child has started) — still cleans up the tmpdir before loopx exits with the signal exit code, matching SPEC §7.3's "between iterations" exit behavior.
- Verify SIGINT/SIGTERM received **between `LOOPX_TMPDIR` creation and the first child spawn** — i.e., the tmpdir has been created but no child has yet been spawned for the run — still cleans up the tmpdir before loopx exits with the signal exit code. This case is distinct from the already-tested "between iterations" case and from pre-tmpdir-creation signal arrivals.
- Verify `AbortSignal` abort received **between `LOOPX_TMPDIR` creation and the first child spawn** (no child process is active) triggers eager cleanup from loopx's signal listener; for `run()`, the first `next()` call settles with the abort error after cleanup completes, and for `runPromise()`, the outstanding promise rejects with the abort error after cleanup completes. The tmpdir is removed before the abort is surfaced. This case is distinct from the already-aborted-signal step-2 check (which runs before tmpdir creation and leaves no tmpdir to clean up).
- Verify `AbortSignal` abort received **between one child exiting and the next child spawning within a single ongoing `next()` call** — including during `goto` target validation/resolution, during first-entry version checking for a target workflow reached via cross-workflow `goto`, or after iteration N's output has been prepared but before iteration N+1's child has been spawned — triggers cleanup of `LOOPX_TMPDIR` before the generator settles. The prior iteration's child has already exited and no new child is yet active, so cleanup proceeds without waiting for a process group. The settled outcome is the abort error (not the prior iteration's success or any in-flight `goto`-resolution error), consistent with the abort-dominance rule.
- Verify the directory is removed after a script exits non-zero.
- Verify the directory is removed after an invalid `goto` target is produced.
- Verify programmatic `break` out of a `for await` triggers cleanup before the generator completes.
- Verify `generator.return()` triggers cleanup.
- Verify `generator.throw(err)` during iteration terminates the active child process group, runs cleanup, and then settles the generator by throwing `err` (cleanup runs before the generator settles).
- Verify aborting the `AbortSignal` mid-iteration triggers cleanup before the generator throws.
- Verify that a signal aborted before the first `next()` call is detected by the step-2 already-aborted check (after step-1 field validation succeeds) and surfaces the abort error before discovery, env loading, target resolution, or tmpdir creation run. No tmpdir is created, so no cleanup is required.
- Verify the step-2 listener-race contract: an abort fired between the step-2 `.aborted` inspection and listener registration must still be observed and must still produce the abort error in preference to later pre-spawn outcomes. An implementation is free to close the window by registering the listener before inspecting `.aborted`, re-inspecting after registration, or any equivalent ordering; the test asserts the *outcome* (abort surfaced, no missed abort), not the implementation.
- Verify that aborting the `AbortSignal` while the generator is suspended between `next()` calls (no active child) triggers cleanup eagerly from loopx's signal listener, without requiring the caller to call `.return()` or resume; a subsequent `next()` call settles with the abort error, and any outstanding `runPromise()` promise rejects with the abort error as soon as cleanup completes.
- Verify the `AbortSignal`-after-final-yield rule: if the signal fires **after** the final iteration's `Output` has been yielded (`{ value: Output, done: false }`) but **before** the consumer has resumed the generator to observe `{ done: true }`, the next `next()` call throws the abort error (not `{ done: true }`), cleanup runs before the throw, and any outstanding `runPromise()` promise rejects with the abort error. Cover both paths to the final iteration: final state reached via `stop: true` and final state reached via `maxIterations` / `-n`.
- Verify two concurrent `loopx run` invocations receive distinct `LOOPX_TMPDIR` values and distinct directories.
- Verify a user-supplied `LOOPX_TMPDIR` in the inherited environment is overridden by the injected value.
- Verify a user-supplied `LOOPX_TMPDIR` in the global env file is overridden by the injected value.
- Verify a user-supplied `LOOPX_TMPDIR` in the `-e` local env file is overridden by the injected value.
- Verify under `-n 0`, no tmpdir is created and no child process is spawned (and therefore no `LOOPX_TMPDIR` is observed).
- Verify a discovery error (e.g., missing `.loopx/` directory) does not create a `LOOPX_TMPDIR`.
- Verify a starting-target resolution failure (e.g., non-existent workflow or missing script) does not create a `LOOPX_TMPDIR`.
- Verify an env-file loading failure (`-e` pointing to a missing or invalid file) does not create a `LOOPX_TMPDIR`.
- Verify cleanup runs after the child process group has exited for programmatic `AbortSignal` abort mid-iteration (i.e., the tmpdir is not removed while a child is still running).
- Verify cleanup runs after the child process group has exited for programmatic consumer-driven cancellation (`break` / `generator.return()`) mid-iteration.
- Verify cleanup failure under the programmatic API prints a warning to stderr but does not change the generator's success/error/abort outcome.
- Verify cleanup failure (simulate via making the tmpdir contents non-removable) prints a warning to stderr but does not change the loop's exit code.
- Verify a workflow's own `node_modules/`, `.loopx/`, or other non-tmpdir state is not affected by cleanup.
- Verify that if a script creates a symlink inside `$LOOPX_TMPDIR` pointing at a file or directory **outside** the tmpdir, cleanup removes the symlink entry itself but does not delete or modify the target. This must hold for both regular files and directories as link targets.
- Verify that if a script replaces the `$LOOPX_TMPDIR` path itself with a symlink before cleanup runs (e.g., removes the tmpdir and creates a symlink of the same name pointing elsewhere, or renames the tmpdir and creates a symlink at the original path), cleanup unlinks the symlink entry at the original path and does **not** traverse the symlink or delete / modify its target. This must hold regardless of whether the symlink target is a file, a directory, or a non-existent path.
- Verify the path-based cleanup guarantee by running a script that renames `$LOOPX_TMPDIR` away from its original path (e.g., `mv "$LOOPX_TMPDIR" /tmp/saved-loopx-dir`) and creates an unrelated directory at the original path: at cleanup time, loopx refuses to recurse into the unrelated directory at the original path (device/inode mismatch → single stderr warning, unrelated directory intact), and loopx does **not** chase the renamed directory to `/tmp/saved-loopx-dir`. The test asserts (a) the unrelated directory at the original path is untouched, (b) the renamed directory at the new path is left intact (this is the documented "does not chase renamed tmpdirs" limitation), and (c) the terminal loop outcome is unchanged.
- Verify that scripts can write subdirectories, binary files, and Unix sockets/FIFOs under `$LOOPX_TMPDIR` without loopx interference.
- Verify `$LOOPX_TMPDIR` lives under `os.tmpdir()` and its basename begins with `loopx-`.
- Verify that if any step-8 precondition fails to be met — simulate failure to create the directory, failure of the identity-capture step, or failure to secure or verify mode `0700` — no child process is spawned: the CLI exits `1` with a stderr error, `run()` throws on first iteration, and `runPromise()` rejects. The test should describe the failing precondition rather than a specific syscall, since the ADR does not require a particular syscall sequence.
- Verify that a tmpdir-creation failure after a partial directory has been created triggers best-effort cleanup of the partial directory, and that failure of that best-effort cleanup does not mask the original creation error (the surfaced error/exit reflects the creation failure). If the partial directory's identity was never recorded (the failure occurred before identity capture completed), cleanup performs only non-traversing actions per the identity-recording gap rule documented under "Abort or signal during tmpdir creation."
- Verify that invalid `RunOptions.env` (values that are not a non-null, non-array object — e.g., `null`, strings, numbers, booleans, functions, arrays — or objects whose snapshotted own-enumerable-string-keyed entries have a key that fails the name pattern, a `LOOPX_*` key, a non-string value, or a value containing NUL bytes such as `{ a: "x\0y" }`) rejects without creating a tmpdir, even when the options would otherwise reach the iteration loop. NUL-byte rejection is a step-1 field-validation error, not a deferred child-spawn error.
- Verify that `run(target, { maxIterations: 0, env: "nope" as any })` rejects on first iteration and `runPromise(target, { maxIterations: 0, env: { "1abc": "x" } })` returns a rejected promise — `env` validation runs before the `maxIterations: 0` early return, and no tmpdir is created.
- Verify that the `options` parameter itself is shape-validated independently of its fields: `run("foo", null as any)`, `run("foo", "bad" as any)`, `run("foo", 123 as any)`, `run("foo", true as any)`, `run("foo", [] as any)`, and `run("foo", (() => {}) as any)` each reject with an invalid-options-shape error — distinct from any individual `RunOptions` field error. `run("foo", undefined)` and `run("foo")` are equivalent and both succeed shape validation. The `runPromise` analogues reject with the same shape error.
- Verify that `runPromise(42 as any, null as any)` rejects with the non-string-`target` error, not the invalid-options-shape error (target is validated before options shape).
- Verify that `runPromise("foo", null as any)` (valid target, invalid options shape) rejects with the invalid-options-shape error before any individual field check would run.
- Verify that a cross-realm `AbortSignal` (e.g., one constructed via `vm.runInNewContext("new AbortController()")` in Node, or a duck-typed object with `aborted` and `addEventListener` but lacking `instanceof AbortSignal`) is rejected at step 1 as an invalid `RunOptions.signal`. Verify that bridging the cross-realm signal through a same-realm `AbortController` (forwarding the cross-realm `abort` event to a same-realm controller and passing `controller.signal`) is accepted.
- Verify abort-dominance during pre-spawn (steps 3 through 7): if an `AbortSignal` aborts after step 2's already-aborted check has passed but before step 8 (tmpdir creation) begins — for example, while a long-running discovery is in progress, while an env file is being loaded, while target resolution is in progress, or after the non-fatal version check warning has been printed but before the underlying tmpdir-creation call is invoked — the run aborts with the abort error in preference to whichever pre-spawn condition would otherwise have surfaced. Cover at minimum: abort-during-discovery winning over a discovery error; abort-during-env-loading winning over an env-file error; abort-during-target-resolution winning over a missing-workflow / missing-script error. No tmpdir is created in any of these cases.
- Verify abort-or-signal during step 8 (tmpdir creation/securing in progress): if an `AbortSignal` aborts, or SIGINT/SIGTERM is received, after step 8 has begun but before all three step-8 preconditions (directory created, identity recorded, mode `0700` ensured) have been met. Cover at minimum the three observable states defined by the "Abort or signal during tmpdir creation" rule:
  - (a) **No directory was created yet** (the abort/signal arrived before the underlying create call produced anything on-disk): no cleanup is needed.
  - (b) **A directory was created but its identity was not recorded yet** (the abort/signal arrived between directory creation and identity capture): loopx must **not** recursively remove the path. Verify that if the path still resolves to the directory loopx created, loopx leaves it in place with a single stderr warning; if the path has been replaced with a symlink or non-directory entry, loopx may unlink that single entry as a non-traversing action. This case is distinct from the identity-recorded case below.
  - (c) **A directory was created and its identity was recorded** (the abort/signal arrived after identity capture, regardless of whether mode-0700 securing had completed): loopx performs identity-gated recursive cleanup per the "Observable behavior" section. Mode-0700 status is not observable to any child in this case (the child was never spawned), so the test should not assert on it.

  In all subcases, loopx does not spawn a child and surfaces the abort/signal outcome. If best-effort cleanup fails, loopx prints a single stderr warning that does not mask the abort/signal outcome.
- Verify abort-wins-over-tmpdir-creation-failure: if the underlying create call (or a subsequent identity-capture or mode-securing step) is about to surface a creation failure (e.g., `ENOSPC`, `EACCES`) and an `AbortSignal` aborts — or SIGINT/SIGTERM arrives — concurrently while step 8 is in progress, the abort/signal wins over the creation failure. The surfaced terminal outcome is the abort/signal, not the creation failure. Any partial directory that was created is still cleaned up best-effort subject to the identity-recording gap rule (recursive removal requires a recorded identity).
- Verify abort-wins-over-pending-outcome at iteration time: if the active child process group has already exited with a non-zero exit code but loopx has not yet surfaced the script-failure error to the generator, and the `AbortSignal` aborts in that window, the generator settles with the abort error (not the script-failure error). Cover the analogous cases for invalid / missing `goto` targets that have been detected but not yet surfaced.
- Verify abort-wins-over-consumer-cancellation: if the consumer calls `break` from a `for await` (which triggers `generator.return()`) and the `AbortSignal` aborts before the generator has terminally settled, the generator settles with the abort error rather than `{ done: true }`. Cover the analogous case for an explicit `generator.return()` and for `generator.throw(err)`: abort wins over the consumer-supplied `err`, so the generator throws the abort error rather than `err`.
- Verify programmatic pre-spawn precedence: when a call combines multiple pre-spawn outcomes (no signal aborted), the surfaced error reflects the earliest-numbered step in the programmatic pre-iteration sequence (field validation → already-aborted signal → discovery → env loading → target resolution → `maxIterations: 0` → version check → tmpdir → spawn). In particular:
  - `runPromise("missing-workflow", { env: { "1abc": "x" } })` rejects with the `env` validation error, not the missing-workflow error (step 1 wins over step 5).
  - `run(42 as any, { env: { "1abc": "x" } })` rejects on first iteration with a step-1 field-validation error; the ADR guarantees that `target`'s non-string rejection surfaces in preference to the `env` error, so the surfaced error identifies the non-string `target`.
  - `run("unknown:target", { env: { "1abc": "x" }, envFile: "./nonexistent" })` rejects on first iteration with the `env` validation error (step 1), not the missing-env-file (step 4) or missing-target (step 5) error.
  - `runPromise("missing-workflow", { envFile: "./nonexistent" })` (no invalid field values) rejects with the env-file loading error in preference to the missing-workflow error, because env loading (step 4) precedes target resolution (step 5). Missing-workflow is a target-resolution failure, not a discovery failure — discovery only builds the workflow list.
  - `run("missing-workflow", { signal: alreadyAbortedSignal })` (no invalid field values) rejects on first iteration with the abort error in preference to the missing-workflow error, because the already-aborted signal check (step 2) precedes target resolution (step 5).
  - `runPromise("x", { signal: "not-a-signal" as any })` rejects with the step-1 field-validation error for `signal`; field validation (step 1) runs before the already-aborted check (step 2), so an invalid-`signal` type surfaces in preference to any already-aborted treatment.
  - `runPromise("missing-workflow", null as any)` rejects with the invalid-options-shape error, not the missing-workflow error; the options shape is checked at step 1 before downstream steps.
  - `runPromise(42 as any, "bad" as any)` rejects with the non-string-`target` error (target is validated before the options shape).
  - `run("missing-workflow", { signal: alreadyAbortedSignal, env: { a: "x\0y" } })` rejects with the step-1 NUL-byte field-validation error, not the abort error or the missing-workflow error; field validation (step 1) precedes the already-aborted check (step 2).
  - `runPromise("missing-workflow", { env: { LOOPX_BIN: "/fake" } })` rejects with the step-1 reserved-name error, not the missing-workflow error.

  In none of these cases is a tmpdir created.
- Verify that a caller that consumes one or more `next()` results and then abandons the generator without calling `.return()` may leave the tmpdir uncleaned (document-only — this is a JS-language limitation, not a loopx guarantee). Verify that the same scenario under `for await` (which calls `.return()` on loop exit, including `break`) always cleans up.

### Named arguments

- Verify `loopx run foo -- adr=0003` sets `process.env.adr === "0003"` (JS/TS) and `$adr == "0003"` (Bash), and that `process.argv.slice(2)` is `[]` / `$# == 0`.
- Verify `loopx run foo -- adr=0003 mode=strict` sets both env vars; order of injection is not observable to the test (the child sees both at spawn time).
- Verify `loopx run foo` (no `--`) results in the child's `process.env` containing none of the script-supplied named-argument keys and `process.argv.slice(2)` being `[]`.
- Verify `loopx run foo --` (bare `--` with no following tokens) is valid and results in no named arguments (same observable state as the no-`--` case above).
- Verify `loopx run foo -- foo=` sets `process.env.foo === ""` (empty string, distinct from unset). For Bash, `[[ -n "${foo+x}" ]]` returns true (set) but `$foo` is empty.
- Verify `loopx run foo -- foo=a=b` sets `process.env.foo === "a=b"` (value may contain `=`).
- Verify `loopx run foo -- foo=$'a\nb'` (a literal newline in the value) passes a newline-containing value through to the child unchanged. Runtime injection is more permissive than `loopx env set`'s file format (which rejects `\n`/`\r`).
- Verify `loopx run foo -- 0003` is a usage error (token does not match `name=value`).
- Verify `loopx run foo -- =value` is a usage error (empty name; fails name pattern).
- Verify `loopx run foo -- foo` is a usage error (no `=`).
- Verify `loopx run foo -- 1abc=x` is a usage error (invalid name pattern).
- Verify `loopx run foo -- a-b=x` is a usage error (invalid name pattern; hyphen).
- Verify `loopx run foo -- a.b=x` is a usage error (invalid name pattern; dot).
- Verify `loopx run foo -- LOOPX_BIN=/fake` is a usage error (reserved name).
- Verify `loopx run foo -- LOOPX_ANYTHING=x` is a usage error (full `LOOPX_*` namespace reserved, not just currently-injected names).
- Verify `loopx run foo -- adr=0003 adr=0004` is a duplicate-name usage error.
- Verify `loopx run foo -- adr=0003 ADR=0004` is **not** a duplicate-name usage error; both are accepted (case-sensitive).
- Verify NUL-in-value rejection at the **parser-unit** level: when a synthetic token containing an embedded NUL byte is fed directly to the run-subcommand parser (e.g., the parser is exercised with a token array such as `["foo", "--", "a=x\0y"]` constructed in test code), the parser surfaces a NUL-in-value usage error. Note that under ordinary shell-driven CLI invocation, POSIX `argv` cannot carry NUL bytes (`bash`'s `$'x\0y'` will not deliver an embedded NUL to loopx's `argv`), so this rule is not exercisable via a real end-to-end shell invocation. It is still specified as a parser invariant because the parser is also reachable by programmatic callers that construct token arrays directly. Programmatic `RunOptions.env` NUL rejection is directly observable end-to-end: `run(target, { env: { a: "x\0y" } })` rejects on first iteration (see the "Named arguments" programmatic tests below, `run(target, { env: { adr: "a\0b" } })`).
- Verify `loopx run foo bar -- adr=x` is a usage error (extra token `bar` between target and `--`).
- Verify `loopx run foo 0003` is a usage error (existing SPEC behavior preserved — no positional arguments between target and `--`).
- Verify `loopx run -n 5 foo -- adr=0003` sets `-n = 5`, target `foo`, env additions `{ adr: "0003" }`.
- Verify `loopx run foo -n 5 -- adr=0003` sets `-n = 5`, target `foo`, env additions `{ adr: "0003" }` (flags-anywhere before `--`).
- Verify `loopx run foo -- -n 5` is a usage error (`-n` does not match `name=value`; flag recognition is disabled after `--`).
- Verify `loopx run foo -- --unknown=x` is a usage error: the name `--unknown` fails the name pattern (leading `-` is not allowed in `[A-Za-z_][A-Za-z0-9_]*`). Note that this surfaces as an invalid-name-pattern error, not an unrecognized-flag error — flag recognition is disabled after `--`, so the token is parsed as a named-argument candidate and rejected on the name rule.
- Verify `loopx run foo -- -h` is a usage error (`-h` does not match `name=value`; no help short-circuit after `--`).
- Verify `loopx run -- adr=0003` is a missing-target usage error.
- Verify `loopx run foo -h` shows run help and exits 0.
- Verify `loopx run foo -- h=-h` is valid and sets env additions `{ h: "-h" }` (the `-h` is a value, not a flag).
- Verify `loopx run foo -e ./env.local -- adr=0003` sets `-e = "./env.local"`, target `foo`, env additions `{ adr: "0003" }`.
- Verify `loopx run foo -e -- adr=0003` binds `-e = "--"` and then produces a usage error on the subsequent `adr=0003` token, which is a non-flag positional token after target (flag parsing still enabled; no `--` has been unconsumed as a separator).
- Verify `loopx run -n 5 -n 10 foo -- adr=0003` is a duplicate-`-n` usage error, independent of the presence of named arguments.
- Verify `loopx run -h foo -- adr=0003` shows run help and exits 0 (short-circuit; named arguments ignored).
- Verify that when `-h` / `--help` triggers run help, run help still performs its existing non-fatal discovery and validation behavior (per SPEC §11.2): in a project with discovery diagnostics, those diagnostics are emitted alongside the help text. The help-trigger is a parser short-circuit over the rest of run-level *parsing* (target consumption, flag validation, env loading for execution, tmpdir creation, child spawning), not a suppression of the run-help command's runtime behavior. Verify specifically that triggering run help does **not** create a `LOOPX_TMPDIR`, does **not** load env files for execution (the run-help command path performs its own discovery, which is distinct from execution-path env loading), and does **not** spawn any child process.
- Verify `loopx run foo -h -- adr=0003` shows run help and exits 0 (short-circuit before `--`; tokens after `--` are ignored).
- Verify `loopx run foo -e -h -- adr=0003` shows run help and exits 0 (help short-circuit wins over `-e` value consumption).
- Verify `loopx run foo -n --help -- adr=0003` shows run help and exits 0 (help short-circuit wins over `-n` value consumption).
- Verify `loopx run foo -- -e -h` is a usage error on `-e` (does not match `name=value`; help short-circuit only applies before `--`).
- Verify `loopx run foo -` treats `-` as a non-flag token after target; since flag parsing is enabled and the target has been consumed, this is an unrecognized-token usage error.
- Verify `loopx run -` treats `-` as the target and fails target validation because `-` does not match the name pattern.
- Verify JS/TS scripts observe named-argument values via `process.env` under both `tsx` and Bun runtimes.
- Verify scripts do **not** observe named-argument values via `process.argv` (JS/TS) or `$@` / `$1` (Bash) — `argv` is empty for all named-argument counts.
- Verify named-argument values are not injected under any `LOOPX_ARG_*` envelope; the env name is the user-supplied name verbatim.
- Verify scripts reached via intra-workflow `goto` receive the same named-argument map as the starting target.
- Verify scripts reached via cross-workflow `goto` receive the same named-argument map as the starting target.
- Verify loop reset to the starting target passes the same named-argument map on each reset.
- Verify named-argument values override same-named values in the local `-e` env file: a workflow run with `-e ./local.env` where `local.env` sets `adr=old` and a CLI of `-- adr=new` sees `adr=new` in the child.
- Verify named-argument values override same-named values in the global loopx env file.
- Verify loopx-injected protocol vars (`LOOPX_BIN`, `LOOPX_TMPDIR`, `LOOPX_PROJECT_ROOT`, `LOOPX_WORKFLOW`) are **not** overridable via named arguments, because the parser rejects `LOOPX_*` names at step 1. (The precedence ordering would also protect these, but the parser rejection is tested as the actual enforcement mechanism.)
- Verify programmatic `run(target, { env: { adr: "0003" } })` produces `process.env.adr === "0003"` inside the child.
- Verify programmatic `run(target, { env: undefined })`, `run(target, { env: {} })`, and `run(target, {})` all produce no additional env entries from `RunOptions.env` (loopx-injected and inherited env still apply).
- Verify programmatic `run(target, { env: "nope" as any })` rejects lazily on first iteration (non-object).
- Verify programmatic `run(target, { env: [] as any })` rejects lazily on first iteration (arrays are explicitly rejected by the non-null, non-array object check).
- Verify programmatic `run(target, { env: null as any })` rejects lazily on first iteration (null is not a non-null object).
- Verify programmatic `run(target, { env: { "1abc": "x" } as any })` rejects lazily on first iteration (invalid name pattern).
- Verify programmatic `run(target, { env: { "a-b": "x" } as any })` rejects lazily on first iteration (invalid name pattern).
- Verify programmatic `run(target, { env: { "LOOPX_BIN": "/fake" } as any })` rejects lazily on first iteration (reserved name).
- Verify programmatic `run(target, { env: { adr: 123 as any } })` rejects lazily on first iteration (non-string value).
- Verify programmatic `run(target, { env: { adr: "a\0b" } })` rejects lazily on first iteration (NUL in value).
- Verify programmatic `runPromise(target, { env: "nope" as any })` returns a rejected promise.
- Verify programmatic `run(target, { env: Object.create({ inherited: "x" }) })` ignores inherited properties: the child env does not contain `inherited`. This confirms the record-like object-model contract — `env` is not required to be a structurally plain object, and inherited properties are not part of the `Object.keys` snapshot.
- Verify programmatic `run(target, { env: Object.create(null) })` is accepted and injects no entries (null-prototype object with no own entries is valid; it simply contributes nothing). Verify that an `Object.create(null)` instance with an own string-keyed entry (e.g., `const e = Object.create(null); e.adr = "0003"; run(target, { env: e })`) injects that entry — the prototype is not constrained.
- Verify programmatic `env` with a non-enumerable own property is not iterated or injected: e.g., `run(target, { env: Object.defineProperty({}, "adr", { value: "0003", enumerable: false }) })` does not set `adr` in the child's env (non-enumerable properties are not in the `Object.keys` snapshot), and the call does not fail validation on account of the property.
- Verify programmatic `env` with a symbol-keyed property is not rejected and does not inject anything for the symbol: e.g., `run(target, { env: { [Symbol("s")]: "x", adr: "0003" } })` injects `adr` and does not surface a non-string-key validation error; the symbol-keyed property is simply absent from the snapshot.
- Verify programmatic `env` accepts a class-instance object with an own enumerable string-keyed property: e.g., `class C { constructor() { this.adr = "0003"; } }; run(target, { env: new C() })` injects `adr` and does not fail because `C`'s prototype is not `Object.prototype`.
- Verify programmatic `run(target, { env: { adr: "0003" } })` with `env.adr = "mutated"` *after* the call does not affect the running loop; the snapshotted value `"0003"` is what is injected (call-time snapshot).
- Verify programmatic `run(target, { env: {} })` with `env.adr = "added"` after the call does not add the new key to the running loop.
