# SPEC Problems

Tracks ambiguities, gaps, and under-specified clauses in `SPEC.md` that prevent `TEST-SPEC.md` from covering the intended behavior cleanly. Each entry is scoped to a specific ADR cycle so it can be resolved (by amending `SPEC.md` and adding the necessary tests) in a follow-up cycle.

## ADR-0004 — Script Execution Context, CLI Grammar, and Workflow Auto-Install

### P-0004-05. Aggregate-failure-report disposition when SIGINT/SIGTERM is observed during an active `npm install` child after a prior workflow's auto-install failure

**Affected SPEC sections:** §10.10 ("Signals during `npm install`"; "Aggregate failure report"; "Signals during the auto-install pass when no npm child is active").

**Problem.** SPEC §10.10's no-active-child signal clause specifies that "Signal termination suppresses the final auto-install aggregate failure report unless it had already been emitted." The active-child signal clause is silent on aggregate-report disposition: it specifies only that "loopx waits for the [npm] child to exit ... and then exits with the signal's code", and that "Remaining committed workflows are not processed". It does not say what happens to a not-yet-emitted aggregate failure report when SIGINT/SIGTERM is observed mid-active-child after a prior workflow has already recorded an entry into the aggregate accumulator.

**Why this prevents clean test coverage.** A multi-workflow auto-install pass where workflow `alpha`'s `npm install` failed (recording an aggregate entry) and workflow `beta`'s `npm install` is currently active when the signal arrives is a reachable scenario through ordinary inputs (or via the §1.4 `LOOPX_TEST_AUTOINSTALL_PAUSE` seam during `beta`'s npm execution). Under one reading, the active-child clause inherits the no-active-child suppression rule (because the aggregate report has not yet been emitted), so the report is suppressed. Under another reading, the active-child clause is independent, and the report is emitted on the way out (preserving the prior failure record). TEST-SPEC.md cannot pin a deterministic assertion until SPEC clarifies which reading is normative.

**Proposed clarification (for the resolution cycle).** Add to §10.10's active-child signal clause a sentence specifying which of the two readings holds. If suppression: "If a prior workflow's auto-install failure was recorded into the aggregate accumulator before the signal was observed, the not-yet-emitted aggregate failure report is suppressed under the same rule as the no-active-child case — the SPEC §10.10 'unless it had already been emitted' carve-out continues to apply." If preservation: "If a prior workflow's auto-install failure was recorded into the aggregate accumulator before the signal was observed and before the active npm child exited, loopx emits the aggregate failure report after the npm child exits and before exiting with the signal's code."

**Test recommendation if clarified as suppression.** Add a TEST-SPEC test in §4.10 with: `alpha` npm exits non-zero (or its safeguard fails) and records an aggregate entry; `beta`'s npm starts via `withFakeNpm({ sleepSeconds: ... })` so the child sleeps; signal SIGINT/SIGTERM while `beta`'s child is active; assert exit code `130` / `143`, active child process group terminated, no aggregate report on stderr, committed workflow files and completed side effects remain on disk, later workflows not processed. The §1.4 `LOOPX_TEST_AUTOINSTALL_PAUSE=child-active-after-failure` ordinal (or equivalent) is needed for deterministic coordination.

**Test recommendation if clarified as preservation.** Same setup as above, but assert the aggregate report is present on stderr after the signal observation (containing `alpha`'s entry), no second copy after the signal, and the rest of the no-rollback parity assertions identical to the suppression-test variants.

### P-0004-06. Whether `loopx install` reads global env files (and is therefore subject to the §8.1 unreadable-global-env failure)

**Affected SPEC sections:** §10.10 ("Environment" — auto-install env handling); §8.1 ("Global Storage" — unreadable global env file failure).

**Problem.** SPEC §10.10's "Environment" bullet says "The `npm install` child inherits loopx's own `process.env` unchanged. `LOOPX_*` protocol variables, `RunOptions.env`, and env-file entries are not injected — these are script-execution-context concerns, not install-time concerns." This rules out *injection* of env-file entries into npm but is silent on whether `loopx install` *reads* env files at all. SPEC §8.1's broad clause says "If the file [global env] exists but is unreadable (e.g., permission denied), loopx exits with code 1 and an error message" — which would apply to `loopx install` if install reads env files, but would not apply if install never consults them.

**Why this prevents clean test coverage.** Existing TEST-SPEC tests assert that *readable* global env entries are not injected into the npm child (T-INST-115 family). The complementary case — what happens to `loopx install` when the global env file is unreadable — is not pinned. Both readings are SPEC-consistent: under "install reads env files", an unreadable global env causes install to exit 1 before the file-level commit; under "install never reads env files", an unreadable global env has no effect on install. TEST-SPEC.md cannot pin a deterministic assertion until SPEC clarifies.

**Proposed clarification (for the resolution cycle).** Add a sentence to §10.10's "Environment" bullet (or to §8.1) explicitly scoping the §8.1 unreadable-global-env failure to the surfaces that *use* env files: "The §8.1 unreadable-global-env failure applies only to surfaces that read env files at runtime — `loopx run` (CLI), `run()` / `runPromise()` (programmatic API), and `loopx env *` commands. `loopx install` does not read env files (global or local) and is unaffected by an unreadable or malformed global env file." (Or, if the alternate reading is preferred: "`loopx install` reads the global env file at startup on the same schedule as `loopx run`; an unreadable global env file causes `loopx install` to exit 1 before any source acquisition or file-level commit.")

**Test recommendation if clarified as "install never reads env files".** Add a TEST-SPEC test under §4.10's "Auto-install of Workflow Dependencies" subsection: create an unreadable `$XDG_CONFIG_HOME/loopx/env` (mode `000`, conditional on `process.getuid() !== 0` per the existing T-INST-113a gating pattern); run `loopx install <source>` with `withFakeNpm`; assert install proceeds according to source / auto-install behavior and does **not** fail due to the unreadable global env (no §8.1 unreadable-file error on stderr; exit code follows the auto-install outcome, not the env-file-readability outcome).

**Test recommendation if clarified as "install reads env files at startup".** Mirror image: create an unreadable global env, run `loopx install <source>`, assert exit `1` with the §8.1 unreadable-file error on stderr, no source acquisition / file-level commit / auto-install attempted.
