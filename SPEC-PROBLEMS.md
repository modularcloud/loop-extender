# SPEC Problems

This file tracks ambiguities, gaps, or under-specified clauses in `SPEC.md` that were identified during TEST-SPEC review and could not be resolved within the originating ADR cycle. Each entry is scoped to the ADR that surfaced it; resolution happens in a follow-up cycle (either a SPEC clarification or a deliberate decision that the existing SPEC text is sufficient).

## P-0004-07: Unrecognized outer `RunOptions` fields — read or ignored? (ADR-0004)

**SPEC reference:** SPEC §9.5 ("Outer `options` field access") — `Recognized fields on the outer 'options' object (signal, env, cwd, envFile, maxIterations) are read using ordinary ECMAScript [[Get]] semantics ... Each recognized field is read at most once per call`.

**Problem.** SPEC §9.5's "Outer `options` field access" clause specifies positively that recognized fields are read using ordinary `[[Get]]` semantics, but does **not** explicitly say that unrecognized own or inherited properties on the outer `options` object must **not** be read. As written, the clause is a positive enumeration over the recognized field names; the inverse — that unrecognized field names are out-of-contract for the outer object's `[[Get]]` reads, and that getters / proxy traps reachable only through unrecognized property names are not invoked by loopx's option-snapshot pass — is left implicit.

A strict implementation could therefore choose to enumerate / log / validate extra own fields on the options object (e.g., for shape-validation diagnostics, for internal options-bag construction via `Reflect.ownKeys`, or for any other implementation-defined housekeeping) and capture a throwing getter installed on an unrecognized field name into the option-snapshot error path — and such an implementation would not violate any current SPEC clause. The current TEST-SPEC pin at T-API-61t encodes the inverse interpretation (unrecognized fields are ignored without invoking their getters / proxy traps) as the most-natural reading of SPEC §9.5's closed positive enumeration, but that interpretation is not yet pinned by SPEC text.

**Question for SPEC.** Is the intended product behavior:

- (a) **Ignore extras without touching them** — loopx's option-snapshot pass reads only the five recognized outer fields, and unrecognized own or inherited properties are not read at all; getters or proxy traps reachable only through unrecognized property names are not invoked. T-API-61t becomes a direct conformance pin. The proposed SPEC §9.5 edit would read approximately: "Loopx reads only the recognized outer `RunOptions` fields (`signal`, `env`, `cwd`, `envFile`, `maxIterations`). Unrecognized own or inherited properties on the outer options object are ignored and are not read; getters or proxy traps reachable only through unrecognized property names are not invoked by loopx's option-snapshot pass."
- (b) **Implementation-defined whether unrecognized fields are read** — loopx may or may not enumerate extra fields; throwing getters on unrecognized field names may or may not be captured into the option-snapshot error path. T-API-61t must be downgraded (its `count === 0` and "promise resolves cleanly" assertions cannot survive as a strict conformance pin) to either an observational test or be removed entirely; callers must avoid attaching side-effecting getters / proxy traps to unrecognized property names if they want cross-implementation determinism.

**Resolution path.** A SPEC clarification on this question would let TEST-SPEC either (i) keep T-API-61t with its full assertion set as a direct conformance pin under the new SPEC clause (case a), or (ii) downgrade or remove T-API-61t to match the implementation-defined contract (case b). Until then, T-API-61t encodes the most-natural reading (case a) as observational coverage, following the existing TEST-SPEC pattern for unsettled SPEC clauses (e.g., T-CLI-01a / T-CLI-01b at the SPEC §11 / §4.3 layer for unrelated grammar questions).

**Surfaced by.** TEST-SPEC review of ADR-0004 (post-acceptance feedback cycle).
