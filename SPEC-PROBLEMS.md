# SPEC Problems

This file tracks ambiguities, gaps, or under-specified clauses in `SPEC.md` that were identified during TEST-SPEC review and could not be resolved within the originating ADR cycle. Each entry is scoped to the ADR that surfaced it; resolution happens in a follow-up cycle (either a SPEC clarification or a deliberate decision that the existing SPEC text is sufficient).

## P-0004-12: Scope of "no stale-tmpdir reaping at startup" — CLI-process startup only, or any run setup including programmatic `run()` / `runPromise()`? (ADR-0004)

**SPEC reference:** SPEC §7.4 ("Run-scoped Temporary Directory") — `Cleanup does not run when loopx itself is killed via SIGKILL or the host crashes; leaked tmpdirs are expected to be reaped by OS temp-cleaning policy. **loopx does not reap stale tmpdirs at startup.**`

**Problem.** SPEC §7.4's startup-non-reaping clause is stated tersely as "loopx does not reap stale tmpdirs at startup." The phrase "at startup" is unambiguous for the CLI surface — `loopx run` invokes a fresh process whose startup is the OS-level fork/exec of the loopx binary — but is less explicit for the programmatic surfaces, where `run()` and `runPromise()` execute inside a long-lived host process whose startup happened arbitrarily long before the call. Two readings are plausible:

- **Reading A — CLI process startup only.** The non-reaping rule applies only to the CLI process's OS-level startup. Programmatic `run()` / `runPromise()` calls inside a long-lived process run their per-run setup (option-snapshot capture, `.loopx/` discovery, env-file loading, target resolution, tmpdir creation) without a separate non-reaping guarantee; an implementation **could** reap stale `loopx-*` entries during programmatic per-run setup without violating SPEC §7.4 as currently written.
- **Reading B — Any per-run setup.** "At startup" generalizes to "at the start of any run-creation pipeline," covering CLI startup **and** the per-run setup performed inside `run()` / `runPromise()` calls. Under this reading, no run-creation surface — CLI, `runPromise()`, or `run()` — may reap stale `loopx-*` entries during its setup.

The interaction with the cross-surface tmpdir-creation contract is what makes the ambiguity load-bearing: SPEC §7.4 / SPEC §9.1 / SPEC §9.2 all share the same tmpdir-creation pipeline (parent selection from `os.tmpdir()`, `mkdtemp` with `loopx-` prefix, mode `0700`, identity capture). If an implementation factors that pipeline into a shared run-setup helper and adds a "clean stale workspaces" sweep at the entry of that helper, the sweep would run on every CLI invocation **and** on every programmatic `run()` / `runPromise()` call. Reading B forbids this; reading A permits it on the programmatic surfaces but forbids it on the CLI surface.

The current TEST-SPEC encodes reading B: T-TMP-32 pins the CLI surface (clearly supported by SPEC §7.4's existing text), and T-TMP-32a / T-TMP-32b extend the same contract to `runPromise()` (eager call-time snapshot path) and `run()` (lazy first-`next()` snapshot path) respectively. The rationale prose in T-TMP-32a / T-TMP-32b explicitly defends extension to programmatic surfaces ("SPEC 7.4's 'loopx does not reap stale tmpdirs at startup' rule is run-scoped, not CLI-scoped — every run-creation path (CLI, `runPromise()`, `run()`) shares the same tmpdir-creation contract per SPEC 7.4 and SPEC 9.2"), but that interpretation is not yet pinned by SPEC text.

**Question for SPEC.** When a programmatic `run()` or `runPromise()` call performs its per-run setup inside a long-lived host process, is the implementation permitted to reap pre-existing `loopx-*` entries under the captured tmpdir parent during that setup? Specifically:

- (a) **Reading B — extend non-reaping to all run-creation surfaces.** Amend SPEC §7.4 to read approximately: "loopx does not reap stale tmpdirs during CLI startup or during any `run()` / `runPromise()` per-run setup." Under this resolution, T-TMP-32 / T-TMP-32a / T-TMP-32b all become direct conformance pins. The proposed amendment matches the existing TEST-SPEC interpretation and the cross-surface tmpdir-creation parity already established at SPEC §9.1 / §9.2 / §7.4.
- (b) **Reading A — restrict non-reaping to CLI startup.** Amend SPEC §7.4 to read approximately: "loopx does not reap stale tmpdirs at CLI startup. Whether `run()` / `runPromise()` invocations inside a long-lived process reap stale `loopx-*` entries during per-run setup is implementation-defined." Under this resolution, T-TMP-32 remains a direct conformance pin, but T-TMP-32a / T-TMP-32b would need to be downgraded to observational coverage or removed entirely; a conforming implementation could reap stale entries on programmatic surfaces and the test suite must not pin against that.
- (c) **Status quo — leave the wording as-is.** Read SPEC §7.4 strictly: "at startup" means only CLI process startup; the programmatic surfaces are unconstrained. T-TMP-32a / T-TMP-32b would be slightly over-pinning the current text and would be downgraded as in (b), unless future evidence (an implementation that wants to reap on programmatic surfaces) prompts a revisit.

**Resolution path.** A SPEC clarification on this question would let TEST-SPEC either (i) retain T-TMP-32a / T-TMP-32b as direct conformance pins under amended §7.4 (case a), or (ii) downgrade or remove them to match the implementation-defined contract (cases b / c). Until then, T-TMP-32a / T-TMP-32b encode reading B as observational coverage, following the existing TEST-SPEC pattern for unsettled SPEC clauses (e.g., the previously-pending T-CLI-01a / T-CLI-01b at the SPEC §11 / §4.3 layer for unrelated grammar questions). T-TMP-32 on the CLI surface remains a direct conformance pin under all three readings since the CLI surface is the unambiguous case.

**Surfaced by.** TEST-SPEC review of ADR-0004 (post-acceptance feedback cycle, post-P-0004-10 / P-0004-11 resolution).
