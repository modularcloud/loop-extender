# SPEC Problems

This file tracks ambiguities, gaps, or under-specified clauses in `SPEC.md` that were identified during TEST-SPEC review and could not be resolved within the originating ADR cycle. Each entry is scoped to the ADR that surfaced it; resolution happens in a follow-up cycle (either a SPEC clarification or a deliberate decision that the existing SPEC text is sufficient).

## P-0004-09: Runtime discovery behavior when the project-root `.loopx` path itself is a failed or non-directory symlink (ADR-0004)

**SPEC reference:** SPEC §5.1 ("Discovery" / "Symlink policy" + "Failed symlink resolution during runtime discovery") — `Symlinks within '.loopx/' are followed during discovery to determine entry types — a symlinked '.loopx' directory, a symlinked '.loopx/<workflow>' directory, and a symlinked entry script file are all treated identically to their non-symlinked equivalents for purposes of "is this a workflow / script?".` ... `When loopx follows a symlinked candidate workflow entry under '.loopx/' or a symlinked top-level script entry inside a candidate workflow to determine its entry type, and that resolution fails because the target is missing or because the symlink chain is cyclic, loopx treats the candidate as absent for runtime discovery purposes.`

**Problem.** SPEC §5.1's "Symlink policy" paragraph specifies positively that a symlinked `.loopx` directory is followed and its discovered (symlink-spelling) path is preserved for spawn-time invocation, `LOOPX_WORKFLOW_DIR` injection, and Bash `$0` derivation. The "Failed symlink resolution during runtime discovery" paragraph then defines failure semantics for two specific layers: (a) candidate workflow entries directly under `.loopx/` and (b) top-level script entries inside a candidate workflow directory. Both are treated as absent / silently skipped when symlink resolution fails.

But neither paragraph defines what happens at runtime discovery time when the project-root `.loopx` path **itself** is a symlink whose follow fails or resolves to the wrong type:

- `.loopx` is a **broken** symlink (target does not exist).
- `.loopx` is a **cyclic** symlink (chain that never terminates at a real entry).
- `.loopx` is a symlink to a **regular file** or other **non-directory** (e.g., FIFO, socket).

It is unclear whether each of these configurations should be treated as: (i) equivalent to a missing `.loopx/` directory (governed by SPEC §5.4 — `loopx run <target>` requires `.loopx/` and fails if it does not exist; `loopx run -h` does not require `.loopx/`); (ii) a fatal discovery error specific to symlink misconfiguration, with a distinct error class from the missing-`.loopx/` failure mode; (iii) a non-fatal warning under `loopx run -h` (paralleling SPEC §5.4's run-vs-help split for validation failures); (iv) silently ignored under SPEC §5.1's existing entry-type-policy rule (which already covers a symlink-to-non-directory or symlink-to-non-file as silently ignored at the **inner** layers — this would extend that same rule to the root layer); or (v) something else.

The interaction with the programmatic surfaces is also unspecified: `run()` and `runPromise()` both invoke discovery during pre-iteration, so the chosen behavior must apply uniformly to both surfaces, and the relationship to SPEC §9.1 / §9.3's pre-iteration failure modes (where a `.loopx/`-discovery failure is one of the enumerated displaceable errors under abort precedence) needs to be made explicit.

This is ADR-0004-scoped because ADR-0004 explicitly clarified runtime symlink discovery and discovery-time path spelling for script execution and `LOOPX_WORKFLOW_DIR` (per the §5.1 / §6.1 / §6.2 / §6.3 amendments listed in the ADR's "Affected SPEC Sections" section), and because the recently-added "Failed symlink resolution during runtime discovery" paragraph (resolving P-0004-08) inherited the same gap by addressing only the inner two layers.

**Question for SPEC.** When `.loopx` itself is a broken symlink, a cyclic symlink, or a symlink to a regular file / FIFO / socket / other non-directory, what is the intended behavior for each of `loopx run <target>`, `loopx run -h`, programmatic `run()`, and programmatic `runPromise()`?

- Plausible candidate (a) — **treat as missing `.loopx/`**: the failed / non-directory-resolved root is observationally equivalent to no `.loopx/` existing, dispatched through SPEC §5.4's existing "Requires `.loopx/`" column. Under this reading, `loopx run <target>` fails as missing-`.loopx/` and `loopx run -h` succeeds with empty discovery output. Symmetric with the SPEC §5.1 inner-layer "treated as absent for runtime discovery purposes" rule.
- Plausible candidate (b) — **fatal discovery error in `loopx run <target>`, distinct from missing-`.loopx/`**: surface an explicit symlink-misconfiguration error so the user sees an actionable diagnostic rather than a generic "no `.loopx/` directory" message. May or may not warn under `loopx run -h` (depending on whether the run-vs-help split applies).
- Plausible candidate (c) — **per-failure-class differentiation at the root**: e.g., broken / cyclic root symlinks treated as missing `.loopx/` (the root is simply absent in the same sense as the inner layers), but a symlink to a regular file / FIFO / socket surfaced as fatal because the user explicitly created the `.loopx` entry but pointed it at the wrong type. Distinguishes "root absent" from "root present but wrong type."
- Plausible candidate (d) — **silent ignore under SPEC §5.1's existing entry-type-policy rule, extended to the root layer**: matches the new "Failed symlink resolution during runtime discovery" rule for the inner two layers, applied uniformly to the root layer. Functionally similar to (a) but framed as an entry-type-policy outcome rather than a missing-`.loopx/` outcome; the dispatch path differs even if the externally observed behavior is identical for the broken / cyclic / non-directory sub-cases.

The interaction with the discovery-cache rule (SPEC §5.1 — "Discovery metadata is cached at loop start for the duration of the loop") is also unspecified: whether a root `.loopx` symlink whose target is replaced mid-run with a different type would be observed at all (and on which iteration boundary), or whether the cached path takes the discovery-time root spelling regardless of mid-run mutation, should be made explicit alongside any chosen behavior.

**Resolution path.** A SPEC clarification on the chosen behavior would let TEST-SPEC.md add coverage for each of the failure variants × surface (`loopx run <target>`, `loopx run -h`, `run()`, `runPromise()`) × failure-class (broken, cyclic, regular file, non-directory non-symlink) combinations. Per the reviewer's recommendation, no tests or SPEC edits are added in the originating cycle until the intended behavior is decided; this entry tracks the gap so a follow-up cycle can resolve it.

**Surfaced by.** TEST-SPEC review of ADR-0004 (post-acceptance feedback cycle, post-P-0004-08 resolution).
