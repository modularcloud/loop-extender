- **T-TMP-12a/12b/12c (3 sub-cases)** — implemented in `apps/tests/tests/e2e/tmpdir.test.ts`. Covers the mkdtemp-fails-on-unwritable-parent ordering across the three surfaces (CLI / `runPromise` / `run`), parameterized over node + bun runtimes (= 3 × 2 = 6 test invocations). 2 currently pass and 4 currently fail. T-TMP-12a × {node, bun} pass because the CLI process crash on the eager `loopx-nodepath-shim-<pid>` mkdirSync at module-load time produces the same observable outcomes the test asserts: exit 1, stderr non-empty, no marker file written, no `loopx-*` directory under the unwritable parent, zero `LOOPX_TEST_CLEANUP_WARNING\t…` lines. T-TMP-12b/12c × {node, bun} fail because the api-driver subprocess imports `loopx`, which triggers the eager shim mkdirSync; the unwritable TMPDIR causes the import to throw before the driver can call `run()` / `runPromise()` and print its JSON envelope, so `result.exitCode` is non-zero. The failures are an ordering issue: pre-ADR-0004 has eager TMPDIR-dependent module-load work (`loopx-nodepath-shim-<pid>` mkdirSync) that crashes when TMPDIR points at an unwritable parent; ADR-0004 implementation must decouple shim location from `LOOPX_TMPDIR` parent or make shim creation lazy / failure-tolerant.
- **T-TMP-12d/12d2/12e/12e2/12e3 (5 sub-cases × 3 surfaces)** — implemented in `apps/tests/tests/e2e/tmpdir.test.ts`. Covers the SPEC §7.4 creation-order sub-step coverage for sub-steps 2 (identity-capture) and 3 (mode-securing), each on its success-cleanup branch (12d / 12e) and on its cleanup-failure branch (12d2 / 12e2 / 12e3). Uses the `LOOPX_TEST_TMPDIR_FAULT` seam (TEST-SPEC §1.4) and, for the cleanup-failure compositions, `LOOPX_TEST_CLEANUP_FAULT`. Parameterized over three execution surfaces (CLI / `run()` / `runPromise()`) and over node + bun runtimes (= 5 × 3 × 2 = 30 test invocations). All 30 currently fail against the pre-ADR-0004 implementation: the impl does not honor `LOOPX_TEST_TMPDIR_FAULT` or `LOOPX_TEST_CLEANUP_FAULT`, so `mkdtemp` succeeds normally and the loop runs to completion (CLI exits 0 instead of 1; programmatic surfaces resolve/yield instead of throwing/rejecting). Implementation work required: (1) honor `LOOPX_TEST_TMPDIR_FAULT={identity-capture-fail,identity-capture-fail-rmdir-fail,mode-secure-fail}` under `NODE_ENV=test` per TEST-SPEC §1.4; (2) honor `LOOPX_TEST_CLEANUP_FAULT={lstat-fail,recursive-remove-fail}` under `NODE_ENV=test`; (3) emit the `LOOPX_TEST_CLEANUP_WARNING\t<payload>` structured marker line on every cleanup-warning emission under `NODE_ENV=test` (TEST-SPEC §1.4 "Cleanup-warning structured marker"); (4) implement the SPEC §7.4 creation-order behaviors themselves (single non-recursive `rmdir` after identity-capture failure; full identity-fingerprint cleanup-safety routine after mode-secure failure; single warning + leave-in-place on cleanup failure; "does not mask the original creation error"). Helper `runTmpdirFaultTest` (defined at module scope below `assertNoTmpdirCreated`) provides the snapshot-before / drive / snapshot-after harness shape across all three surfaces; new constant `TMPDIR_FAULT_SURFACES` parameterizes over CLI / run / runPromise.
- **T-TMP-12f/12f2/12f3/12f4/12f5 + T-TMP-12g + T-TMP-12h (version-check-vs-tmpdir-creation ordering, 15 sub-cases × 3 surfaces)** — implemented in `apps/tests/tests/e2e/tmpdir.test.ts`. Covers the SPEC §7.1 "step 5 (workflow version check) before step 6 (LOOPX_TMPDIR creation)" ordering by composing an unwritable tmpdir parent (so `mkdtemp` fails) with each of the five SPEC §3.2 normatively-warning `package.json` failure-mode branches: (i) unsatisfied range, (ii) invalid JSON, (iii) invalid semver range, (iv) unreadable (`chmod 000`), (v) non-regular path (`package.json/` is a directory — the new P-0004-03 warning branch). The CLI surface gets one ID per variant (T-TMP-12f / 12f2 / 12f3 / 12f4 / 12f5); the programmatic surfaces share one ID per surface (T-TMP-12g for `run()`, T-TMP-12h for `runPromise()`) parameterized over all five variants. Per `forEachRuntime`, each (surface × variant) is exercised on node + bun (= 5 × 3 × 2 = 30 test invocations). All 30 are conditional on `process.getuid() !== 0` and currently fail against the pre-ADR-0004 implementation: the eager `loopx-nodepath-shim-<pid>` `mkdirSync` in `execution.ts` module-load crashes with EACCES under the unwritable `TMPDIR` *before* the SPEC §7.1 step 5 version check runs, so the SPEC §3.2 `package.json` warning is never emitted. The test asserts the warning text appears in stderr **scoped to a line that mentions the workflow name `ralph`**, which prevents the shim's "EACCES: permission denied" error from spuriously satisfying the predicate. Implementation work required: (1) decouple the shim location from `LOOPX_TMPDIR` parent or make the shim creation lazy / failure-tolerant so module-load does not crash on unwritable `TMPDIR`; (2) ensure SPEC §7.1 step 5's workflow version check (and its SPEC §3.2 warning emission) runs to completion before SPEC §7.1 step 6's `mkdtemp` is attempted on both the CLI and the programmatic surfaces; (3) emit the SPEC §3.2 warning text containing the workflow name so the predicate distinguishes it from unrelated EACCES error lines. Module-scope helpers `setupPkgJsonVariantFixture` (constructs the variant-specific `package.json`), `stderrHasPkgJsonVariantWarning` (workflow-name-scoped variant-specific predicate), and `runPkgJsonVariantBeforeTmpdirTest` (shared harness across all three surfaces) are defined alongside the existing `runTmpdirFaultTest` helper.
- **T-TMP-16/16a/16b/16c/16d/16e/16f/16g/16h/16i/16j (goto-resolution-failure cleanup triggers, 10 sub-cases — mixed surfaces)** — implemented in `apps/tests/tests/e2e/tmpdir.test.ts`. Covers SPEC §7.4's two distinct goto-resolution cleanup triggers: (a) "missing workflow or script during goto resolution" — T-TMP-16/16a (CLI), T-TMP-16c/16d (`runPromise()`), T-TMP-16e/16f (`run()`) for qualified targets, plus T-TMP-16g/16h/16i for the bare-name same-workflow lookup path; and (b) "invalid goto target" — T-TMP-16b for delimiter-syntax violations (`a:b:c` — multi-colon) on all three surfaces and T-TMP-16j for SPEC 4.1 name-restriction violations (`-bad` — leading dash) on all three surfaces. Each test follows the same shape: `ralph/index.sh` observes `$LOOPX_TMPDIR` into a marker file external to the tmpdir, then emits the failure-mode goto value; the harness asserts the surface-appropriate failure surfaces, the error mentions the expected category-distinct phrasing (workflow name + "not found" for missing-workflow; the missing script + workflow name for missing-script; "delimiter" / "multiple colons" / "only one ':' delimiter" for multi-colon — explicitly *not* matching "not found in `.loopx/`"; "must match `[a-zA-Z…`" / "name-restriction" / "name pattern" for name-restriction — explicitly *not* matching "not found" *or* "delimiter" / "multiple colons"), the marker captured a non-empty tmpdir path (proving LOOPX_TMPDIR was injected), and the recorded path no longer exists (proving cleanup ran on the failure path). Per `forEachRuntime`, single-surface tests run on node + bun (× 2) and the 2 multi-surface IDs (16b, 16j) run on (3 surfaces × 2 runtimes) (= 8 single-surface invocations × 1 + 2 multi-surface IDs × 6 = 8 + 12 = 20; total: 6 single-surface IDs × 2 runtimes + 1 single-surface ID × 2 runtimes + 2 multi-surface IDs × 3 surfaces × 2 runtimes — counted correctly: 9 IDs × 1-or-3 surfaces × 2 runtimes = 30 test invocations). All 30 currently fail against the pre-ADR-0004 implementation on the LOOPX_TMPDIR injection assertion (`expect(observed.length).toBeGreaterThan(0)`): the impl does not inject `LOOPX_TMPDIR` so the marker captures empty string. Goto-resolution failures themselves already produce the right exit code / rejection (the loop / target-validation paths have been correct since pre-ADR), so the test fails on the LOOPX_TMPDIR-related assertions only. Module-scope helpers `buildGotoCleanupScript`, `driveRunPromiseGotoCleanup`, and `driveRunGotoCleanup` provide the shared fixture / driver shapes across CLI / `runPromise()` / `run()` surfaces.
